# Конвенции bmm-translib

Этот файл — словесное описание контракта, зафиксированного в [`common.hpp`](common.hpp).
Если код и этот файл разойдутся — правьте оба сразу, `common.hpp` считается
источником истины для типов, этот файл — источником истины для *обоснований*.

## 1. Порядок переменных: `BitOrder::LSB_FIRST`

**Решение (зафиксировано явно, не подразумевается):** переменная с индексом
`i` — это бит `i` индекса минтерма в таблице истинности (`assignment[i]` —
значение i-й переменной), `x0` — младший бит индекса. Формально:
`index = sum(assignment[i] << i, i=0..n-1)`.

**Почему LSB_FIRST, а не MSB_FIRST — по каждой библиотеке отдельно, т.к.
ограничения у них разного рода:**

- **kitty/mockturtle — это жёсткое ограничение, не выбор.**
  `kitty::dynamic_truth_table` определяет bit `i` слова как значение функции
  на входе, где `i` в двоичном виде даёт значения `x0..x_{n-1}` с `x0` в
  младшем разряде — это фиксировано самой библиотекой и не настраивается.
  `mockturtle::simulate<kitty::dynamic_truth_table>` сопоставляет PI i-го по
  порядку создания (`create_pi()`) с переменной `i` этой таблицы. Раз TT и
  AIG-симуляция уже LSB_FIRST по построению библиотеки — на границе
  `Aig::to_tt()`/`aig_to_tt` пришлось бы разворачивать биты при любом другом
  выборе, что было бы чистым источником off-by-one багов без какой-либо
  пользы. Этим ограничением определяется выбор для всего проекта.
- **CUDD/Sylvan — свободный выбор, ограничения нет.** И `Cudd_bddIthVar`, и
  `sylvan::Bdd::bddVar(index)` берут произвольный целочисленный индекс
  переменной без встроенной семантики "какой это бит числа" — сопоставление
  переменная↔позиция в присвоении целиком на совести вызывающего кода. Мы
  используем тот же индекс `i`, что и везде (`bddVar(i)` ↔ `assignment[i]`),
  просто ради единообразия с kitty/mockturtle, а не потому что Sylvan этого
  требует.
- **BRiAl — тоже свободный выбор.** `BoolePolyRing::variable(i)` аналогично
  берёт произвольный индекс без встроенного порядка. Тот же принцип:
  используем индекс `i` synced с остальными представлениями ради
  единообразия.

Итог: реального конфликта требований между библиотеками нет (только
kitty/mockturtle к чему-то принуждают, остальные три — нейтральны), поэтому
LSB_FIRST, продиктованный kitty/mockturtle, безопасно распространяется на
весь проект без потери совместимости с CUDD/Sylvan/BRiAl.

Другого значения `BitOrder` не будет — enum с одним элементом существует,
чтобы явно задокументировать выбор в типах, а не чтобы допускать альтернативы.

## 2. Механизм ошибок: `Result<T>`, не исключения

Выбран **`Result<T>`-подход** (аналог Rust `Result`/`std::expected`), а не
исключения. Причина: 20 функций трансляции почти всегда должны уметь вернуть
`NotImplemented` на этапе, когда тело ещё не написано, и это ожидаемый,
частый, не-исключительный путь — гонять его через `throw`/`catch` на каждый
тестовый пример дороже и менее читаемо, чем проверить `is_ok(r)`.

- Если компилятор поддерживает `<expected>` (C++23, определяется через
  `__cpp_lib_expected`) — `Result<T>` это `std::expected<T, Error>`.
- Иначе — fallback на `std::variant<T, Error>` с тем же набором свободных
  функций (`is_ok`, `value`, `error`, `ok`, `fail`), чтобы код студентов не
  зависел от того, какой бэкенд реально используется. Какой бэкенд выбран для
  конкретной сборки — смотри `STATUS.md` (генерируется автоматически, см.
  `verify/test_runner.hpp` и корневой `CMakeLists.txt`).
- Никаких кастомных `throw NotImplementedError{}` в теле функций трансляции
  — только `return fail<T>(ErrorCode::NotImplemented, "...")`. Тестовый
  раннер (`verify/test_runner.hpp`) полагается именно на это для SKIP.

## 2а. Graceful degradation при нехватке памяти: единственное разрешённое исключение из п.2

`ErrorCode::OutOfMemory` — отдельный статус от `TooManyVariables` (см.
комментарий у `ErrorCode` в `common.hpp`): `TooManyVariables` — статическая
проверка `n_vars() > kMaxTruthTableVars` ДО начала работы; `OutOfMemory` —
для функций, где размер результата **не оценить заранее** по входу, и рост
может обнаружиться только по факту исчерпания памяти в процессе построения
(типичный кандидат — `bdd/bdd_to_aig.*`: раздутый порядок переменных BDD
может дать неограниченно большой AIG, см. doc-комментарий в этом файле).

Для таких функций — **единственное** разрешённое исключение из общего
правила п.2 "никаких исключений, только `Result<T>`": оберните ГРАНИЦУ
функции (весь её тело целиком) в `try { ... } catch (const std::bad_alloc&) { return fail<T>(ErrorCode::OutOfMemory, "..."); }`
и верните `OutOfMemory` вместо падения процесса. Это **не** механизм
передачи ошибок между внутренними шагами — внутри функции по-прежнему
работайте через `Result<T>`/обычные возвраты; `catch` стоит РОВНО один раз,
на входной точке, как последний рубеж перед `std::terminate`. Если внутри
функции есть свои `Result<T>`-возвраты ошибок — они возвращаются как обычно
и не имеют отношения к этому `catch`.

Функции, где это обязательно (перечень пополняется по мере обнаружения
новых источников неконтролируемого роста, сейчас установлено для):
- `bdd/bdd_to_aig.*` — см. doc-комментарий в файле.

## 3. Единый интерфейс представления

Каждый из 5 представлений (`Aig`, `Bdd`, `Anf`, `Thr`, `TruthTable`) реализует:

```cpp
uint32_t n_vars() const;
bool evaluate(const Assignment& assignment) const;   // assignment.size() == n_vars(), предусловие
Result<TruthTable> to_tt() const;                    // TooManyVariables, если n_vars() > kMaxTruthTableVars
```

Это проверяется на этапе компиляции концептом `Representation` в конце
`common.hpp` — если добавляете новое представление или меняете сигнатуру,
`static_assert` подскажет, что именно не совпало.

`to_tt()` возвращает `Result`, а не голый `TruthTable`, потому что реальные
AIG/BDD/ANF/Thr могут представлять функции с сотнями переменных (бенчмарки
EPFL, например), а таблица истинности материализуется только до
`kMaxTruthTableVars = 24` (2^24 строк, разумный потолок для in-memory
таблицы). `n_vars()`/`evaluate()` такого ограничения не имеют.

## 4. Область охвата: один выход

`bmm` транслирует **одновыходные** булевы функции. `mockturtle::aig_network`
в общем случае поддерживает много PO — обёртка `Aig` в `common.hpp` требует
ровно один PO и возвращает `ErrorCode::Unsupported`, если их больше одного.
Расширение на многовыходные функции — не в скоупе этого учебного проекта
(потребовало бы менять контракт `to_tt()`/`evaluate()` на векторный, что
пока не оправдано).

## 5. BRiAl как C++-зависимость (разрыв закрыт во второй итерации)

**Обновление второй итерации:** `.devcontainer/Dockerfile` теперь собирает
BRiAl **отдельным шагом** как самостоятельную системную C++-библиотеку
(autotools, `git clone` + `configure`/`make install` в `/usr/local` — блок
"BRiAl" в Dockerfile, тот же паттерн, что уже применялся к CUDD/Sylvan), a
не только как приватный бэкенд SageMath. Так и было задумано с самого
начала (BRiAl — своё кольцо полиномов над GF(2) через собственный ZDD,
физически не в одном менеджере с Sylvan/BDD — обмен между `bdd_to_anf`/
`anf_to_bdd` идёт через явную сериализацию структур, не через общий
указатель на узел, см. п.6 ниже про требования к параллелизму для этой
пары), просто в первой итерации инфраструктурный образ этот шаг ещё не
делал. В актуальном образе `BMM_HAVE_BRIAL` должен быть `1` "из коробки".

Ниже — history этого решения и почему `common.hpp` всё равно оставляет
рабочий fallback, а не жёстко требует BRiAl:

Раньше (первая итерация) образ `.devcontainer` ставил BRiAl **только** как
бэкенд SageMath (`BooleanPolynomialRing()` из Python), явно избегая
отдельного conda-пакета `brial`, потому что он конфликтует с версией,
которую тянет `sage`. Это не гарантировало наличие C++-заголовков libbrial
(`polybori.h` и т.п.) и слинкованной библиотеки, доступной для
`find_package` из чистого C++ — а протаскивать Python туда, где по
требованиям проекта нужен чистый C++, мы не хотели. Поэтому тип `Anf`
определён в двух вариантах (детектирование через
`__has_include(<polybori.h>)`, не привязано к тому, что именно поставил
конкретный Dockerfile — так репозиторий не ломается на чуть другом образе):

**Третья итерация (изоляция от common.hpp):** сам тип `Anf` и
BRiAl-детектирование физически лежат в **`core/anf_repr.hpp`**, а не в
`core/common.hpp`. Причина: `common.hpp` подключают вообще все файлы
проекта через `bmm_core`, поэтому раньше `BMM_HAVE_BRIAL` был не
переключателем "для функций, работающих с ANF", а переключателем для
**всего проекта разом** — недоступность/конфликт BRiAl ломал сборку даже
тех функций (`aig_to_thr`, `thr_to_bdd` и т.п.), которые вообще не касаются
ANF. Теперь `core/anf_repr.hpp` подключают явно только файлы, которым
реально нужен `Anf`: `aig_to_anf.*`, `bdd_to_anf.*`, `thr_to_anf.*`, весь
`anf/*.*`, `verify/reference_builders.*`, `verify/config_dump.cpp`. API
типа `Anf` не изменился:

- `BMM_HAVE_BRIAL=1` (заголовки найдены CMake'ом, см. `CMakeLists.txt`,
  переменные `BRIAL_INCLUDE_DIR`/`BRIAL_LIBRARY`) — `Anf` оборачивает
  настоящий `BoolePolynomial`.
- `BMM_HAVE_BRIAL=0` (заголовки не найдены) — `Anf` использует резервное
  представление: явное множество мономов над GF(2) (`AnfFallback`,
  XOR-семантика при добавлении повторного монома — это математически верно
  для полинома Жегалкина, просто без ускоряющих ZDD-структур BRiAl).

**Задумано так, что "не заглушка ради компиляции", а обе ветки полностью
рабочие с разной асимптотикой — но по факту это верно не для всех функций,
работающих с `Anf`.** Найдено и подтверждено ревью (см. `SESSION_REPORT.md`,
`aig/README.md` §1.3, `bdd/README.md` §4.4): `anf_to_bdd`/`bdd_to_anf` в
сборке с `BMM_HAVE_BRIAL=0` возвращают `NotImplemented`/физически не могут
обработать вход за пределами `kMaxTruthTableVars=24` — т.е. для ЭТИХ двух
функций утверждение "обе ветки рабочие" ложно, `AnfFallback`-путь для них не
реализован вовсе (не "медленнее", а недоступен). Остальные функции,
работающие с `Anf` (`tt_to_anf`, `anf_to_tt`, `aig_to_anf`, `thr_to_anf`),
действительно имеют рабочую `AnfFallback`-ветку, разница только в
асимптотике/производительности (BRiAl обычно на порядки быстрее на плотных
структурах, см. `aig/README.md` §4а). Если добавляете новую функцию с `Anf`
на входе/выходе — не полагайтесь на формулировку выше как на гарантию,
проверяйте фактическое поведение `#else`-ветки отдельно.

Какая ветка активна в вашей сборке — смотрите в `STATUS.md` (строка "ANF
backend") или в выводе CMake-конфигурации. Если у вас в контейнере всё же
есть заголовки libbrial — включение реальной ветки будет автоматическим,
ничего вручную переключать не нужно.

## 5а. Google OR-Tools — ILP-солвер для распознавания пороговых функций

Добавлен во второй итерации (`.devcontainer/Dockerfile`, блок "Google
OR-Tools"; `CMakeLists.txt`, `find_package(ortools CONFIG REQUIRED)`,
таргет `ortools::ortools`). Нужен там, где распознавание "функция —
пороговая, каковы веса" сводится к LP/ILP (см. `aig/aig_to_thr.hpp`,
`anf/anf_to_thr.hpp`, `thr/tt_to_thr.hpp`, и особенно подробно —
`bdd/bdd_to_thr.hpp`, где расписан весь пайплайн по шагам). До этой
итерации doc-комментарии этих функций честно писали "LP/ILP-солвер не
входит в текущий стек зависимостей — либо принесите свой, либо реализуйте
вручную" — теперь солвер есть, `bmm_aig`/`bmm_anf`/`bmm_bdd`/`bmm_thr`
линкуют `ortools::ortools`.

## 6. Требования к параллелизму (обязательны для соответствующих функций)

- `aig/*`, `anf/*` → **TBB**: `tbb::task_group` для рекурсивного обхода
  графа + `tbb::concurrent_hash_map` для мемоизации посещённых узлов.
  Внимание: `libtbb-dev`/oneTBB **не установлен** в текущем
  `.devcontainer`-образе (см. Dockerfile) — `CMakeLists.txt` сначала
  пробует `find_package(TBB)`, а если не находит — подтягивает oneTBB через
  `FetchContent` (собирается один раз, кешируется, а не на каждый билд).
- `bdd/*` → **никакого собственного параллелизма** — только Sylvan/Lace API
  (`sylvan::Bdd`, `sylvan_obj.hpp`). Sylvan сам параллелит операции над BDD
  через Lace; заводить свои потоки/TBB здесь — значит бороться с
  планировщиком Lace за ядра.
- `thr/*` и любые `*_to_tt`/`tt_to_*` → **OpenMP** `#pragma omp parallel for`
  над плоским `uint64_t[]`/`bool[]`-массивом. Без указателей и объектов в
  горячем цикле — это осознанный задел под будущий перенос этих функций на
  GPU (см. `profiling/README.md`), не стилистическая прихоть.

**Приоритет при конфликте правил** (у части функций входной/выходной формат
попадает сразу в две категории, например `aig_to_tt` — это и "aig/\*", и
"\*_to_tt"): правила применяются в этом порядке, первое подошедшее — решает:

1. Если `Bdd` — источник или назначение, вообще неважно что с другой
   стороны: **только Sylvan/Lace**, никакого TBB/OpenMP в этой функции.
   Это не стилистика, а требование безопасности: вызывать Sylvan из потоков,
   не зарегистрированных как Lace worker'ы, небезопасно, и заводить рядом
   независимый TBB/OpenMP тред-пул, который дёргает Sylvan API, — верный
   способ словить гонку или падение внутри Sylvan.
2. Иначе, если функция строит или обходит `Aig`/`Anf` как граф с рекурсией и
   мемоизацией (`aig_to_*`, `anf_to_*`, `tt_to_aig`, `tt_to_anf`) — TBB.
3. Иначе (`thr/*`, `*_to_tt`, `tt_to_thr`) — OpenMP над плоским массивом.

Частный случай — `tt_to_anf`: сам быстрый Мёбиус/Жегалкин-transform (см.
литературную ссылку в `anf/tt_to_anf.hpp`) естественно ложится на плоский
`uint64_t[]`-массив (butterfly-паттерн, как в FFT) — OpenMP тут напрашивается
не меньше, чем TBB для последующей сборки результата в граф/ZDD-структуру
`Anf`. Правило №2 выше формально предписывает TBB для всей функции; какую
часть в итоге распараллеливать чем — решение студента, реализующего эту
конкретную функцию, инфраструктура не диктует жёстко глубже этого файла.

## 7. Верификация: независимые пути, не общий код

`verify/ground_truth`, `verify/metamorphic`, `verify/sat_encoding` намеренно
не переиспользуют внутренние структуры `aig/bdd/anf/thr` — если бы
`ground_truth` вызывал, скажем, `Aig::evaluate()`, он проверял бы код на
самом себе. `sat_encoding` дополнительно не линкуется с C++-библиотекой SAT
солвера напрямую: в образе есть только бинарники `/usr/local/bin/cadical` и
`/usr/local/bin/kissat` (без `libcadical.a`/`cadical.hpp` — Dockerfile
удаляет `/opt/src/cadical` после сборки). Поэтому verify/sat_encoding
формирует DIMACS CNF и вызывает решатель как **внешний процесс**, разбирая
его exit code (10=SAT, 20=UNSAT, по конвенции SAT Competition) — это
честный независимый путь верификации, просто без линковки. Если в будущем
Dockerfile научится ставить `libcadical`+`cadical.hpp`, `verify/sat_encoding`
можно будет переключить на прямой C++ API без изменения внешнего интерфейса
модуля (`check_equivalence(...)` останется той же функцией).

## 7а. Лимит n для exhaustive-тестов в CI: n ≤ kMaxGroundTruthVars

Зафиксировано явно во второй итерации задания. **Исправлено** (было указано
"= 16" — фактическое значение в коде `verify/ground_truth/ground_truth.hpp`
на момент проверки (см. `SESSION_REPORT.md`) — `kMaxGroundTruthVars = 12`, не
16; расхождение между этим файлом и кодом не отслеживалось и накопилось со
временем — сверяйте значение в коде, если оно важно для расчёта бюджета
времени). Растущий набор тестовых функций (`growing_test_functions`/
`growing_threshold_test_functions`) идёт от n=1,2,3 до `kMaxGroundTruthVars`
включительно и именно этим набором гоняются `verify/test_runner.hpp`-проверки
на каждый PR/push **там, где такой CI настроен** (см. оговорку в конце файла
про `.github/workflows/ci.yml`). Это про **корректность** (exhaustive-перебор
всех 2^n точек), не про производительность.

Если для какой-то функции нужно проверить поведение/производительность на
большем n (например, реальные схемы из `benchmarks/` с сотнями переменных)
— это **не** обязательный CI-прогон: используйте `benchmarks/` (см. п.3
ниже про TBB-бенчмарки для `aig/*`/`anf/*`) отдельным, не блокирующим PR
шагом. Не поднимайте `kMaxGroundTruthVars` "чтобы протестировать получше" —
это сделает каждый CI-прогон дороже для всей команды ради задачи, для
которой есть специально выделенное место.

## 8. `NotImplemented` — это не ошибка

`verify/test_runner.hpp` обязан отличать "функция ещё не написана"
(`ErrorCode::NotImplemented`) от "функция написана неправильно" (любой
другой `Error` или несовпадение с ground truth). Первое — `SKIP`, второе —
`FAIL`. CI обязан оставаться зелёным, пока функции не реализованы; падать
