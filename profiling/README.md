# Профилирование bmm-translib

Цель: видеть, как TBB/OpenMP/Lace-части кода загружают ядра CPU, без
сложной настройки.

## Основной путь: Tracy Profiler

Используется [Tracy](https://github.com/wolfpld/tracy) — header-only
клиент с real-time визуализацией таймлайна по потокам/ядрам, зонами
(`ZoneScoped`), нативным пониманием TBB/OpenMP-регионов и заделом на
GPU-трейсинг (пригодится, когда TT/Thr перенесут на CUDA — см.
`core/CONVENTIONS.md` п.6 про плоские массивы без указателей уже сейчас,
это осознанная подготовка к этому переносу).

**Исправление URL:** в исходном задании репозиторий Tracy указан как
`github.com/wolfai/tracy` — такого репозитория не существует. Реальный
upstream — **`github.com/wolfpld/tracy`** (автор Bartosz Taudul, ник
`wolfpld`). `CMakeLists.txt` в корне репозитория уже ссылается на
правильный адрес.

Зоны (`ZoneScoped;`) уже расставлены в начале каждой из 20 функций
трансляции (см. `aig/*.cpp`, `bdd/*.cpp`, `anf/*.cpp`, `thr/*.cpp`) — даже
в нереализованных стабах: вы увидите структуру вызовов в Tracy сразу после
сборки, просто с пустыми/быстрыми зонами.

### Как собрать с Tracy

Собрано по умолчанию (`BMM_ENABLE_TRACY=ON` в `CMakeLists.txt`). Явно:

```sh
cmake -S . -B build -DBMM_ENABLE_TRACY=ON
cmake --build build --parallel
```

Чтобы собрать БЕЗ инструментирования (например, для замера чистой
производительности без even минимальных накладных расходов Tracy):

```sh
cmake -S . -B build -DBMM_ENABLE_TRACY=OFF
```

### Как подключить Tracy GUI

Клиент (то, что встроено в `test_aig`/`test_bdd`/`test_anf`/`test_thr` и в
сами библиотеки `bmm_aig`/`bmm_bdd`/`bmm_anf`/`bmm_thr`) собирается вместе
с проектом и слушает на TCP-порту при запуске программы. GUI-сервер (то,
что рисует таймлайн) — отдельная программа, которую в контейнере ставить
не обязательно:

1. **Если вы работаете локально с Coder workspace через VS Code
   Remote/code-server**: пробросьте порт Tracy (по умолчанию `8086`)
   аналогично тому, как code-server проброшен на `13337` (см.
   `coder-deploy/template/main.tf`, `coder_app "code-server"`) — либо через
   `ssh -L 8086:localhost:8086 <workspace>`, либо через встроенный
   port-forward Coder CLI: `coder port-forward <workspace> --tcp 8086`.
2. Соберите Tracy GUI (`tracy-profiler`) **у себя локально** (не в
   контейнере — это тяжёлый GUI с зависимостями на графику, ставить его в
   headless Linux-контейнер контрпродуктивно): следуйте инструкции сборки
   в самом репозитории Tracy (`profiler/` в `github.com/wolfpld/tracy`),
   либо возьмите готовый релизный бинарь с GitHub Releases того же
   репозитория, если он доступен под вашу ОС.
3. Запустите `test_aig`/`test_bdd`/`test_anf`/`test_thr` (или отдельный
   бенчмарк, см. `benchmarks/`) внутри workspace — Tracy-клиент начнёт
   слушать порт сразу при старте программы.
4. Откройте Tracy GUI локально, подключитесь к `localhost:8086` (порт
   проброшен на шаге 1) — увидите таймлайн потоков в реальном времени.

### Как читать core-affinity/timeline

- Каждая горизонтальная полоса в таймлайне — один OS-поток; для
  TBB/OpenMP это будут worker-потоки соответствующего пула, для Sylvan —
  Lace worker'ы.
- Зоны (`ZoneScoped` из тела функции трансляции) рисуются как вложенные
  прямоугольники внутри полосы потока — ширина = время выполнения.
- Если на многоядерной машине вы видите, что активна только ОДНА полоса
  всё время работы `aig_to_bdd`/`bdd_to_*`/`thr_to_bdd`/`anf_to_bdd` —
  это **ожидаемо** (Bdd-функции намеренно не создают собственный
  параллелизм, см. `core/CONVENTIONS.md` п.6, правило 1) — параллелизм
  внутри Sylvan/Lace виден отдельно, как активность внутри самого Sylvan,
  если он собран с поддержкой Tracy/своим профилированием (не входит в
  скоуп этого скелета).
- Если функция помечена на `aig/*`/`anf/*` (кроме тех, что трогают Bdd) и
  вы видите одну полосу — вероятно, TBB `task_group` не создаёт реальный
  параллелизм на маленьких тестовых функциях (n=1..3 из
  `verify/ground_truth`) — это нормально, TBB сам решает, стоит ли
  распараллеливать такую мелкую работу. Смотрите профиль на больших входах
  из `benchmarks/` (EPFL и т.п.), не на тестах из `verify/`.

## Fallback без GUI: `perf_report.sh`

Если Tracy не поднимается (нет графики локально, порт не пробрасывается,
и т.п.) — используйте `profiling/perf_report.sh <бинарь> [аргументы...]`:
запускает `perf stat` (task-clock/cycles/cache-misses) и, если доступен
`perf record`+`flamegraph`, строит flame graph в SVG. Быстрая проверка без
какого-либо GUI, прямо в терминале Coder workspace.
