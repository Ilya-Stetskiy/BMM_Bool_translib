# thr/ — заметки по реализации

Пять функций трансляции с `Thr` (пороговая функция: веса `w_1..w_n` +
порог `theta`) на входе или выходе: `tt_to_thr`, `thr_to_aig`, `thr_to_bdd`,
`thr_to_anf`, `thr_to_tt`. Этот файл — не описание API (см. doc-комментарии
в соответствующих `.hpp`) и не журнал изменений (см. `git log`), а
**рассуждения и измерения**, которые не выводятся напрямую из кода:
результаты жёсткого код-ревью этой папки и прогона реального датасета
(`thr/bench_real_corpus.cpp`), полученные в docker-окружении проекта
(`genetica-boolean-lib:latest`), а не только чтением исходников.

## 1. Штатный OpenMP/TBB-бенчмарк в `test_thr.cpp` — что он реально показывает

`test_thr.cpp` содержит секции `*_openmp_scaling`, гоняющие каждую функцию
на синтетическом входе (`make_bench_threshold`: все веса 1, majority-подобный
порог) под `omp_set_num_threads(1)` и под дефолтным числом потоков
(`benchmarks/openmp_scaling.hpp`). Три из четырёх секций дали не шум, а
воспроизводимый, качественно однозначный результат.

### 1.1. `thr_to_anf` — параллельная версия МЕДЛЕННЕЕ последовательной

| n | single_ms | parallel_ms | speedup |
|---|---|---|---|
| 8 | 0.384 | 6.613 | **0.058** |
| 12 | 5.992 | 42.847 | **0.140** |
| 16 | 65.982 | 111.492 | **0.592** |

При n=8 "параллельная" версия в 17 раз медленнее последовательной. Причина
— в [thr_to_anf.cpp:56-63](thr_to_anf.cpp#L56): быстрый Мёбиус/Жегалкин-
transform распараллеливает ВНЕШНИЙ цикл по блокам (`i`, шаг `2*step`).
На последнем уровне трансформации (`step = size/2`) внешний цикл имеет
**ровно одну итерацию** — весь этот уровень (полноценная `1/log2(size)`-
доля общей работы, т.к. каждый уровень трансформа делает одинаковый объём
работы — `size/2` XOR-операций) выполняется на одном потоке, пока
остальные простаивают. Дополнительно — `log2(size)` отдельных
`#pragma omp parallel for` регионов на один вызов (по одному на уровень
трансформа) означают `log2(size)` fork/join-переходов вместо одного, что
при малых `size` доминирует над самой полезной работой. Фикс —
`collapse(2)` вместо распараллеливания только внешнего измерения, либо явный
выбор оси параллелизма по величине `step` относительно числа блоков.

### 1.2. `thr_to_tt` — то же самое, ещё резче

| n | single_ms | parallel_ms | speedup |
|---|---|---|---|
| 8 | 0.00515 | 3.893 | **0.0013** |
| 12 | 0.180 | 3.206 | **0.056** |
| 16 | 3.792 | 24.588 | **0.154** |

При n=8 — в 754 раза медленнее. [thr_to_tt.cpp:43](thr_to_tt.cpp#L43)
всегда врубает `#pragma omp parallel for` независимо от n, без порога
"слишком мелко — считай последовательно" — при n=8 полезной работы всего
4 блока по 64 бита, а стоимость fork/join OpenMP-региона на этом фоне
доминирует безраздельно. Отдельно (не по этому замеру, а по чтению кода):
горячий цикл [thr_to_tt.cpp:56](thr_to_tt.cpp#L56) пересчитывает полную
взвешенную сумму заново на каждой из 2^n точек (`for v in 0..n`), хотя
собственный doc-комментарий функции ([thr_to_tt.hpp:14-17](thr_to_tt.hpp#L14))
прямо предлагает инкрементальный код Грея как "необязательную оптимизацию"
— не реализовано, хотя именно она убрала бы лишний множитель `n`.

### 1.3. `tt_to_thr` — тот же паттерн, мягче

| n | single_ms | parallel_ms | speedup |
|---|---|---|---|
| 8 | 16.915 | 36.241 | 0.467 |
| 12 | 493.321 | 465.847 | 1.059 |
| 16 | 5658.06 | 5749.08 | 0.984 |

При n=8 параллельная версия вдвое медленнее; к n=16 сходится к
"примерно то на то" — правдоподобно тот же класс причин (накладные расходы
OpenMP-региона на фоне небольшой полезной работы обхода 2^n точек перед
вызовом OR-Tools), но эффект не такой драматичный, как у 1.1/1.2 —
отдельно не диагностировался.

### 1.4. `thr_to_aig_openmp_scaling` — бенчмарк измеряет не ту переменную

| n | single_ms | parallel_ms | speedup |
|---|---|---|---|
| 8 | 0.038 | 0.027 | 1.413 |
| 12 | 0.059 | 0.057 | 1.038 |
| 16 | 0.072 | 0.076 | 0.946 |

Цифры выглядят как "почти без эффекта", но они **не являются валидным
измерением**: `thr_to_aig` распараллелен на 100% через **TBB**
(`tbb::task_group`, см. [thr_to_aig.hpp:40-44](thr_to_aig.hpp#L40) —
конвенция `core/CONVENTIONS.md` §6 правило 2 требует именно TBB для функций,
строящих `Aig`/`Anf` как граф, приоритетнее общего правила "`thr/*` →
OpenMP"). `benchmarks/openmp_scaling.hpp` крутит `omp_set_num_threads()` —
на TBB-арену это не действует никак. Обе половины замера гоняются под
идентичной конфигурацией TBB; разница — чистый шум измерения, не сигнал ни
за, ни против реального параллельного ускорения `thr_to_aig`. В проекте уже
есть правильный инструмент для этого (`benchmarks/tbb_scaling.hpp`,
используется `aig/test_aig.cpp`/`anf/test_anf.cpp`), но для `thr_to_aig` он
не подключён — секция `thr_to_aig_openmp_scaling` в `test_thr.cpp` меряет
не то, что заявляет.

## 2. `thr_to_aig.cpp` — два риска, не подтверждённых (и не опровергнутых) текущими тестами

Оба ниже — статический анализ кода, который **не подтверждён замером**
(см. §1.4 — правильного TBB-харнесса для этой функции сейчас нет) и
**не воспроизведён на текущем тестовом наборе** (`test_thr --section
thr_to_aig` проходит все проверки — ground_truth + metamorphic + SAT —
на `verify::growing_threshold_test_functions(kMaxGroundTruthVars=16)`,
случайные веса из [-5, 5]).

- **Глобальный `std::mutex` вокруг каждого `create_and`**
  ([thr_to_aig.cpp:95-98](thr_to_aig.cpp#L95)). `build_tree` плодит
  TBB-задачи на каждую пару поддеревьев сумматора, но всё вычисление
  (`safe_xor`/`safe_or` сводятся к `safe_and`) держит **один и тот же**
  `ntk_mutex` на каждый AND-гейт — лок обязателен для корректности
  (`mockturtle::aig_network`'овская таблица структурного хэширования не
  потокобезопасна для конкурентной записи), но раз почти вся вычислительная
  работа проходит через него, TBB-параллелизм рискует быть декоративным:
  сериализация плюс накладные расходы планирования задач могут оказаться
  дороже, чем честный последовательный обход. Нужен правильный TBB-замер
  (§1.4), прежде чем делать выводы количественно — а затем, если
  подтвердится, редизайн (шардированное хэширование, независимые локальные
  сети на поддерево со слиянием в конце, или более крупный grain size).
- **Ключ мемоизации полного сумматора теряет полярность сигнала**
  ([thr_to_aig.cpp:115-119](thr_to_aig.cpp#L115)). `FA_Input` строится из
  `ntk.node_to_index(ntk.get_node(a))` для `a`/`b`/`cin` —
  `get_node()` отбрасывает complement-бит. Если где-то в дереве два разных
  вызова `full_adder` дадут одинаковый (по индексам узлов) набор операндов,
  но с разной полярностью — второй вызов молча переиспользует результат
  первого, который для другой полярности неверен. Не триггерится на
  текущем тестовом наборе (см. выше) — нужен целевой adversarial-тест
  (например, конструкция, где промежуточный sum/carry-сигнал на одной ветке
  дерева совпадает по индексу узла, но не по полярности, с сигналом на
  другой ветке), а не просто расширение существующего случайного набора.

## 3. `tt_to_thr.cpp` — гвард `n>=32` не защищает ни от чего практического

[tt_to_thr.cpp:20](tt_to_thr.cpp#L20) пускает `n` вплоть до 31, тогда как
каждая другая n-лимитированная функция этого модуля (`thr_to_tt.cpp:16`,
`thr_to_anf.cpp:30`) капается на `kMaxTruthTableVars=24`
(`core/common.hpp:76`). При n=30-31 код выделяет `std::vector<int8_t>
is_on(2^n)` (~1-2 ГБ) и затем строит **одно ограничение OR-Tools/SCIP на
каждую из до ~2 млрд точек** — реалистичный итог не "долго", а исчерпание
памяти/зависание задолго до завершения. Гвард стоит привести к тому же
`kMaxTruthTableVars`, что и у соседей по папке (см. также §5 — реальные
измерения показывают, что практический предел ещё ниже, около n≈16-20).

Отдельно: doc-комментарий самой функции
([tt_to_thr.hpp:21-24](tt_to_thr.hpp#L21)) рекомендует Chow-параметры
(`verify/metamorphic/metamorphic.hpp::compute_chow_parameters`) как дешёвый
необходимый фильтр, отбраковывающий заведомо непороговые функции ДО дорогого
точного ILP-шага — это не реализовано. Каждый непороговый вход платит полную
цену построения `O(2^n)` ограничений плюс SCIP branch-and-bound, который для
MIP должен ДОКАЗАТЬ инфизибильность (качественно дороже, чем найти
решение), прежде чем вернуть `ErrorCode::Unsupported`.

## 4. `thr_to_bdd.cpp` — слабый комбинатор хэша в ключе мемоизации

[thr_to_bdd.cpp:12-14](thr_to_bdd.cpp#L12): `hash(i) ^ (hash(t) << 1)` —
чистый XOR без умножения/перемешивания, более склонен к коллизиям, чем
стандартный комбинатор (`boost::hash_combine`-стиль: умножение на нечётную
константу + сдвиги). Мелкая находка, но заявленный приоритет — эффективность,
а `unordered_map`-таблица DP здесь может расти до `O(n · Σ|w_i|)` состояний.

## 5. Реальный датасет: `thr/bench_real_corpus.cpp`

В отличие от §1 (синтетический `make_bench_threshold` — все веса единичные)
и `verify::growing_threshold_test_functions` (единичные веса ИЛИ случайные
веса из узкого диапазона [-5, 5]) — ни один из существовавших до этой сессии
тестовых входов не проверял то, о чём прямо предупреждает
[thr_to_bdd.hpp:26-42](thr_to_bdd.hpp#L26): размер BDD зависит не только от
n, но и от того, НАСКОЛЬКО РАЗНЕСЕНЫ веса. Нужен пример с реальными,
содержательными, неравномерными весами.

`bench_real_corpus.cpp` — ПОСТОЯННЫЙ инструмент (не одноразовый, как
`bench_bdd_heuristics.cpp` и подобные в истории `anf/` — тот же принцип, что
`anf/bench_real_corpus.cpp`), источник — официально опубликованная
взвешенная система голосования: **Коллегия выборщиков США, 2024-2028**
(веса по переписи 2020 года). 50 штатов + округ Колумбия = 51 "переменная",
вес — число голосов выборщиков штата, функция — `sum(w_i·x_i) >= 270` из
538 (простое большинство). Источник — National Archives,
https://www.archives.gov/electoral-college/allocation (сверено WebFetch при
подготовке корпуса; сумма всех 51 веса вручную перепроверена — равна 538).

`n=51 > kMaxTruthTableVars(24)` — `thr_to_anf`/`thr_to_tt`/`tt_to_thr` не
могут принять этот вход целиком, поэтому полная функция прогоняется только
через `thr_to_aig`/`thr_to_bdd`. Для остальных трёх используются РЕАЛЬНЫЕ
ПРЕФИКСЫ того же списка (первые n штатов по таблице источника, те же веса)
с порогом, пересчитанным как простое большинство именно этого префикса.

### 5.1. Полная функция (n=51) — обе неограниченные по n функции полиномиальны на реальных весах, как и предсказано

| функция | размер | время |
|---|---|---|
| `thr_to_aig` | 1235 гейтов | 0.35 мс |
| `thr_to_bdd` | 6720 узлов | 6.99 мс |

Подтверждает doc-предупреждение `thr_to_bdd.hpp` от противного: веса
Коллегии выборщиков (3-54) не "экспоненциально разнесены" — BDD остаётся
маленьким.

### 5.2. Растущие реальные префиксы — все 5 функций

| n | `thr_to_aig` | `thr_to_bdd` | `thr_to_anf` (мономов) | `thr_to_tt` | `tt_to_thr` |
|---|---|---|---|---|---|
| 8 | 0.37мс | 0.009мс | 191мс (1) | 46.7мс | 88.4мс |
| 12 | 0.046мс | 0.130мс | 225мс (1 111) | 6.05мс | 549мс |
| 16 | 0.861мс | 0.335мс | 657мс (17 447) | 21.0мс | **11 627мс** |
| 20 | 0.156мс | 0.250мс | 2 506мс (279 069) | 21.9мс | пропущено (см. ниже) |
| 24 | 0.326мс | 0.601мс | **68 621мс** (4 411 919) | 272мс | пропущено |

`thr_to_aig`/`thr_to_bdd` остаются под миллисекундой и не монотонны по n
(например n=8 медленнее n=12) — на этих размерах время целиком в шуме
измерения, полиномиальность подтверждена размером результата (гейты/узлы),
не временем.

### 5.3. `thr_to_anf` на плотных реальных входах — первый эмпирический сигнал в пользу гипотезы `anf/README.md` §9.3

Пороговые функции по определению имеют плотный ANF без известной компактной
формулы для произвольных весов ([thr_to_anf.hpp:13-18](thr_to_anf.hpp#L13))
— именно тот "по-настоящему плотный вход при n=20-24", которого не хватало
`anf/`-корпусу, чтобы проверить гипотезу из `anf/README.md` §9.3 про
потенциально квадратичную сборку `poly += mono`. Число мономов растёт
закономерно — ×~16 на каждые +4 переменные (1 → 1111 → 17447 → 279069 →
4411919), т.е. функция почти максимально плотная. Удельное время на моном:

| n | мономов | время/моном |
|---|---|---|
| 12 | 1 111 | 202 мкс |
| 16 | 17 447 | 38 мкс |
| 20 | 279 069 | **9.0 мкс** |
| 24 | 4 411 919 | **15.6 мкс** |

От n=12 до n=20 удельная стоимость ПАДАЕТ (фиксированные накладные расходы
Мёбиус-трансформа/OpenMP-регионов — §1.1 — амортизируются на растущем числе
мономов). На n=24 она РАЗВОРАЧИВАЕТСЯ и растёт на 73% (9.0 → 15.6 мкс/моном)
— первый реальный (не гипотетический) сигнал в пользу того, что сборка
`poly += mono` в `thr_to_anf.cpp` (тот же паттерн, что и в
`anf/tt_to_anf.cpp`, где гипотеза впервые появилась) становится дороже на
моном при очень большом числе мономов, а не остаётся линейной. Одной точки
перегиба недостаточно, чтобы установить точную степень роста — нужны ещё
точки за пределами `kMaxTruthTableVars=24`, которых по контракту функции
здесь получить нельзя.

### 5.4. `tt_to_thr` — практический предел ILP/SCIP на реальных весах: n≈16, не n=32

| n | `tt_to_thr` | рост за +4 к n |
|---|---|---|
| 8 | 88.4мс | — |
| 12 | 549мс | ×6.2 |
| 16 | **11 627мс** | ×21.2 |
| 20 | не завершилось в разумное время (прогон остановлен вручную) | — |

Для сравнения — `thr_to_tt` на тех же n растёт куда мягче (46.7 → 6.1 → 21.0
→ 21.9 мс, т.е. фактически плоско). Рост времени `tt_to_thr` явно
сверхэкспоненциален по n — SCIP branch-and-bound на реальных (не
специально сконструированных для лёгкого решения) весах ведёт себя заметно
хуже, чем 2^n сам по себе. Практический предел применимости этой функции —
около n≈16, на два порядка ниже формального гварда `n>=32` (см. §3). В
`bench_real_corpus.cpp` `tt_to_thr` поэтому пропускается для n>16 —
тот же приём, что `anf/bench_real_corpus.cpp` использует для пропуска
`anf_to_bdd` на `bent_mm(n>32)`.

## 6. Как повторить

Корректность (5 секций `test_thr`, включая синтетический OpenMP-бенчмарк
§1):
```
MSYS_NO_PATHCONV=1 docker run --rm -v "D:/Proga/web/bmm-translib:/workspace" -w /workspace genetica-boolean-lib:latest bash -c "cmake --build build --target test_thr -j\$(nproc) && ./build/test_thr"
```

Реальный корпус (§5, Коллегия выборщиков):
```
MSYS_NO_PATHCONV=1 docker run --rm -v "D:/Proga/web/bmm-translib:/workspace" -w /workspace genetica-boolean-lib:latest bash -c "cmake --build build --target bench_real_corpus_thr -j\$(nproc) && ./build/bench_real_corpus_thr"
```

Прогон §5 занимает ориентировочно 15-20 минут — доминирует одна точка
(`thr_to_anf` на n=24, ~69 секунд × 12 повторов). `tt_to_thr` для n>16
пропускается кодом автоматически (см. §5.4) — не нужно останавливать
процесс вручную, если не изменять диапазон n в `run_all()`.
