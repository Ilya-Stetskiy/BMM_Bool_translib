# thr/ — состояние, качество, лучшие практики, находки

Пять функций трансляции с `Thr` (пороговая функция: веса `w_1..w_n` +
порог `theta`, `f(x) = [Σw_i·x_i ≥ theta]`) на входе или выходе:
`tt_to_thr`, `thr_to_aig`, `thr_to_bdd`, `thr_to_anf`, `thr_to_tt`. Этот
файл — не API-документация (см. doc-комментарии в `.hpp`) и не журнал
изменений (`git log`), а полная инженерная картина: что каждая функция
реально делает, что в ней сделано хорошо, что — находки/риски, как она
ведёт себя на реальных данных, а не только на синтетике.

Методология везде одна: корректность — `ctest`/`test_thr` (exhaustive
ground_truth + metamorphic на n≤`kMaxGroundTruthVars`=16, для `thr_to_aig`
дополнительно независимая SAT-проверка); производительность — либо
встроенный синтетический OpenMP/TBB-бенчмарк `test_thr.cpp`, либо
постоянный корпус реальных данных `thr/bench_real_corpus.cpp` (Коллегия
выборщиков США). Все числа ниже получены реальной сборкой в docker-образе
проекта (`genetica-boolean-lib:latest`), не оценены на глаз.

## 2. `thr_to_tt` — таблица истинности прямым перебором

**Алгоритм**: тривиальная точечная оценка `f(x) = [Σw_i·x_i ≥ theta]` по
всем `2^n` точкам, блоками по 64 бита (один блок = одно 64-битное слово
`TruthTable`). [thr_to_tt.cpp](thr_to_tt.cpp).

**Сделано хорошо**:
- Горячий цикл работает с сырым указателем на 64-битные слова
  (`tt_data[block_idx] = current_block`), без объектов/указателей внутри —
  ровно то, что требует doc-комментарий как задел под перенос на GPU.
- Потокобезопасность обоснована явно в коде и верна: каждый OpenMP-поток
  пишет в свой, непересекающийся индекс блока — никаких блокировок не
  нужно, и их нет.
- Корректно обработаны краевые случаи `n=0` и `n<6` (суб-словный случай,
  отдельный `limit`).
- Простая, читаемая, полностью соответствует собственному doc-комментарию
  ("самая простая из 20 функций проекта").

**Находки**:
- Пересчитывает полную взвешенную сумму `Σw_i·x_i` заново на каждой из
  `2^n` точек (`for v in 0..n`), хотя doc-комментарий прямо предлагает
  инкрементальный код Грея как **необязательную** оптимизацию — не
  реализовано. Лишний множитель `n` в горячем цикле.
- `#pragma omp parallel for` без порога по размеру (в отличие от
  `thr_to_anf` на ветке `Thr`, см. §6) — на малых n (единицы-десятки
  блоков) накладные расходы OpenMP-региона доминируют. Встроенный
  синтетический бенчмарк (`make_bench_threshold`, majority-подобная
  функция) это подтверждает: speedup=**0.0013 при n=8** (0.005мс
  последовательно против 3.9мс "параллельно" — в 754 раза медленнее),
  0.056 при n=12, 0.154 при n=16.

**Производительность (реальный корпус, Коллегия выборщиков)**:

| n | 8 | 12 | 16 | 20 | 24 |
|---|---|---|---|---|---|
| время | 12-47мс | 2-6мс | 2-40мс | 18-45мс | 250-320мс |

(разброс — три независимых прогона корпуса в разное время, числа для n≤16
в основном шум измерения на суб-миллисекундной задаче; на n=24 стабильно
250-320мс).

## 3. `tt_to_thr` — распознавание пороговости через ILP

**Алгоритм**: классическая редукция Мурога (Muroga, 1971) — для каждой из
`2^n` точек таблицы истинности строится одно ограничение ILP-модели
(`Σw_i·x_i - T ≥ 0` для единичных точек, `≤ -1` для нулевых), решает Google
OR-Tools/SCIP. [tt_to_thr.cpp](tt_to_thr.cpp).

**Сделано хорошо**:
- Параллельный проход по таблице истинности сделан аккуратно: каждый
  OpenMP-поток получает **свой** `Assignment` (объявлен внутри
  `#pragma omp parallel`, до `#pragma omp for`) — не переиспользует общий
  объект между потоками, никакой false sharing.
- `int8_t is_on(num_minterms)`, не `std::vector<bool>` — тот же
  сознательный выбор, что и в `thr_to_anf`/`thr_to_tt` (упаковка бит в
  `vector<bool>` ломает потокобезопасность конкурентной записи); здесь это
  явно прокомментировано и последовательно.
- Корректная ориентация ограничений (строгое `≥`/`≤ -1` для целых весов,
  не `>`/`<`, что было бы неверно для целочисленной модели).
- Graceful degradation: непороговая функция возвращает `ErrorCode::Unsupported`
  (не падение, не `NotImplemented`) — ровно то, что требует doc-комментарий.

**Находки**:
- Гвард `n >= 32` (core/common.hpp `kMaxTruthTableVars=24` используется
  везде в модуле, кроме этой функции) не защищает ни от чего практического:
  при n=30-31 код выделяет `std::vector<int8_t>` на ~1-2 ГБ и пытается
  построить до ~2 млрд строк ILP-ограничений.
- doc-комментарий самой функции рекомендует Chow-параметры
  (`verify/metamorphic/metamorphic.hpp::compute_chow_parameters`) как
  дешёвый необходимый фильтр, отбрасывающий заведомо непороговые функции
  ДО дорогого точного ILP-шага — не реализовано. Каждый непороговый вход
  платит полную цену построения `O(2^n)` ограничений плюс SCIP
  branch-and-bound, который для MIP обязан ДОКАЗАТЬ инфизибильность.
- **Практический потолок — n≈16, а не n=32 и не даже n=24.** На реальных
  (не подогнанных под лёгкое решение) весах Коллегии выборщиков:

  | n | 8 | 12 | 16 |
  |---|---|---|---|
  | время | 88.4мс | 549мс | **11 627мс** |

  Рост между точками — ×6.2, затем ×21.2 — явно сверхэкспоненциален по n.
  При n=20 один вызов не уложился в разумное время (прогон остановлен
  вручную после нескольких минут). Для сравнения — встроенный синтетический
  бенчмарк (majority-функция, легко решаемая для солвера) даёт куда более
  мягкую картину (n=16 — единицы секунд, не 11.6с) — **синтетика
  систематически недооценивает реальную стоимость этой функции**, метод
  тестирования тоже находка.

## 4. `thr_to_bdd` — BDD через DP по частичным суммам

**Алгоритм**: рекурсия с мемоизацией по паре (индекс переменной, остаточный
порог) — классическая техника представления псевдобулева ограничения через
BDD (Behle 2007-2008, Hosaka et al. 1997). [thr_to_bdd.cpp](thr_to_bdd.cpp).

**Сделано хорошо**:
- Bounding через предрасчитанные `max_remain`/`min_remain` — до входа в
  рекурсию отсекает заведомо-true/заведомо-false ветки без построения
  под-BDD; реальная, не косметическая оптимизация.
- Ключ мемоизации — ровно пара `(i, current_t)`, как требует doc-комментарий
  (не только индекс переменной — разные остаточные бюджеты на одном уровне
  дают разные под-BDD).
- Корректно следует приоритетному правилу параллелизма (`CONVENTIONS.md`
  §6, правило 1): `Bdd` — выход, значит только Sylvan/Lace, никакого
  собственного TBB/OpenMP — в коде нет ни одной попытки завести свой
  тред-пул.
- `n=0` обработан отдельно и корректно.

**Находки**:
- Комбинатор хэша `hash(i) ^ (hash(t) << 1)` — чистый XOR без
  умножения/перемешивания, более склонен к коллизиям, чем стандартный
  `hash_combine`. Мелкая находка, но таблица DP здесь может расти до
  `O(n · Σ|w_i|)` состояний.
- Рекурсия не ограничена по глубине явно — потенциальный риск переполнения
  стека на очень больших n с "плохими" весами; не наблюдалось на
  практике (включая n=51 в реальном корпусе).

**Производительность (реальный корпус)**: на полной функции (n=51,
веса — реальные голоса выборщиков, 3-54) — **6720 узлов, 3.4-7.0мс**.
Подтверждает doc-предупреждение от противного: веса Коллегии выборщиков не
"экспоненциально разнесены" — BDD остаётся маленьким, ровно как и
предсказывает теория для "практических" весов.

## 5. `thr_to_aig` — AIG через дерево сумматоров + компаратор

**Алгоритм**: приведение отрицательных весов к неотрицательным (`x_i →
1-x_i` с коррекцией `theta`), дерево полных сумматоров (TBB `task_group`,
рекурсивное разбиение) для `Σw_i·x_i`, затем bit-serial компаратор
результата с `theta`. Каждый элемент — через AND+инвертор (De Morgan), без
обращения к BDD (собственный doc-комментарий явно предупреждает: путь через
`thr_to_bdd`+`bdd_to_aig` не гарантированно полиномиален, этот путь —
гарантированно). [thr_to_aig.cpp](thr_to_aig.cpp).

**Сделано хорошо**:
- Математически корректная редукция знака весов — единственная функция
  модуля, чью корректность подтверждает **независимая SAT-эквивалентность**
  в дополнение к ground_truth/metamorphic (`BMM_STATUS thr_to_aig PASS
  ... + sat_encoding`) — самая строгая верификационная планка из всех 5.
- Раннее отсечение констант (`theta_adj <= 0` → `true`,
  `theta_adj > max_sum` → `false`) — вырожденные случаи не строят вообще
  никакого дерева сумматоров.
- Гарантированный размер `O(n·B)` независимо от конкретного распределения
  весов — архитектурно именно то, ради чего этот путь существует вместо
  BDD, и код этому верен (никогда не идёт через `Bdd`).
- Мемоизация полного сумматора (`FA_Input` → `(sum, cout)`) переиспользует
  одинаковые под-вычисления вместо повторного построения.
- Bit-serial компаратор (LSB→MSB, `ge = AND/OR` в зависимости от бита
  `theta`) — на первый взгляд не совпадает с описанием в doc-комментарии
  ("от старшего к младшему"), но математически проверен вручную (индукция
  по битам) и корректен — это известный приём, просто с направлением
  обхода, отличным от буквального текста комментария.

**Находки**:
- Каждое создание AND-гейта (`safe_and`) держит **общий** `std::mutex` —
  лок обязателен для корректности (`mockturtle::aig_network`'овская таблица
  структурного хэширования не потокобезопасна для конкурентной записи), но
  раз почти вся вычислительная работа идёт через него, TBB-параллелизм
  дерева сумматоров рискует быть декоративным — сериализация плюс
  накладные расходы планирования задач могут оказаться дороже честного
  последовательного обхода.
- Ключ мемоизации полного сумматора (`ntk.node_to_index(ntk.get_node(a))`)
  теряет полярность (complement-бит) сигнала — теоретический риск неверного
  переиспользования гейта при совпадении узлов разной полярности в разных
  вызовах. **Не подтверждено**: `test_thr --section thr_to_aig` проходит
  все проверки на случайных весах `[-5,5]`, n≤16 — нужен целевой
  adversarial-тест, не просто более широкий случайный набор.
- **Штатный бенчмарк `thr_to_aig_openmp_scaling` измеряет не ту переменную.**
  `thr_to_aig` распараллелен через **TBB**, а `benchmarks/openmp_scaling.hpp`
  крутит `omp_set_num_threads()` — на TBB-арену это не действует. Обе
  половины замера идут под идентичной конфигурацией TBB; цифры "speedup"
  (1.41/1.04/0.95 на n=8/12/16) — чистый шум, не сигнал ни за, ни против
  реального параллельного ускорения. Значит два предыдущих пункта до сих
  пор **не измерены честно** — в проекте уже есть правильный инструмент
  (`benchmarks/tbb_scaling.hpp`, используется `aig/test_aig.cpp`), но для
  `thr_to_aig` он не подключён.

**Производительность (реальный корпус)**: на полной функции (n=51) —
**1235 гейтов, 0.35-1.6мс**. Подтверждает гарантированную полиномиальность
на практике; единственная функция, чью параллельную эффективность нельзя
пока честно оценить (см. находку выше).

## 6. `thr_to_anf` — таблица истинности + Мёбиус + сборка ANF

**Алгоритм**: полный перебор `2^n` точек в таблицу истинности → быстрое
преобразование Мёбиуса/Жегалкина in-place (butterfly-паттерн) → сборка
`BoolePolynomial`/`AnfFallback` из полученных коэффициентов.
[thr_to_anf.cpp](thr_to_anf.cpp) — **на `master` до сих пор старая версия
с двумя перформанс-багами**, оба независимо исправлены и подтверждены на
ветке `origin/Thr` (коммиты `130a55c`/`00e726a`), но ветка ещё не
смержена. Ниже — оба состояния.

### 6.1. Что было (master) — два реальных перф-бага

- **OpenMP без порога по размеру, на ОБОИХ шагах** (построение ТТ и
  Мёбиус) — при n=8 (2^8=256 точек) накладные расходы fork/join
  доминируют. Встроенный бенчмарк: speedup=**0.058 при n=8** (в 17 раз
  медленнее последовательной версии), 0.14 при n=12, 0.59 при n=16.
- **Неверная ось параллелизма в Мёбиус-трансформе.** `#pragma omp parallel
  for` был на ВНЕШНЕМ цикле по блокам (шаг `2*step`); на последнем уровне
  трансформации (`step=size/2`) внешний цикл имел РОВНО одну итерацию —
  весь этот уровень (полноценная `1/log2(size)`-доля общей работы)
  выполнялся на одном потоке, пока остальные простаивали. Плюс —
  `log2(size)` отдельных `#pragma omp parallel for` регионов на один
  вызов вместо одного.
- Построение каждого монома через n последовательных умножений
  (`m = m * ring.variable(b)`) — O(n) операций `BoolePolynomial::operator*`
  на **каждый** из до 4.4 млн мономов при n=24.

### 6.2. Что исправлено (origin/Thr, проверено этой сессией)

1. `#pragma omp parallel for if(size > 65536)` (n>16) — отсекает OpenMP
   там, где он был чистым вредом; единый `#pragma omp parallel` +
   `#pragma omp for collapse(2)` вместо N раздельных regions на шаг
   трансформа — устраняет и накладные расходы, и дисбаланс последнего
   уровня.
2. Построение монома напрямую через `polybori::BooleExponent` (вектор
   индексов переменных) + `BooleMonomial(vars, ring)` за один вызов вместо
   n умножений.

Оба подтверждены: `test_thr` PASS (ground_truth+metamorphic), число
мономов на реальном корпусе идентично исходной версии на каждом n (1 /
1111 / 17447 / 279069 / 4411919) — результат не изменился, изменилась
только скорость:

| n | было (master) | + `collapse(2)` | + `BooleExponent` (origin/Thr сейчас) |
|---|---|---|---|
| 8 | 191мс | 0.33мс | 0.45мс |
| 12 | 225мс | 5.03мс | 4.16мс |
| 16 | 657мс | 109мс | 65.0мс |
| 20 | 2506мс | 2378мс | 1635мс |
| 24 | 68621мс | 57773мс | **31976мс** |

Удельная стоимость на моном (n=20→24) тоже смягчилась: было 9.0→15.6
мкс/моном (+73%), стало 5.86→7.25 мкс/моном (+24%) — часть сверхлинейного
роста была именно в стоимости построения монома (растёт вместе с n), не в
самом ZDD-объединении.

**Рекомендация**: смержить оба исправления из `origin/Thr` в `master` —
оба строго лучше, корректность перепроверена, регрессий не найдено.

### 6.3. Отклонённый эксперимент — дерево слияния вместо линейного fold

По итогам §6.2 логичным следующим шагом выглядело заменить линейный fold
`poly = ((...(0+m1)+m2)...)+mk` на сбалансированное дерево слияния (тот же
паттерн, что `add_bitvectors`/`build_tree` в `thr_to_aig.cpp`). Измерено
(локальный эксперимент, не запушен):

| n | линейный fold | дерево слияния | разница |
|---|---|---|---|
| 8 | 0.45мс | 0.37мс | ≈шум |
| 12 | 4.16мс | 3.00мс | -28% |
| 16 | 65.0мс | 55.2мс | -15% |
| 20 | 1635мс | 1725мс | +5.5% |
| 24 | 31976мс | **42108мс** | **+32%** |

Дерево выигрывает на малых-средних n и проигрывает именно там, где стоял
вопрос — на n=24 оно на треть медленнее. Правдоподобная причина:
`BoolePolynomial::operator+` для ОДНОГО нового монома у PolyBoRi, судя по
всему, имеет специализированный дешёвый путь (вставка ~O(n) за счёт
structural sharing, независимо от размера существующего полинома) — то же
объясняет, почему изолированный замер в `anf/README.md` §9.5 увидел
линейный, не квадратичный рост сборки. А union двух СРАВНИМО БОЛЬШИХ
полиномов — это уже generic ZDD `Apply`, который на операцию дороже, чем
серия дешёвых point-вставок, даже при вдвое меньшем числе операций на
уровень дерева. Дерево меняет много дешёвых вставок на меньшее число
дорогих union — невыгодный обмен именно там, где общая масса работы
велика. TBB поверх дерева не пробовался — раз последовательная форма уже
проиграла, а BRiAl/CUDD к тому же не потокобезопасен (тот же вывод
независимо получен для `aig_to_anf.cpp` — ThreadSanitizer подтвердил гонку
в `CCuddCore::release()`).

**Вывод**: для сборки `BoolePolynomial` из массива Жегалкин-коэффициентов
линейный fold с прямым построением монома (§6.2, уже в `origin/Thr`)
остаётся лучшим известным вариантом. Дерево слияния здесь — тупиковый путь
для ЭТОЙ операции, не общая ошибка идеи "дерево вместо fold" (в
`thr_to_aig.cpp`, где на каждом уровне сливаются равномерные битовые
вектора одинакового размера, тот же паттерн реально работает — там нет
конкурирующего "дешёвого пути" для one-off вставки). Не реализовывалось и
не пушилось — зафиксировано здесь как отрицательный результат.

## 7. Сквозные лучшие практики модуля

Наблюдения, повторяющиеся во всех (или почти всех) 5 функциях — не
случайность, а последовательно применяемый стиль:

- **`int8_t`, не `std::vector<bool>`, в каждом OpenMP-горячем плоском
  массиве** (`tt_to_thr.cpp`, `thr_to_anf.cpp`) — осознанный, явно
  прокомментированный выбор: битовая упаковка `vector<bool>` ломает
  потокобезопасность конкурентной записи по независимым логическим
  индексам.
- **`Result<T>`-дисциплина**: исключения не пересекают границу функции,
  кроме единственного разрешённого случая (`CONVENTIONS.md` §2а —
  `catch(bad_alloc)` → `ErrorCode::OutOfMemory`) — соблюдается во всех 5
  функциях (хотя `thr_to_anf.cpp` сейчас перехватывает несколько шире
  буквы конвенции — `catch(std::exception&)`, не только `bad_alloc`; на
  практике не новая проблема, тот же уровень отклонения был и до
  правок ветки `Thr`, просто сообщение об ошибке стало информативнее за
  счёт `e.what()`).
- **Единообразная обработка `n=0`** — базовый случай константной функции
  явно выделен и корректно обработан в каждой из 5 функций, ни одна не
  полагается на то, что общий код "случайно" сработает для вырожденного
  входа.
- **`kMaxTruthTableVars=24`** — согласованный лимит в `thr_to_tt.cpp` и
  `thr_to_anf.cpp` (единственное исключение — `tt_to_thr.cpp` с `n>=32`,
  см. §3, задокументированная находка, не стиль).
- **`ZoneScoped`/Tracy** — профилирующий макрос присутствует в каждой
  функции, единообразная точка входа для профилирования всего модуля.
- **Приоритетное правило параллелизма (`CONVENTIONS.md` §6) соблюдено
  без исключений**: `thr_to_bdd` — только Sylvan/Lace (никакой попытки
  завести свой тред-пул); `thr_to_aig` — TBB, потому что строит `Aig` как
  граф (приоритетнее общего "`thr/*` → OpenMP"); `thr_to_tt`/`tt_to_thr`/
  `thr_to_anf` — OpenMP над плоским массивом. Все пять функций правильно
  определяют, к какой категории относятся, ни одна не нарушает приоритет.
- **Гарантия сложности вместо "среднего случая"**: `thr_to_aig` и
  `thr_to_tt`/`tt_to_thr` дают предсказуемую (полиномиальную по входу)
  сложность независимо от конкретных весов — это архитектурный выбор,
  явно объяснённый в doc-комментариях (`thr_to_aig.hpp` прямо
  предупреждает не идти через `Bdd` ради этой гарантии), а не побочный
  эффект простоты реализации.

## 8. Реальный датасет: `thr/bench_real_corpus.cpp`

Синтетический набор в `verify::growing_threshold_test_functions()` — либо
единичные веса (AND/OR/Majority), либо случайные веса из узкого диапазона
[-5, 5]. Ни один из этих случаев не проверяет то, о чём прямо предупреждает
[thr_to_bdd.hpp](thr_to_bdd.hpp): размер BDD зависит не только от n, но и
от того, НАСКОЛЬКО РАЗНЕСЕНЫ веса. Нужен пример с реальными, содержательными,
неравномерными весами — и синтетика, как показали §3/§6, ещё и системно
занижает реальную стоимость там, где сложность зависит от структуры данных
(ILP-solve, плотность ANF), а не только от n.

`bench_real_corpus.cpp` — ПОСТОЯННЫЙ инструмент (не одноразовый), источник —
официально опубликованная взвешенная система голосования: **Коллегия
выборщиков США, 2024-2028** (веса по переписи 2020 года). 50 штатов + округ
Колумбия = 51 "переменная", вес — число голосов выборщиков штата, функция —
`sum(w_i·x_i) >= 270` из 538 (простое большинство). Источник — National
Archives, https://www.archives.gov/electoral-college/allocation (сверено
WebFetch при подготовке корпуса; сумма всех 51 веса вручную перепроверена —
равна 538).

`n=51 > kMaxTruthTableVars(24)` — `thr_to_anf`/`thr_to_tt`/`tt_to_thr` не
могут принять этот вход целиком, поэтому полная функция прогоняется только
через `thr_to_aig`/`thr_to_bdd` (§6.1 в таблице выше). Для остальных трёх
используются РЕАЛЬНЫЕ ПРЕФИКСЫ того же списка (первые n штатов по таблице
источника, те же веса) с порогом, пересчитанным как простое большинство
именно этого префикса — веса по-прежнему настоящие, урезан только n. Все
числа по каждой функции — в соответствующих разделах §2-§6 выше.

## 9. Как повторить

Корректность (5 секций `test_thr`, включая синтетический OpenMP-бенчмарк):
```
MSYS_NO_PATHCONV=1 docker run --rm -v "D:/Proga/web/bmm-translib:/workspace" -w /workspace genetica-boolean-lib:latest bash -c "cmake --build build --target test_thr -j\$(nproc) && ./build/test_thr"
```

Реальный корпус (§8, Коллегия выборщиков):
```
MSYS_NO_PATHCONV=1 docker run --rm -v "D:/Proga/web/bmm-translib:/workspace" -w /workspace genetica-boolean-lib:latest bash -c "cmake --build build --target bench_real_corpus_thr -j\$(nproc) && ./build/bench_real_corpus_thr"
```

Прогон корпуса занимает ориентировочно 15-20 минут — доминирует одна точка
(`thr_to_anf` на n=24). `tt_to_thr` для n>16 пропускается кодом
автоматически (см. §3) — не нужно останавливать процесс вручную, если не
менять диапазон n в `run_all()`.

Числа §6.1/§6.2/§6.3 (ветка `Thr`, исправления `thr_to_anf`) получены на
`origin/Thr` — чтобы воспроизвести, нужно временно подложить
`thr/thr_to_anf.cpp` из этой ветки поверх `master` перед сборкой (сама
ветка не собирается как единое целое с текущим `master` — разошлась раньше
слияния с `anf`/`aig`).
