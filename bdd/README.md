# bdd/ — заметки по реализации

Пять функций трансляции с `Bdd` (`sylvan::Bdd`) на входе или выходе:
`bdd_to_aig`, `bdd_to_anf`, `bdd_to_thr`, `bdd_to_tt`, `tt_to_bdd`. Этот
файл — не описание API (см. doc-комментарии в соответствующих `.hpp`) и не
журнал изменений (см. `git log`), а **рассуждения и измерения**, которые не
выводятся напрямую из кода: результаты код-ревью этой папки — как на
`master`, так и на ветке студента `egor`, которая на момент написания этого
файла ещё не смержена, — плюс баг, найденный и исправленный по ходу ревью, и
конструктивный план того, что делать дальше.

**Важная оговорка про структуру этого файла.** Обычно такой README (см. по
аналогии [`thr/README.md`](../thr/README.md)) описывает то, что реально есть
на `master`. Здесь ситуация асимметричная: `bdd_to_aig` и `tt_to_bdd` на
`master` полностью реализованы и хорошего качества; `bdd_to_tt` на `master`
рабочий, но с известной проблемой производительности; `bdd_to_anf` и
`bdd_to_thr` на `master` — **честные заглушки** (`ErrorCode::NotImplemented`).
Настоящая работа над этими тремя функциями идёт на ветке `origin/egor`
(автор — student4), которая на n-ное число коммитов разошлась с `master` и
пока не смержена. Поэтому ниже для каждой функции явно указано, к какому
состоянию/ветке/коммиту относится описание — не путайте "review of egor" с
"текущее состояние master".

## 1. `bdd_to_aig` — эталонное качество (`master`)

[bdd_to_aig.hpp](bdd_to_aig.hpp) требует: структурный обход BDD в AIG через
де Морган (MUX через 3 AND), итеративно (не рекурсивно), с мемоизацией по
**каноническому** (без учёта complement-бита) узлу, и обязательным
`try/catch(std::bad_alloc&)` (`CONVENTIONS.md` п.2а — единственная функция
модуля, где раздутый порядок переменных BDD может дать неограниченно
большой AIG).

**Сделано хорошо** — и на `master`, и на ветке `egor` (это две независимо
написанные, но эквивалентные по сути реализации):
- Итеративный обход через явный стек с двумя фазами (pre-order —
  раскрыть узел, post-order — построить сигнал) — ровно тот паттерн,
  который требует doc-комментарий, никакой рекурсии.
- Мемоизация по каноническому узлу (`raw & ~COMP_BIT`), инверсия полярности
  применяется к сигналу **после** выборки из кэша, а не заводится отдельный
  кэш на каждую полярность — именно то, о чём явно предупреждает
  doc-комментарий как о частой ошибке, которую здесь избежали.
- `try/catch(std::bad_alloc&)` на границе функции — сделано, как требует
  `CONVENTIONS.md` п.2а.
- (ветка `egor`) Хранит узлы как голые `uint64_t` и переиспользует
  `phmap::flat_hash_map` вместо `std::unordered_map` — чуть быстрее за счёт
  меньшего числа аллокаций; безопасность этого проверена отдельно: корень
  (`f_syl`) жив всё время работы функции, поэтому Sylvan транзитивно держит
  живыми всех потомков, и реконструкция `sylvan::Bdd` из голого хэндла между
  фазами не рискует garbage collection.

**Единственная стилевая придирка** (не баг, обе ветки): `COMP_BIT =
0x8000000000000000ULL` — ручное дублирование внутренней константы Sylvan
(`sylvan_complement` в `sylvan_mtbdd.h`). Работает корректно (константа
верная), но стоит явно сослаться на источник в комментарии — иначе при
смене версии Sylvan это будет неочевидно сломать.

Багов не найдено ни статическим чтением, ни прогоном `test_bdd`
(`BMM_STATUS bdd_to_aig PASS ... + sat_encoding` — единственная функция
модуля этой папки, чью корректность дополнительно подтверждает независимая
SAT-эквивалентность через `verify/sat_encoding`).

## 2. `tt_to_bdd` — без замечаний (`master`, идентично на `egor`)

[tt_to_bdd.hpp](tt_to_bdd.hpp): классическое разложение Шеннона
(`sylvan::Bdd::Ite`), Sylvan сам дедуплицирует одинаковые поддеревья через
unifying table.

Реализация — рекурсия с собственным кэшем `(var, block_start) -> Bdd`
(`phmap::flat_hash_map`) поверх штатной дедупликации Sylvan. Кэш не
обязателен (Sylvan и так дедуплицирует), но и не вреден — экономит повторные
рекурсивные спуски. Упаковка ключа кэша (`var<<56 | block_start`) не
переполняется при `n <= kMaxTruthTableVars = 24` — блок влезает в 56 бит с
большим запасом. Замечаний нет; файл не менялся между `master` и `egor` (`git
diff` не показывает изменений).

## 3. `bdd_to_tt` — на `master` работает, но противоречит собственной
   спецификации; на `egor` доведено до правильной асимптотики за 3 итерации

### 3.1. Что на `master` сейчас

[bdd_to_tt.hpp:28-41](bdd_to_tt.hpp) прямо предупреждает: "ЧАСТАЯ ОШИБКА...
наивный BOTTOM-UP обход НЕ снижает асимптотику для ПОЛНОЙ материализации
TT" — требуется **top-down/диапазонное** разбиение с `memset`/`fill` на
терминалах, а не поточечный обход.

[bdd_to_tt.cpp](bdd_to_tt.cpp) делает именно то, от чего предостерегали,
просто в другой форме — не bottom-up, а **точка за точкой**: для каждого из
`2^n` минтермов — независимый спуск от корня. Комментарий в самом коде это
признаёт: "Простая и надёжная реализация... для n<=24 это занимает ~0.1-0.2
секунды" — осознанный временный костыль, не недосмотр.

Асимптотика — Θ(n·2^n) вместо Θ(2^n): при n=24 (`kMaxTruthTableVars`) это
~400 млн шагов вместо ~16 млн записей бит, и каждый шаг — конструирование/
разрушение RAII-обёртки `sylvan::Bdd` (атомарный рефкаунтинг), а не один
проход с `memset` на терминалах. Функционально корректно, но реализация
прямо противоречит задокументированному для неё же алгоритму.

### 3.2. Эволюция на ветке `egor` — три итерации до правильной асимптотики

**Итерация 1 (`c281baf`, "bdd_to_tt update").** Полностью новый алгоритм:
единый DFS по структуре BDD со стеком кадров `{node, mask, value}` (`mask`
— какие переменные уже зафиксированы на пути, `value` — их значения).
Обрыв на терминале-0 (`continue`) и батч-заполнение на терминале-1 (перебор
подмасок незафиксированных бит, `(sub-1)&floating`) происходят СРАЗУ по
достижении терминала, не дожидаясь фиксации всех n переменных — суммарное
число "развёрток" узлов ограничено числом листьев дерева решений глубины
≤n, то есть ≤2^n, а не n·2^n. БДД (`Then`/`Else`) на этапе заполнения бит
больше не трогается вообще — лишний множитель n от повторных обращений к
Sylvan/RAII-рефкаунтингу пропал. Ровно то, чего требовал doc-комментарий,
просто не буквально через `memset`, а через битовый трюк перечисления
подмасок.

**Итерация 2 (`e559302`, "bdd_to_tt weak version").** Дальнейшая
оптимизация терминала-1: вместо установки бит по одному даже внутри
свободного поддерева, код детектирует **непрерывные блоки полностью
свободных 64-битных слов** и мержит их в линейный `for`-цикл записи одним и
тем же `word_mask` подряд по нескольким словам (bit-trick
`(~free_high)&(free_high+1)` — изолирует степень двойки, равную длине
хвостовой серии единиц). Это особенно окупается на функциях с большим
числом "мусорных" переменных высокого индекса (частый случай в реальных
бенчмарках).

Разобрал битовые трюки построчно вручную (перечисление подмасок,
изоляция trailing-ones run, инвариант "свободные позиции `high_vals`/
`sub_high` всегда 0") — противоречий не нашёл. Заявленное автором ускорение
(сравнение с итерацией 1 на синтетическом бенчмарке):

| Метрика | Было (итерация 1) | Стало (итерация 2) |
|---|---|---|
| Total Time | 7.247 мс | 3.889 мс (≈2×) |
| Median (p50) | 2.496 мкс | 0.615 мкс (≈4×) |
| Average | 37.547 мкс | 20.152 мкс |
| 95th % (p95) | 194.210 мкс | 118.615 мкс |

**Итерация 3 (`1ed0b29`, "bdd to tt optimization")** — дальнейшая
доработка того же подхода (65 добавлений / 85 удалений в том же файле),
детально не разбиралась построчно отдельно от итерации 2, т.к. верификация
(см. ниже) проводилась именно на этом, финальном на момент ревью, коммите.

### 3.3. Верификация (не только чтение кода)

`bdd_to_tt` — единственная функция в этой папке, для которой ревью не
ограничилось чтением исходников:

- Собрано и прогнано в реальном девконтейнере
  (`genetica-boolean-lib:latest`, тот же образ, что в `.devcontainer/`) —
  `test_bdd` целиком, 5 прогонов с разными `--rng-seed`: везде `PASS`.
- Написан отдельный fuzz-тест (не входит в проект), специально нацеленный
  на новую блочную оптимизацию: границы `n=1..9`, "паритет" по всем
  переменным на `n` вплоть до 24 (худший случай для блочного слияния —
  свободных бит нет вообще), и — что важнее всего — функции, реально
  зависящие лишь от 1-6 переменных, разбросанных среди `n=8..24` "пустых"
  переменных (ровно тот сценарий, который максимально нагружает блочную
  склейку). **Все случаи для n=1..23 прошли побитово точно.**
  Единственный сбой — на `n=24` с ПОЛНОСТЬЮ случайной (несжимаемой)
  таблицей истинности, но проблема оказалась в `tt_to_bdd` (не в
  проверяемом `bdd_to_tt`) при построении BDD максимального размера из
  белого шума под маленькими лимитами Sylvan/Lace в тестовом скрипте —
  ресурсный артефакт харнесса, не дефект кода (такой вход и не встречается
  ни в реальном наборе тестов проекта, `kMaxGroundTruthVars=16`, ни в
  реалистичном использовании).

**Вывод по функции**: на ветке `egor` `bdd_to_tt` доведена от "прямо
противоречит собственной спецификации" до корректной, проверенной сборкой
и fuzz-тестом реализации с верной асимптотикой. **Рекомендация**: смержить
в `master` (аналогично тому, как это сформулировано для `thr_to_anf` в
[`thr/README.md`](../thr/README.md#63-отклонённый-эксперимент--дерево-слияния-вместо-линейного-fold)
— «оба исправления строго лучше, регрессий не найдено»). Единственная
мелкая находка, оставшаяся не устранённой ни в одной из трёх итераций:
пропал `catch(std::bad_alloc&)` — теперь `OutOfMemory` попадает в общий
`catch(std::exception&)` и репортируется как `InvalidArgument`. Не
нарушение контракта (`bdd_to_tt` не входит в обязательный список
`CONVENTIONS.md` п.2а — размер результата заранее ограничен
`kMaxTruthTableVars`), но регрессия точности кода ошибки относительно
самой первой версии на `master`.

## 4. `bdd_to_anf` — стаб на `master`; на `egor` гибридная реализация с
   найденным (и исправленным) багом в fallback-пути

### 4.1. На `master`

`bdd_to_anf.cpp` — 4 строки, `return fail<Anf>(ErrorCode::NotImplemented,
...)`. Корректно оформлено (именно `NotImplemented`, не выдуманное
исключение — тестраннер полагается на это для `SKIP`, `CONVENTIONS.md`
п.8).

### 4.2. На ветке `egor` — архитектура

[bdd_to_anf.hpp](bdd_to_anf.hpp) требует гибридную стратегию: основной
bottom-up обход по BDD с положительным разложением Дэвио (`ANF(node) =
ANF(low) ⊕ x·(ANF(low) ⊕ ANF(high))`), с fallback на TruthTable + Fast
Möbius Transform, если промежуточные полиномы разрастаются сверх порога.

Реализация на `egor` (`bdd_to_anf.cpp`, 392 строки) в целом соответствует
этому плану:
- `DenseBuilder` — плотный снапшот BDD через regular/complement, той же
  pre/post-order схемой, что и `bdd_to_aig`. Проверено: дети всегда
  получают id раньше родителя (гарантируется порядком обхода), поэтому
  `assert(node.lo < i)` в дальнейшем коде — действительно инвариант, не
  случайность.
- `AnfEngine::compute` — bottom-up Дэвио на `BoolePolynomial`, с порогом
  `threshold=50000` на длину монома на каждом узле.
- `FallbackEngine::compute` — при превышении порога: `bdd.to_tt()` (полная
  материализация через `Bdd::to_tt()` из `core/common.hpp`, не через
  standalone `bdd_to_tt`), затем Fast Möbius Transform in-place, затем
  сборка `BoolePolynomial`/`AnfFallback` из полученных коэффициентов.

### 4.3. Найденный и исправленный баг — направление XOR в fallback-трансформе

**Симптом** (реальный прогон `test_bdd` на девконтейнере, не синтетика):
`BMM_STATUS bdd_to_anf FAIL [ground_truth] 'or_all_n16' разошлась в точке
#0 (x=0000000000000000)`.

**Причина**: в `FallbackEngine::compute` цикл быстрого преобразования
Мёбиуса XOR-ил в неправильную сторону:

```cpp
// было (баг):
if (kitty::get_bit(raw_tt, j + k + bit)) {
    kitty::flip_bit(raw_tt, j + k);        // XOR верхней половины В нижнюю
}
```

Правильное преобразование (truth table → коэффициенты Жегалкина) должно
XOR-ить нижнюю половину в верхнюю. Проверено вручную для n=1: код давал
`[f(0)^f(1), f(1)]`, должно быть `[f(0), f(0)^f(1)]` — переставлены
местами.

**Почему баг не всплывал систематически**: этот путь — не основной, он
срабатывает только когда bottom-up ANF по BDD превышает порог 50000
монотомов. Для симметричных плотных функций (OR/AND/паритет от n
переменных) итоговый ANF имеет `2^n-1` термов — порог впервые превышается
примерно на n=16 (`2^16-1=65535 > 50000`, а `2^15-1=32767` — ещё нет), то
есть это первый тестовый случай в растущем наборе, где вообще заходит в
этот код. Прогон полного `test_bdd` до фикса проходил `PASS` 3 раза подряд
(в другом окружении/раскладе) и лишь один раз показал `FAIL` — вероятно,
`BoolePolyRing`/BRiAl держит какое-то состояние колец между тест-кейсами,
влияющее на то, попадает ли расчёт точно в порог; для сути бага это не
важно, баг детерминированно воспроизводится в изоляции (см. ниже) при любом
входе, реально проходящем через этот путь.

**Верификация фикса**:
```cpp
// стало:
if (kitty::get_bit(raw_tt, j + k)) {
    kitty::flip_bit(raw_tt, j + k + bit);
}
```
- Изолированное репро (OR от 16 переменных, построено напрямую через
  Sylvan, минуя тестовую инфраструктуру) — до фикса: `poly.length()=2,
  hasConstantPart=1`, `OR_16(all-zero) = 1` (неверно). После фикса:
  `poly.length()=65535` (= `2^16-1`, ровно ожидаемый размер ANF от OR-16),
  `hasConstantPart=0` (верно — у OR нет константного члена), `OR_16
  (all-zero) = 0`, 0 расхождений на 200 случайных точках.
- Полный `test_bdd`, 5 прогонов с разными `--rng-seed` после фикса: везде
  `PASS` по всем секциям.

**Статус фикса**: закоммичен в локальной ветке `egor-fix-anf-mobius`
(коммит `5f879b7`, поверх `egor`@`1ed0b29`), **не запушен** — ждёт решения,
как передать (Егору напрямую как патч, или запушить в `egor`/влить в
`master`).

### 4.4. Известный, не исправленный пробел — non-BRiAl fallback никогда
   не считает

`AnfEngine::compute`:
```cpp
static ComputeResult compute(const DenseBDD& dense, uint32_t n_vars, size_t threshold) {
#if BMM_HAVE_BRIAL
    ... реальный bottom-up Дэвио на BoolePolynomial ...
    return {ComputeStatus::SUCCESS, std::move(result)};
#else
    return {ComputeStatus::FALLBACK_NEEDED, AnfFallback()};
#endif
}
```
В сборке без BRiAl (`AnfFallback`/`std::set<Monomial>`) эта функция
**всегда** возвращает `FALLBACK_NEEDED`, ничего не считая — bottom-up-путь
по BDD в этой конфигурации не выполняется ни разу, он написан только под
BRiAl. А `FallbackEngine::compute` вызывает `bdd.to_tt()`, который
отваливается `ErrorCode::TooManyVariables` при `n_vars() > 24`
(`kMaxTruthTableVars`). Итог: **в non-BRiAl сборке `bdd_to_anf` физически
не может обработать BDD больше 24 переменных** — теряет ровно то
преимущество перед полной материализацией TT, ради которого BDD-путь
вообще нужен. `CONVENTIONS.md` п.5 обещает "обе ветки полностью рабочие,
просто с разной асимптотикой" — здесь это не так.

В штатном докер-образе проекта `BMM_HAVE_BRIAL=1` "из коробки"
(`CONVENTIONS.md` п.5), так что в текущем CI/девконтейнере эта дыра не
проявляется — но код содержит нерабочую ветку, которая должна быть рабочей
по контракту. Не устранено (отдельная, не пересекающаяся с §4.3 задача).

## 5. `bdd_to_thr` — стаб на `master`; на `egor` частичная реализация
   (K≤6) с образцовой верификацией результата

### 5.1. На `master`

`bdd_to_thr.cpp` — 4 строки, `NotImplemented`, честно оформлено.

### 5.2. На ветке `egor` — пайплайн

[bdd_to_thr.hpp](bdd_to_thr.hpp) описывает многоступенчатый фильтр:
унитонность по кофакторам → вектор Чоу через bottom-up обход BDD → таблица
"вектор Чоу → веса" для n≤8 → ILP на простых импликантах (OR-Tools) для
n>8.

Реализация (`bdd_to_thr.cpp`, 428 строк) сознательно упрощает шаги 1-2 для
малого K: вместо bottom-up обхода структуры BDD строит полную truth table
в лоб (`K≤6` ⇒ максимум 64 точки) и считает унитонность/вектор Чоу
напрямую по ней — с явным обоснованием в коде ("выигрыш от DP не имеет
практического значения, риск логических ошибок выше"). Разумный
инженерный компромисс для маленького K, не подмена алгоритма ради
видимости.

**Главная сильная сторона, подтверждённая и статически, и по факту** —
финальная верификация перед выдачей результата:
```cpp
if (!verify_threshold_from_tt(truth, final_weights, final_threshold, K)) {
    return fail<Thr>(ErrorCode::Unsupported, "... верификация не пройдена");
}
```
Если найденные веса не воспроизводят реальную таблицу истинности —
возвращается `Unsupported`, а не неверный ответ. Это полностью нейтрализует
любые ошибки в справочной таблице ниже (см. §5.3, п.2) — они не могут
привести к ложноположительному результату, только к ложноотрицательному
(функция, которая на самом деле пороговая, распознаётся как
"не пороговая"/"не реализовано"). Это ровно тот тип защиты, который стоит
перенимать в любом другом месте проекта, где эвристика/таблица
подстановки используется как быстрый путь к точному результату.

### 5.3. Находки

1. **Условие входа в быстрый путь смотрит на заявленную арность, а не на
   реальное число переменных.** `if (n_vars <= 6)` использует
   `bdd.n_vars()`, а не `K` (число переменных, реально встречающихся в
   BDD), которое считается только ВНУТРИ этой ветки. Функция с заявленной
   арностью 7+, но реально зависящая всего от 3 переменных, никогда не
   попадёт в рабочий путь — упадёт в `NotImplemented`, хотя K=3 уже
   полностью поддерживается. Fail-safe (не даёт неверный ответ), но
   занижает реальное покрытие. Обход, считающий `unique_vars`, сам по себе
   дешёвый (O(|BDD|), не O(2^n_vars)) — можно и нужно делать его всегда,
   независимо от `n_vars`, и проверять `K<=6` уже после.
2. **`CHOW_DATABASE` содержит минимум 2 записи с математически неверным
   вектором Чоу.** Пересчитано вручную:
   - "Проекция" (K=2, `f(x0,x1)=x0`): реальный вектор Чоу — `(2,1)`, не
     `(2,0)`, как в записи (доля выполняющих присвоений с фиктивной
     переменной `x1=1` — ровно половина от sat, не ноль).
   - "AND2 с фиктивной переменной" (K=3): реальный вектор — `(2,2,1)`, не
     `(2,2,0)`.
   Хуже того: сам сценарий "переменная не влияет на функцию, но входит в
   K" структурно невозможен — фиктивная переменная физически не будет
   узлом в приведённом (reduced) BDD, значит никогда не попадёт в
   `unique_vars`. Эти две записи — мёртвый код, не совпадут ни с одним
   реальным запросом. **Не опасно** благодаря верификации из §5.2, но
   таблицу явно печатали руками, а не генерировали программно.
3. Раз в 8 ручных записях уже 2 математически неверные — таблица на 2730
   записей (упомянутый в переписке ориентир на "полную базу") требует
   генератора с автоматической верификацией каждой записи перед вставкой,
   а не переписывания вручную.
4. `.hpp` обещает справочную таблицу для `n<=8` (шаг 3 пайплайна), код
   останавливается на `n<=6` — не критично само по себе, но стоит явно
   пометить в коде, что 7 и 8 пока не покрыты.

**Эмпирическое подтверждение** (реальный прогон `test_bdd`): секция
`bdd_to_thr` стабильно даёт `SKIP` ("функция ещё не реализована"). Причина
— механика `verify/test_runner.hpp`: `SKIP` выставляется, если
`ErrorCode::NotImplemented` встретился НА ЛЮБОМ тестовом случае из растущего
набора (не только на первом пробном) — весь прогон досрочно
останавливается. Учитывая, что рабочий путь покрывает только K≤6 и всего 8
(частично неверных) записей в таблице, это происходит рано — находки 1 и 2
выше не теоретические придирки, а прямая причина того, что секция целиком
не набирает `PASS` уже сейчас.

### 5.4. Roadmap (по приоритету, что реально даёт покрытие за минимум
   усилий)

1. **Гейтинг по K, не по `n_vars`** (минуты) — см. находку 1. **Сделано**
   (было уже на момент начала работы над §5.4а — проверено чтением кода).
2. **Пересобрать `CHOW_DATABASE` программно** — **сделано** в отдельной
   сессии, см. §5.4а ниже за полной методологией и результатом.
3. **Поднять лимит до n≤8** — НЕ сделано в этой сессии (см. §5.4а,
   "Что осталось за рамками" — из объёма этой конкретной задачи, отдельная
   работа: расширение `FastThresholdResult<K>`/диспетчера на K=7,8 плюс
   генерация записей для них тем же генератором).
4. **ILP-путь для n>8** (отдельная, большая задача) — извлечение простых
   импликант из структуры BDD (Minato 1993, ZDD-based) + модель OR-Tools
   (ограничения по простым импликантам ON/OFF-набора + симметрия/порядок
   из вектора Чоу), `INFEASIBLE → Unsupported`. Обязательно прогнать
   результат через тот же `verify_threshold_from_tt`. Можно честно
   оставить на следующую итерацию — `NotImplemented` для `K>8` пока
   корректно и ожидаемо.
5. **Тестовая инфраструктура** — **сделано**: `bdd/test_bdd.cpp`
   (`TEST_CASE("bdd_to_thr")`) теперь строит явный набор через
   `growing_threshold_test_functions(6)` + `ground_truth_from_thr` вместо
   дефолтного `growing_test_functions()` (см. §5.4а). Ограничено n≤6, а не
   дефолтным `kMaxGroundTruthVars=12` — иначе первый же случай с n=7 упёрся
   бы в пока не реализованный K>8-путь (см. п.3/4 выше) и замаскировал бы
   результат работы над самой базой посторонним, уже известным
   ограничением.
6. **Обход лимита K≤6 через другое представление — проверено эмпирически,
   работает.** По запросу пользователя ("можно обойти лимит другим путём?")
   добавлены кейсы в `verify/chain_tests.cpp` (§3 "прямой путь vs обход"):
   `Bdd -> Tt -> Thr` (через уже существующие `bdd_to_tt` + `tt_to_thr`,
   общий ILP-путь Muroga, НЕ завязанный на `CHOW_DATABASE`) при n=7/9/11/13
   — все строго больше жёсткой границы K≤6. Результат (реальный прогон,
   не оценка): прямой `bdd_to_thr` во всех четырёх случаях недоступен
   (`NotImplemented`, как и ожидалось), обходной путь через `Tt` — **и
   единственный, и рабочий**: n=7 — 8.9мс, n=9 — 37.7мс, n=11 — 151.6мс,
   n=13 — 417мс (растёт с n, ожидаемо для ILP-подхода, но не взрывообразно
   на этой конкретной, симметричной функции — "maj"). Это НЕ отменяет
   ценность пункта 3 (прямой путь с готовой таблицей всё равно на порядки

   рабочий обходной путь для n=7..24 уже сейчас, без ожидания расширения

   §3 за полной таблицей и `thr/README.md §3` (`tt_to_thr`) за честными
   границами применимости самого ILP-пути на РЕАЛЬНЫХ (не симметричных)
   весах — там практический потолок ниже (~n=16 на состязательных весах,
   не на "maj").

### 5.4а. `CHOW_DATABASE` пересобрана программно — методология и результат

По запросу пользователя ("найди возможность улучшить пункт: CHOW_DATABASE
по-прежнему 8 ручных записей, не 2730 (полная база)") написан офлайн-генератор
[`verify/diag_chow_database_gen.cpp`](../verify/diag_chow_database_gen.cpp),
использующий ОБЩУЮ с runtime-кодом реализацию Chow-логики
([`bdd/chow_detail.hpp`](chow_detail.hpp) — `pack_key`/`compute_chow_from_tt`/
`is_unate_from_tt`/`verify_threshold_from_tt` вынесены туда из
`bdd_to_thr.cpp`, чтобы генератор и runtime не могли разойтись в деталях
канонизации незаметно).

**Метод, в два независимых шага:**

1. **Прямая конструкция**: перебор целочисленных весов
   `w_1>=w_2>=...>=w_K>=0` в ограниченном диапазоне суммы + порог `T`,
   построение результирующей функции напрямую (`sum(w_i·x_i) >= T`) — она
   ГАРАНТИРОВАННО пороговая по построению, распознавать ничего не нужно.
   Дёшево, находит подавляющее большинство канонических записей для K≤6
   быстро (для K≤4 — доли секунды; K=5 — ~13 минут; K=6 — ~20 минут при
   уменьшенной границе весов, см. ниже).
2. **Монотонный DFS + независимый оракул**: для каждого K отдельно —
   исчерпывающий перебор ВСЕХ Dedekind(K) монотонных функций K переменных
   (стандартный DFS по решётке подмножеств: если `f(y)=1` для какого-то
   `y ⊊ x`, то `f(x)=1` обязано быть тоже, иначе можно выбирать 0/1
   свободно). Для каждой найденной монотонной функции, если её канонический
   ключ уже есть в базе после шага 1 — всё сходится; если нет — вызывается
   НЕЗАВИСИМЫЙ ILP-оракул `thr::tt_to_thr` (Muroga, уже покрыт
   ground_truth/metamorphic тестами проекта) как истина в последней
   инстанции: либо он находит веса (шаг 1 пропустил редкий случай — запись
   добавляется), либо подтверждает, что функция ДЕЙСТВИТЕЛЬНО не пороговая
   (шаг 1 верно её не нашёл).

Раз шаг 2 перебирает ВСЕ Dedekind(K) функций без пропусков и разрешает
КАЖДОЕ расхождение через оракул — для того K, где это отработало полностью,
**полнота базы доказана**, не является эвристикой или предположением.

**Результат по K, реально измеренный (не оценка):**

| K | Dedekind(K) (монотонных функций) | Канонических записей в базе | Полнота | Время генерации |
|---|---|---|---|---|
| 1 | 3 | 1 | доказана (оракул: 0 расхождений) | <1мс |
| 2 | 6 | 2 | доказана (оракул: 0 расхождений) | <1мс |
| 3 | 20 | 5 | доказана (оракул: 0 расхождений) | <1мс |
| 4 | 168 | 17 | доказана (18 расхождений, оракул подтвердил — все НЕ пороговые) | ~1с |
| 5 | 7581 | 92 | доказана (4204 расхождения, оракул подтвердил — все НЕ пороговые) | ~13 мин |
| 6 | 7 828 354 | 994 | **best-effort, НЕ доказана** — см. ниже | ~20 мин |

**Почему K=6 — best-effort, а не доказанная полнота.** Dedekind(6)=7 828 354
— каждое расхождение шага 1 требует отдельного вызова ILP-оракула; по опыту
K=5 (где 4204 расхождения заняли ~67с, т.е. ~16мс/вызов — почти всё время
шага 2 ушло именно на них) экстраполяция на K=6-масштаб расхождений (в
пробном прогоне — 7 558 702 нерасхожденных случая) заняла бы часы, что
непрактично для одной сессии. Решение: для K=6 шаг 2 запускается БЕЗ
вызова оракула — только считает количество расхождений (информационно), не
разрешая их. `IS_FULL_DATABASE_FOR_K[6] = false` — runtime-код при
K=6-запросе, не найденном в базе, честно возвращает `NeedFallback`
(→`NotImplemented`), а НЕ ложное "точно не пороговая" (`NotThreshold`).
Обе цифры (994 найденных, 7 558 702 непроверенных) — реальный вывод
генератора, не оценка.

**Итоговый размер базы — 1111 записей**, не 2730 (число, упомянутое в
переписке как ориентир на "полную базу" — источник этой оценки в проекте не
задокументирован, и она не подтверждается независимым перебором: подсчёт
для K≤5 доказан точно через Dedekind-числа и не оставляет места для другой
цифры на этих K; для K=6 база — сознательно неполная best-effort выборка,
а не окончательная величина, так что 1111 — это "по крайней мере 1111",
не "ровно 2730").

**Верификация результата**: `bdd/test_bdd.cpp` (`bdd_to_thr`) теперь
использует `growing_threshold_test_functions(6)` (см. п.5 roadmap выше) —
секция впервые даёт **PASS**, а не `SKIP`. Полный `ctest` пройден без
регрессий.

**Что осталось за рамками этой задачи**: n≤8 (шаг 3 roadmap выше) и
ILP-путь для n>8 (шаг 4) — не тронуты, честно оставлены как открытые
пункты, а не выданы за решённые.

## 7. Сквозные лучшие практики модуля

Наблюдения, повторяющиеся в большинстве функций этой папки — часть из них
уже стиль, часть — то, что стоит взять за образец в других модулях:

- **Приоритетное правило параллелизма (`CONVENTIONS.md` §6, правило 1)
  соблюдено без исключений**: ни одна из 5 функций не заводит собственный
  TBB/OpenMP вокруг вызовов Sylvan — обход BDD-узлов всегда идёт только
  через `sylvan::Bdd`/`sylvan_obj.hpp`.
- **Мемоизация по каноническому (без complement-бита) узлу** —
  последовательно применяется и в `bdd_to_aig`, и в `DenseBuilder`
  `bdd_to_anf` (ветка `egor`) — оба места отдельно и явно проверены на
  корректность обработки complement-рёбер (инверсия применяется к
  результату уже ПОСЛЕ выборки из кэша, не размножает кэш на полярность).
- **`try/catch(std::bad_alloc&)` на границе функции** — там, где это
  реально требуется (`bdd_to_aig`, единственная функция с неограниченным
  потенциальным ростом результата) сделано корректно; в остальных функциях
  либо не требуется по контракту (`CONVENTIONS.md` п.2а), либо (как
  выяснилось для `bdd_to_tt` на `egor`, §3.3) может случайно потеряться при
  рефакторинге — стоит проверять при каждом ревью, а не полагаться, что раз
  сделано один раз, останется навсегда.
- **Top-down/диапазонное разбиение вместо точка-за-точкой для полной
  материализации TT** (`bdd_to_tt`, §3) — самый концентрированный урок
  этой папки: асимптотика Θ(2^n) для материализации `2^n`-битной таблицы
  не бесплатна автоматически, наивный код легко получает лишний множитель
  n, даже если каждый отдельный шаг выглядит невинно.
- **Верификация результата перед выдачей, когда путь построения — эвристика
  или таблица подстановки** (`bdd_to_thr`, §5.2) — единственное в этой
  папке место, которое полностью защищено от ложноположительных ответов
  независимо от качества самой эвристики/таблицы. Стоит рассмотреть как
  общий паттерн: там, где результат можно дёшево проверить (сравнить с
  исходным представлением на достаточном числе точек или структурно), а
  сам путь построения эвристический/приближённый/неполный — эта проверка
  окупается на порядок больше, чем попытка сделать саму эвристику идеальной.
- **`NotImplemented`, не выдуманные исключения, для незаконченных функций**
  (`bdd_to_anf`/`bdd_to_thr` на `master`) — соблюдено ровно так, как того
  требует `CONVENTIONS.md` п.8: тестраннер отличает `SKIP` от `FAIL`
  именно по этому коду ошибки.

## 9. Как повторить

Полная сборка и прогон (в девконтейнере, тот же образ, что описан в
`.devcontainer/`):
```
MSYS_NO_PATHCONV=1 docker run --rm -v "D:/Proga/web/bmm-translib:/workspace" -w /workspace genetica-boolean-lib:latest bash -c "cmake --build build --target test_bdd -j\$(nproc) && ./build/test_bdd"
```

Всё, что относится к ветке `egor` в этом файле (§3-§5), получено на
отдельном `git worktree` (`git worktree add <path> origin/egor --detach`),
а не прямо в рабочей копии `master` — сама ветка `egor` не смержена и не
собирается как единое целое с текущим `master`.
