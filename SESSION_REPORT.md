# Отчёт по сессии: ревью, перенос кода, исправления, тестовая инфраструктура

Сводная документация одной длинной рабочей сессии над `bmm-translib` —
библиотекой трансляции между пятью представлениями булевых функций (AIG,
BDD, ANF/полином Жегалкина, Thr/пороговые функции, TruthTable). Сессия шла
от «прочитать и оценить код» до «внести и проверить правки, добавить новый
класс тестов, подключить реальные данные». Документ — не журнал коммитов
(см. `git log`) и не описание API (см. `core/CONVENTIONS.md`, doc-комментарии
в `.hpp`), а рассказ о том, что делалось и почему, в порядке, в котором это
происходило.

## 0. Отправная точка

20 функций трансляции (по 4 на каждое из 5 представлений — `X_to_Y` для
каждой упорядоченной пары), разбросанные по веткам `master`/`egor`/`Thr`/
`aig`/`anf` учебного монорепозитория (несколько студентов, разные ветки,
разная степень готовности). До этой сессии в `master` часть функций были
честными заглушками (`bdd_to_anf`, `bdd_to_thr`), часть — рабочими, но с
известными, задокументированными в `*/README.md` находками, ещё не
исправленными в самом коде.

## 1. Жёсткое ревью существующего кода (без изменений)

Прочитаны все 4 `README.md` (`aig/`, `anf/`, `bdd/`, `thr/`) и
`core/CONVENTIONS.md`, затем построчно — текущий код всех 20 функций
трансляции (не пересказ READMEs, а самостоятельная проверка). Собран список
находок, не отражённых (или отражённых не полностью) ни в одном README:

- **`thr/thr_to_aig.cpp`** — мемоизация `full_adder` теряла полярность
  сигнала (ключ строился из `node_to_index(get_node(s))`, что не различает
  `s` и `!s`) — потенциальный тихий возврат неверной суммы/переноса при
  совпадении узлов разной полярности.
- **`anf/anf_to_thr.cpp`** — прямое использование `std::holds_alternative`/
  `std::get` вместо `is_ok`/`value`/`error` — работало только потому, что
  сборка использует `std::variant`-fallback для `Result<T>`; при переходе на
  `std::expected` файл перестал бы компилироваться без предупреждения.
- **Бессистемная защита от `OutOfMemory`**: `catch(bad_alloc)` присутствовал
  в одних функциях и отсутствовал в других без видимой логики (например,
  отсутствовал в `aig_to_thr`/`tt_to_aig`/`aig_to_anf`/`aig_to_bdd`, хотя
  именно `aig_to_thr` строит ILP-модель, которая теоретически может
  исчерпать память).
- **`thr/tt_to_thr.cpp`** — единственная из трёх ILP-based `*_to_thr`
  функций без фолбэк-цепочки солверов (только `"SCIP"`, тогда как
  `aig_to_thr.cpp`/`anf_to_thr.cpp` уже пробуют `SCIP→SAT→CBC`); плюс
  `static_cast<int64_t>` от `double` без округления (усечение к нулю вместо
  `std::round`/`llround`, использованных в соседних функциях).
- **Ни одна из `tt_to_thr`/`aig_to_thr`/`anf_to_thr`** не проверяла решение
  солвера перед выдачей — единственное место в проекте с такой защитой было
  `bdd_to_thr.cpp` (`verify_threshold_from_tt`).
- **`anf/tt_to_anf.cpp`** — конструирование монома через n последовательных
  умножений (`mono *= ring.variable(var)`), тот же антипаттерн, который уже
  был найден и исправлен для `thr_to_anf.cpp` на неслитой ветке `origin/Thr`
  (`BooleExponent`-конструирование за один вызов).
- **`thr/thr_to_tt.cpp`** — `#pragma omp parallel for` без порога по
  размеру (по замерам — до 754 раз медленнее последовательной версии на
  малых n), при том что рядом (`anf/anf_to_tt.cpp`) уже есть проверенный
  паттерн порога.
- **`bdd/bdd_to_thr.cpp`** (на ветке `egor`) — гейтинг по `n_vars`, а не по
  реальному числу уникальных переменных K; плюс две математически неверные
  записи в ручной таблице `CHOW_DATABASE` (пересчитаны вручную и
  подтверждены найденной ошибкой — «Проекция» и «AND2 с фиктивной
  переменной»).
- **`anf/anf_to_bdd.cpp`** без BRiAl — полностью недоступна (`NotImplemented`
  безусловно), второе (после уже задокументированного `bdd_to_anf`)
  нарушение заявления `CONVENTIONS.md` п.5 «обе ветки полностью рабочие».
- **Структурный вывод**: лучший, уже написанный и проверенный код
  (`bdd_to_tt`/`bdd_to_anf`/`bdd_to_thr` на `egor`, фикс `thr_to_anf` на
  `origin/Thr`) не был в `master` — колонка «кто отвечает» в корневом
  `README.md` пустая, никто не сливал ветки.

Полный текст ревью — в истории диалога этой сессии; здесь — выжимка находок,
которые затем были исправлены (см. §2 ниже).

## 2. Перенос лучшего кода в master + точечные фиксы (пошагово, с проверкой на каждом шаге)

По прямому запросу пользователя («перенеси весь лучший код в master»,
затем — «исправь все найденные ошибки, пошагово: сначала меняешь, потом
точечно проверяешь») каждое изменение из списка §1 внесено и проверено
ОТДЕЛЬНО, прежде чем переходить к следующему.

### 2.1. Порт из неслитых веток

| Файл | Источник | Что перенесено |
|---|---|---|
| `bdd/bdd_to_tt.cpp` | `origin/egor` | Θ(2^n) вместо Θ(n·2^n) — DFS по BDD с батч-заполнением куба вместо перебора точка-за-точкой |
| `bdd/bdd_to_anf.cpp` | `origin/egor` + `egor-fix-anf-mobius` | гибридная реализация (Давио bottom-up + TT/Мёбиус fallback) с уже исправленным багом направления XOR в Мёбиус-трансформе |
| `bdd/bdd_to_thr.cpp` | `origin/egor` | K≤6 через справочную таблицу векторов Чоу + `verify_threshold_from_tt` |
| `thr/thr_to_anf.cpp` | `origin/Thr` | `BooleExponent`-конструирование монома + OMP-порог + `collapse(2)` |

**Важная поправка при переносе, не отражённая в исходных ветках**: `egor` и
`egor-fix-anf-mobius` были сделаны ДО того, как в `core/common.hpp`
появился механизм `var_to_level`/`var_at_level()` (нужен только для
`anf_to_bdd`, единственного производителя `Bdd` с физически переставленным
порядком уровней, см. `anf/README.md` «Часть 2»). Прямой перенос кода
egor «как есть» тихо ломался бы на любом `Bdd`, пришедшем от `anf_to_bdd()`
(что сейчас — умолчание, эвристика `Force`). В `bdd_to_tt.cpp` и
`bdd_to_anf.cpp` (`DenseBuilder`/`AnfEngine`) добавлены вызовы
`var_at_level()` там, где раньше стоял голый `TopVar()`.

**Проверено отдельной диагностикой** (временный `anf/diag_var_to_level.cpp`,
удалён после проверки): построена ANF с перемешанными индексами клик
(натуральный порядок заведомо не оптимален), пропущена через `anf_to_bdd`
(подтверждён нетривиальный `var_to_level`, не identity), `bdd_to_tt`/
`bdd_to_anf` на этом BDD сверены с независимым эталоном (`anf_to_tt`) — 0
расхождений на всех 4096 точках. Без фикса это разошлось бы.

### 2.2. Точечные фиксы (каждый — правка → узкая проверка → следующий)

- **`thr/thr_to_aig.cpp`** — ключ мемоизации `full_adder` кодирует
  индекс+полярность (`(idx<<1)|complement`) до сортировки, вместо голого
  индекса узла.
- **`anf/anf_to_thr.cpp`** — `std::holds_alternative`/`std::get` заменены на
  `is_ok`/`value`/`error`; добавлены `catch(bad_alloc)` и верификация
  решения перед выдачей.
- **`aig/aig_to_thr.cpp`, `aig/tt_to_aig.cpp`, `aig/aig_to_anf.cpp`,
  `aig/aig_to_bdd.cpp`** — добавлены отсутствовавшие `catch(bad_alloc)`.
- **`thr/tt_to_thr.cpp`** — добавлена цепочка солверов `SCIP→SAT→CBC`,
  `static_cast` заменён на `std::llround`, добавлена верификация решения.
- **`anf/tt_to_anf.cpp`** — монoм строится через `polybori::BooleExponent`
  вместо n последовательных умножений.
- **`thr/thr_to_tt.cpp`** — добавлен порог по размеру для OpenMP
  (`kParallelThresholdBlocks`, согласован с порогом `anf_to_tt.cpp`).
- **`bdd/bdd_to_thr.cpp`** — гейтинг переписан на K (обход дерева BDD для
  подсчёта уникальных переменных выполняется всегда, не только при
  `n_vars<=6`); исправлены две записи `CHOW_DATABASE` (пересчитаны вручную
  дважды, найдена и исправлена **дополнительная** ошибка, которую не
  called out в исходном ревью — неверный `threshold` в записи «Проекция»,
  не только неверный ключ).

**Проверено отдельной диагностикой** (`bdd/diag_bdd_to_thr.cpp`, удалён
после проверки): 8 целевых случаев — обе исправленные записи, K-гейтинг
(MAJ3 с 5 фиктивными переменными при `n_vars=8`, до фикса гарантированно
уходил в `NotImplemented`), уже верные записи (регрессии нет), непороговая
функция (по-прежнему корректно отклоняется). Все 8 — PASS.

Полный `ctest` после всех правок §2.1-2.2 — **38/38 тестов, 0 провалов**

## 3. Найдена и исправлена системная проблема тестовой инфраструктуры

При попытке пошагово проверить `tt_to_anf`/`anf_to_thr` изолированно секция
зависала (SIGTERM по внутреннему таймауту 120с) — и это оказалось не
регрессией от правок §2, а **предсуществующим багом**, подтверждённым прямым
сравнением со старой (неизменённой) версией `tt_to_anf.cpp`: идентичное
время на `or_all_n11` (7606мс старая версия против 7231мс новая — в пределах
шума).

**Корень**: `Anf::to_tt()` (`core/anf_repr.hpp`) был наивным циклом «вызвать
`evaluate()` на каждой из 2^n точек», а `Anf::evaluate()` сам по себе — это
O(размер полинома) за вызов (в отличие от, например, `TruthTable::evaluate()`,
O(1)). `verify/test_runner.hpp` вызывал эквивалент этого до 4 раз на каждую
тестовую функцию (`find_mismatch` + `compute_chow_parameters` дважды +
`compare_anf_degree`/`check_cofactor_commutativity`) — итого
O(2^n × размер × 4) ≈ O(4^n)-подобный рост на ПЛОТНЫХ функциях (`or_all`,
`random`) уже на штатном `kMaxGroundTruthVars=12` (не гипотетическом
большем n — граница, использованная во всех предыдущих сессиях проекта).

**Почему это не нарушение принципа «verify/ не полагается на код студентов»**
(`CONVENTIONS.md` п.7): `Anf::to_tt()`/`Anf::evaluate()` — методы САМОГО ТИПА
`Anf` в `core/` (инфраструктура, которую студенты не пишут), не один из 20
тестируемых переводов — ровно тот же уровень доверия, что уже оказан
`TruthTable::evaluate()`/`Bdd::evaluate()`.

**Фикс**:
1. `core/anf_repr.hpp::Anf::to_tt()` (обе ветки, BRiAl и `AnfFallback`) —
   заменён наивный цикл на настоящее преобразование Мёбиуса/Жегалкина,
   O(n·2^n) вместо O(2^n·размер).
2. `verify/test_runner.hpp` — добавлен `detail::make_fast_evaluator()`:
   материализует `to_tt()` ОДИН раз на представление, дальше все 4 проверки
   переиспользуют один и тот же O(1)-поиск по битовой маске вместо
   повторного вызова `evaluate()`.

**Проверено** отдельной диагностикой (`anf/diag_verify_anf_to_tt.cpp`,
удалена после проверки): новый `to_tt()` сверен с независимым эталоном
(прямой перебор через `evaluate()`, не через `to_tt()`) на 154 функциях
(n=6..10, включая плотные `or_all`/`random`) — 0 расхождений.

**Эффект**: полный `ctest` — **38/38 (позже 39/39, см. §4), 0 провалов**;
секции, ранее не укладывавшиеся в 120с (`tt_to_anf`, `anf_to_thr` и другие
секции, где `Anf` участвует хоть с одной стороны), теперь занимают доли
секунды. Итоговое общее время прогона (включая штатные `[benchmark]`-секции
TBB/OpenMP scaling, которые не про корректность и не про эту находку) —
697.56с; без них — 26 секций отдельных функций + 4 инфраструктурные
`*::to_tt` проверки + 2 SAT-проверки уложились в ~35с суммарно.

## 4. Тесты композиции: round-trip, длинные циклы, прямой путь против обходного

Новый постоянный файл `verify/chain_tests.cpp` (+ разделяемая инфраструктура
`verify/chain_utils.hpp`), зарегистрирован как `test_chains` в `ctest`.
Обобщённое представление «любое из пяти» (`std::variant<Aig, Bdd, Anf, Thr,
TruthTable>`) + диспетчер по всем 20 функциям перевода — один и тот же код
для любой цепочки шагов, без ручного дублирования.

- **Round-trip (X→Y→X)** по всем 20 упорядоченным парам представлений (12
  общих + отдельно все с участием `Thr`, с учётом, что не любая функция
  пороговая) — **0 расхождений** на растущем наборе тестовых функций.
- **Более длинные циклы** (3-5 шагов): полный `Aig→Bdd→Anf→Thr→Tt→Aig` на
  заведомо пороговых функциях, `Anf→Bdd→Aig→Tt→Anf` (4 шага) и
  `Bdd→Tt→Anf→Bdd` (3 шага) на произвольных функциях — **0 расхождений**.
- **Прямой перевод против обхода через третье представление** — на вопрос
  пользователя «возможно ли, что обходной путь быстрее прямого» получен
  однозначный эмпирический ответ: **да**, подтверждено на реальных
  измерениях, и разрыв растёт с n:

  | X→Y | n | прямой, мс | обходной (Z), мс | во сколько раз обходной быстрее |
  |---|---|---|---|---|
  | Anf→Bdd | 11 | 9.8-13.5 | 0.9-1.1 (via Tt) | ~10-13x |
  | Bdd→Anf | 11 | 0.8-1.4 | 0.13-0.2 (via Aig) | ~5-7x |
  | Anf→Bdd | 17 | 1525-1907 | 127-150 (via Tt) | ~12-13x |
  | Bdd→Anf | 17 | 705-760 | 107-165 (via Tt) | ~5-7x |
  | Anf→Aig | 13 | 4.4-5.7 | 3.7-4.6 (via Tt) | ~1.2-1.7x |

  Обратный эффект (прямой путь быстрее) — там, где прямая функция уже
  тривиальна (`Aig↔Bdd`, доли миллисекунды: заводить дополнительный шаг
  перевода дороже самой полезной работы).

Полная разбивка и все числа — `verify/CHAIN_TESTS_REPORT.md`
(перегенерируется при каждом запуске `test_chains`).

## 5. Реальные (не синтетические) наборы данных

По запросу «найди реальные датасеты, прогони их через обычные тесты и через
цепочки» — новый постоянный файл `verify/real_datasets_tests.cpp`
(таргет `test_real_datasets`), использующий ту же инфраструктуру
`chain_utils.hpp`.

Источники (по одному на представление, из которого его реалистичнее всего
строить напрямую):

- **Aig** — EPFL Combinational Benchmark Suite (`github.com/lsils/benchmarks`,
  тот же lsils, что и mockturtle; скачивается `benchmarks/scripts/
  download_epfl.sh`, уже был в проекте). Загрузка через `lorina`/
  `mockturtle::aiger_reader` (уже были доступны как зависимость mockturtle,
  раньше не использовались ни одним файлом проекта). Схемы EPFL обычно
  многовыходные — `bmm::Aig` требует ровно один PO (`CONVENTIONS.md` п.4),
  поэтому написана `extract_single_output()` — копирует всю логику схемы,
  оставляет один конкретный реальный выход.
- **Anf** — AES S-box (FIPS-197, GF(2⁸)-обращение + аффинное преобразование,
  вычислено из определения), обобщённый χ (Keccak/SHA-3, FIPS-202),
  бент-функция Майорана-МакФарланда — та же математика, что уже была
  проверена и задокументирована в постоянном `anf/bench_real_corpus.cpp`
  этой же сессии-предшественнице.
- **Thr** — Коллегия выборщиков США 2024-2028 (National Archives), тот же
  источник и веса, что в `thr/bench_real_corpus.cpp`.
- **Bdd/TruthTable** — не датасет сами по себе (производные структуры, не
  формат распространения данных) — получаются переводом функций выше,
  что и проверяется round-trip'ами.

Для больших входов (EPFL `router` — 60 входов; Коллегия выборщиков целиком —
51; χ/bent при n=40-64), где 2^n точек перебрать физически невозможно,
добавлено **приближённое сравнение через случайную выборку**
(`reprs_equivalent_sampled`, 2000 точек, детерминированный seed) — не
доказывает эквивалентность строго (это не SAT-солвер), но на практике ловит
почти любую структурную ошибку перевода.

Результаты — `verify/REAL_DATASETS_REPORT.md` (перегенерируется при каждом
запуске `test_real_datasets`). **Итог последнего прогона: 63 проверки, 0
провалов, 30.74с.**

### 5.1. Две новые находки, обнаруженные именно РЕАЛЬНЫМИ (не синтетическими) данными

При первых прогонах (ещё без защит ниже) тест реально зависал на несколько
минут — не из-за бага в самом тесте, а потому что он честно наткнулся на
два структурных риска, которые синтетические тесты предыдущих сессий не
вскрывали:

1. **`aig_to_anf`/`aig_to_bdd` на реальной схеме EPFL `router` (60 входов,
   arbiter-подобная управляющая логика)** — не укладываются в разумное время.
   `aig/README.md` ранее утверждал «`aig_to_anf` — ни разу не подвела»
   основываясь ТОЛЬКО на синтетических семействах (chain/star/block/mux/
   random) — на настоящей, произвольно устроенной схеме того же порядка
   размера (`n=60`) утверждение не подтвердилось. Это тот же класс риска,
   что уже задокументирован для `anf_to_bdd` в `anf/README.md` §5.2
   (структурный взрыв без предохранителя), просто впервые замечен и для
   `aig_to_anf`/`aig_to_bdd` — на РЕАЛЬНОМ, не намеренно сконструированном
   входе.
2. **`tt_to_thr`/`aig_to_thr`/`anf_to_thr` (распознавание пороговости через
   ILP) на реальных, НЕравномерных весах Коллегии выборщиков (3..54) —
   на порядки медленнее, чем на синтетических (единичных/MAJ-подобных)
   весах того же n.** Round-trip `Thr->Tt->Thr` при n=16 занял >10с (сам
   `thr_to_tt` — единицы мс; вся разница — в обратном ILP-шаге `tt_to_thr`);
   при n=20 не уложился и в несколько минут. Это КОЛИЧЕСТВЕННОЕ
   подтверждение уже известного (`thr/README.md` §3: «практический потолок
   тут — n≈16, а не n=32») наблюдения, но на новом, реальном примере веса
   — не подогнанном под тест синтетическом наборе.

**Меры защиты, добавленные в сам тест** (не в библиотеку — библиотека ведёт
себя как задокументировано, просто хуже, чем можно было предположить по
одним лишь синтетическим данным): для больших/структурно неизвестных
датасетов автоматический прогон допускает только переводы в `Tt`/`Thr`
(обе стороны дёшево и сразу проверяют размер статически, до тяжёлой работы)
и не пытается round-trip обратно в `Thr` для датасетов с реальными
неравномерными весами при n>=16 — не потому что это невозможно проверить, а
потому что для АВТОМАТИЧЕСКОГО, не присматриваемого прогона (`ctest`) риск
подвесить/уронить весь тестовый бинарник без возможности поймать это
изнутри (`Lace fatal error` — `abort()`, не ловится) перевешивает пользу.
Точечно, отдельным запуском с внешним `timeout`, эти случаи всё ещё можно
исследовать — как это уже делает `anf/README.md` §2 для `anf_to_bdd`.

## 8. Дополнительное улучшение: FORCE-эвристика порядка переменных для `aig_to_bdd`

По итогам §5.1/§7 — самая конкретная, проверяемая находка, которую можно
было закрыть в рамках этой сессии, а не просто задокументировать: `aig_to_bdd`
был единственной из трёх «структурно рискованных» функций, для которой уже
существовал проверенный образец решения в этом же проекте (`anf_to_bdd`,
FORCE-эвристика, `anf/README.md` §3.5/§5.2) — оставалось перенести тот же
механизм на другой источник графа взаимодействия переменных.

**Что сделано:**

1. Общая (не привязанная к ANF) часть `anf_to_bdd`'s эвристик —
   `enum VariableOrderHeuristic`, `InteractionGraph`, `identity_rank`,
   `compute_rank_by_degree`, `compute_rank_force` (алгоритм FORCE, Aloul/
   Markov/Sakallah 2003) — вынесена в новый `core/bdd_order_heuristics.hpp`.
   `anf/anf_to_bdd.cpp` переведён на этот общий код (оставив специфичное для
   ANF — `compute_rank_by_length_freq` и построение графа по мономам);
   рефакторинг проверен на регрессию (`test_anf anf_to_bdd` — PASS, тот же
   результат, что и до выноса).
2. `aig/aig_to_bdd.cpp` получил новую функцию `aig_to_bdd_with_heuristic(aig,
   heuristic)` и граф взаимодействия переменных, построенный ПО ФАНИНАМ
   AND-гейтов (а не по мономам, как в ANF): для каждого гейта — носитель
   (support, битовый вектор входных переменных) каждого фанина накапливается
   снизу вверх, рёбра добавляются между парами `support(fanin0) x
   support(fanin1)`. Добавлен предохранитель `kMaxCrossProductPerGate=4096`:
   если это произведение размеров носителей превышает порог, рёбра с этого
   гейта просто не генерируются (граф локально деградирует к отсутствию
   информации для этой пары фанинов) — иначе на широких реальных схемах
   (например, `voter.aig`, EPFL, n=1000, ~14760 гейтов) сама постройка графа
   стала бы узким местом. `aig_to_bdd(aig)` (умолчание) теперь эквивалентен
   `aig_to_bdd_with_heuristic(aig, Force)` — тем же обоснованием выбора
   умолчания, что и `anf_to_bdd` (см. `anf/anf_to_bdd.cpp`: FORCE не хуже
   1.95x от лучшей эвристики на 5 разных структурных семействах, тогда как
   альтернативы уходят в 8-12x на своих худших случаях).
3. Регрессия проверена: `test_aig` (все секции, включая `aig_to_bdd`),
   `test_chains` (23/23, без изменений в числах по сравнению с §4) —
   PASS, без изменений в поведении на существующих (небольших/синтетических)
   входах.

**Эмпирическая проверка находки из §5.1 (временный `verify/
diag_router_bdd.cpp`, удалён после того как числа записаны здесь — та же
методология, что и раньше в этой сессии для одноразовых диагностик):**
прямой вызов `aig_to_bdd_with_heuristic` на `router.aig` (EPFL, n=60) —
и с `Force`, и с `MinIndex` (натуральный PI-порядок, старое поведение) —
завершается за единицы-десятки миллисекунд (`Force`: 16.3мс/59 узлов,
`MinIndex`: 7.8мс/40 узлов). **Важное уточнение к §5.1/§7**: само построение
BDD для `router.aig` НИКОГДА не было медленным — ни до, ни после этого
изменения. Значит зависание, из-за которого исходно введён предохранитель в
`verify/real_datasets_tests.cpp`, было вызвано НЕ `aig_to_bdd`, а
`aig_to_anf` (что и написано текстом в самом предохранителе) — широкий
предохранитель просто заодно скрыл и `Aig->Bdd`, для которого проблемы не
было. `verify/real_datasets_tests.cpp` обновлён: предохранитель для
`Aig -> Bdd` теперь снят (см. `bdd_target_is_force_protected`), `Aig ->
Anf` остаётся под защитой (это по-прежнему непроверенный и неисправленный
риск). `epfl_router_po0: Aig->Bdd` и `Aig->Bdd->Aig` теперь реально
выполняются в автоматическом прогоне (`ctest`), а не пропускаются — итог
вырос с 63 до 65 проверок, 0 провалов, время прогона практически не
изменилось (30.74с -> 31.20с).

**Побочная находка при попытке расширить ту же защиту на `Anf -> Bdd`:** при
пробной проверке — раз уж `anf_to_bdd` тоже давно использует FORCE — сняли
предохранитель и для источника `Anf`, и обнаружили, что `bent_mm_n40`
(реальная бент-функция Майорана-МакФарланда, n=40) занимает у `anf_to_bdd`
**~48с** (round-trip — ~54с) даже с FORCE. Это НЕ баг и не то же самое, что
находка про `router.aig`: бент-функции по определению не имеют
эксплуатируемой линейной структуры (максимальная нелинейность) — известный в
литературе худший случай для компактности BDD при ЛЮБОМ порядке переменных,
никакая эвристика порядка тут принципиально не может помочь. Решение —
оставить `Anf -> Bdd` для больших non-exhaustive ANF-датасетов под тем же
предохранителем (не добавлять в автоматический `ctest`, это раздуло бы время
прогона всего файла в ~4 раза ради случая, который не является зависанием и
не чинится изменением кода), но задокументировать здесь как честно найденный
предел применимости `anf_to_bdd`/BDD как представления вообще для функций с
такими свойствами.

## 9. Продолжение: внешние датасеты (ANF-корпус, CNF→AIG) — два реальных бага найдены и исправлены

Отдельная сессия работы с внешними датасетами `benchmarks/data/` (найдены и
скачаны ранее, но не подключены к самой библиотеке): реальный ANF-корпус
`persons.iis.nsk.su` (n=100, M=10000 мономов) и DIMACS/SATLIB CNF-корпуса.
Задача сама по себе — «написать парсер и посмотреть, что будет» — по ходу
вскрыла два самостоятельных, воспроизводимых бага в существующем коде,
не только в новых парсерах.

### 9.1. Парсер `.anf` (`benchmarks/anf_dimacs_loader.hpp`)

Тривиальный формат (`p anf N M` + мономы построчно, 1-индексация в файле),
собирает `bmm::Anf` напрямую (BRiAl `BooleMonomial`/фолбэк `AnfFallback`).
Проверен на 3 реальных файлах корпуса (n=100, M=10000, разная плотность
переменных). Все три парсятся корректно; `anf_to_tt`/`anf_to_thr`
корректно и быстро отказывают (статические лимиты); `anf_to_aig` строит
58-74 тыс. гейтов за 50-140мс, эквивалентность подтверждена калиброванной
случайной выборкой (`Anf::evaluate()` при M=10000 стоит 40-240мс/вызов —
пришлось адаптировать число сэмплов вместо фиксированных 2000, тот же
принцип, что и предыдущая находка про `Anf::to_tt()`).

### 9.2. Баг №1 (найден и исправлен): крах `anf_to_bdd` — "Lace fatal error: Task stack overflow"

При попытке прогнать `anf_to_bdd` на этом же реальном плотном ANF получен
не медленный, а мгновенный КРАХ процесса (`Lace fatal error: Task stack
overflow! Aborting`, неперехватываемый `abort()`). Расследование (бинарным
поиском по параметру): причина НЕ в нативном C-стеке Lace-воркера
(`lace_set_stacksize()` — не помогло, проверено эмпирически, 256 MiB стека
крах не убирает), а в размере очереди SPAWN-задач Lace — втором параметре
`lace_start(n_workers, dqsize)`. Проект везде вызывал его с `dqsize=0`
("разумный дефолт" по документации Lace) — дефолта не хватает для объёма
рекурсивных `Ite()`-вызовов Sylvan на достаточно плотном/большом входе.
`dqsize=1<<18` — крах воспроизводится; `dqsize=1<<21` (2М записей/воркер) —
нет (проверено эмпирически на этом самом случае). Исправлено во ВСЕХ 5
местах, где вызывается `lace_start` в проекте (`verify/test_main.cpp`,
`verify/chain_tests.cpp`, `verify/real_datasets_tests.cpp`,
`anf/bench_real_corpus.cpp`, `thr/bench_real_corpus.cpp`) — это не
находка "в одной функции", а системная недостача ресурса, затрагивающая
любой вызов, использующий Bdd, если раньше кто-то полагался на дефолт.
После фикса — не крах, а честно медленное (~48-54с) поведение, то же самое
качество, что уже задокументировано для `bent_mm_n40` в §5.1.

### 9.3. Баг №2 (найден и исправлен): зависание `aig_to_anf` на реальных данных

`aig_to_anf` (обратный перевод) зависал (>180с) как на EPFL `router.aig`
(§5.1, n=60, **всего 317 гейтов**), так и на AIG, построенном из этого
плотного реального ANF (n=100, 73646 гейтов). Малый размер `router.aig`
здесь ключевой контраргумент: он доказывает, что СТАТИЧЕСКАЯ проверка по
размеру AIG (аналог `kMaxTruthTableVars`) здесь принципиально не работает
— маленькая схема тоже может дать экспоненциальный ANF, большая не
обязана взорваться. Опробованы (эмпирически, оставлены как дешёвый
дополнительный слой, но НЕ основной механизм — не остановили реальный
случай сами по себе): проверка размера результата ПОСЛЕ `mul_poly` и
проверка произведения длин операндов ПЕРЕД `mul_poly` — обе не помогли,
потому что стоимость размазана по десяткам тысяч гейтов, а не
концентрируется в одной операции. Рабочий фикс — **дедлайн по реальному
времени** (10с), проверяемый периодически в обеих ветках (BRiAl и
TBB-фолбэк): единственный механизм, не зависящий от того, ГДЕ именно
накапливается стоимость. `aig_to_anf` теперь возвращает честный
`Result`-отказ за секунды вместо зависания на минуты — на ВСЕХ
проверенных реальных случаях, включая CNF-производные AIG из §9.4 (даже
n=20 не проходит, см. там же). Заодно снят предохранитель для
`Aig -> Anf` в `verify/real_datasets_tests.cpp` (был общий блок-скип) —
`epfl_router_po0: Aig->Anf` теперь реально выполняется в автоматическом
`ctest`-прогоне и корректно даёт SKIP с honest-сообщением вместо тихого
пропуска.

Регрессия по обоим багам: полный `ctest` — 34/34, `real_datasets_tests` —
65/65 (было 63 до §8, без изменений от этих фиксов), время прогона не
изменилось.

### 9.4. Импортёр `CNF → Aig` (`benchmarks/cnf_dimacs_loader.hpp`)

DIMACS CNF (`benchmarks/data/dimacs-cnf`, `benchmarks/data/satlib` — 241 +
50306 файлов) не переводится в AIG через Tseitin (это кодирование для
СОХРАНЕНИЯ эквисатисфиability через вспомогательные переменные, обратное
направление уже есть в `verify/sat_encoding` для AIG→CNF) — единственная
осмысленная и единственно корректная цель, раз `bmm::Aig` требует РОВНО
один PO (CONVENTIONS.md п.4), это прямой ИНДИКАТОР выполнимости: CNF = AND
по клозам, клоза = OR по литералам. `mockturtle::aig_network::
create_nary_or`/`create_nary_and` уже реализуют это стандартными
AND/NOT-гейтами (De Morgan внутри `create_or`) — сам импортёр тривиален,
основная работа — разбор формата (переиспользует существующий
`bmm::verify::CnfFormula`/`CnfClause`, общий тип с `verify/sat_encoding`).

**Верификация — НЕ доверяя ни cnf_to_aig, ни mockturtle**: независимая
проверка через внешний SAT-солвер (CaDiCaL, уже подключённый проектом
через `verify/sat_encoding::solve_dimacs`) ДВАЖДЫ — (1) на исходной CNF
напрямую (эталон: SAT/UNSAT + witness), (2) на Tseitin-кодировании
ПОСТРОЕННОГО нами AIG с принудительным `po=true` — если импортёр корректен,
оба ответа обязаны совпасть, а witness исходной CNF обязан давать
`aig.evaluate(witness) == true`. Проверено на 5 отобранных вручную файлах
(`hole6.cnf` — классический pigeonhole, UNSAT; `aim-50`/`aim-100`, оба
`-no-`/`-yes1-` варианта) — все совпадения точные. Дополнительно — сплошной
прогон (`sweep`) по всем 241 файлам `benchmarks/data/dimacs-cnf`: 25-минутный
бюджет исчерпан на ~80+/241 (часть реальных SAT-инстансов сама по себе
занимает у CaDiCaL много времени — свойство конкретных инстансов, не
находка о библиотеке), 0 расхождений на всём пройденном пути; единственный
FAIL за всё время прогона — `faw.cnf` (не распарсился), корректно и
ожидаемо: этот файл — на деле письмо с прикреплённым текстом (email-шапка
"Date:/From:/Subject:" перед содержимым), не валидный DIMACS CNF —
парсер правильно его отверг, а не тихо принял мусор.

**Реальные CNF-производные AIG прогнаны через остальные переводы**
(`aig_to_bdd`/`aig_to_anf`/`aig_to_thr`/`aig_to_tt`, разные n):
- `uf20-01.cnf` (n=20, случайный 3-SAT): `Aig->Bdd` — 28мс, `Aig->Tt` —
  106мс, оба PASS. `Aig->Thr` — тоже PASS, но заняло **20.7с** (ILP-
  распознавание пороговости на функции без очевидной пороговой структуры
  дорого даже при n=20 — согласуется с уже известной характеристикой
  `aig_to_thr`, здесь подтверждено на реальных SAT-данных, не только на
  синтетике/весах Коллегии выборщиков из §5.1). `Aig->Anf` корректно и
  быстро (9мс) отказывает по бюджету/размеру (см. §9.3).
- `anomaly.cnf` (n=48, PLANNING/BlocksWorld): `Aig->Bdd` — 67мс, PASS.
  `Aig->Anf` отказывает за 3мс (не 10с дедлайн — здесь сработала более
  дешёвая проверка размера/операндов, см. §9.3, разные реальные случаи
  ловятся разными слоями защиты).
- `aim-100-1_6-yes1-1.cnf` (n=100, случайный 3-SAT-подобный AIM):
  **`Aig->Bdd` не укладывается и в 240с** — та же природа, что и
  `bent_mm_n40` (§5.1/§8): случайные/adversarial SAT-формулы — известный в
  литературе худший случай для BDD-компактности при ЛЮБОМ порядке
  переменных, не следствие отсутствия эвристики (FORCE уже включена по
  умолчанию). НЕ добавлено в автоматический прогон — задокументировано
  здесь как честно найденный предел применимости, симметрично §8.

### 9.5. Итог

Оба найденных бага (Lace `dqsize`, `aig_to_anf` без дедлайна) были
чисто инженерными — не связаны с реальными данными по существу, но
проявились ТОЛЬКО на реальных данных (плотный ANF/CNF-производный AIG
достаточного размера/сложности), потому что вся синтетика предыдущих
сессий была либо слишком мала, либо слишком структурирована, чтобы их
разбудить. Это иллюстрирует исходную цель работы с реальными датасетами
(задача пользователя в начале сессии) — они находят не только
"ожидаемо трудные" случаи (бент-функции, adversarial SAT), но и
самостоятельные баги в инфраструктуре, которые синтетические тесты
маскировали годами.

Оставшаяся задача — мост DDDMP→Sylvan — реализована и проверена, см. §10.

## 10. Мост DDDMP→Sylvan (`benchmarks/dddmp_loader.hpp`)

Третья из открытых задач: `benchmarks/data/iis-nsk/bdd-bench` — 895 готовых
BDD (формат DDDMP-2.0, CUDD) для классических аппаратных бенчмарков
(ISCAS/MCNC/IWLS/**EPFL**/...), плюс встроенные `.bdd.bdd`/`.fdd.fdd`/
`.zdd.zdd` эталоны во всех файлах ANF-корпуса `100-10k-rnd` (§9.1). DDDMP —
формат CUDD, не Sylvan — единственный практичный путь получить из него
`sylvan::Bdd`: загрузить через СОБСТВЕННЫЙ загрузчик CUDD
(`Dddmp_cuddBddArrayLoad`, встроен в `libcudd.a` этого образа как
зависимость BRiAl, `--enable-dddmp`) в `DdNode*`, затем рекурсивно
перестроить эквивалентный `sylvan::Bdd` через `Cudd_T`/`Cudd_E`/
`Cudd_IsComplement` + `Ite()`.

### 10.1. Препятствия при подключении API (все решены эмпирически)

- **`dddmp.h` требует `util.h`, которого нет в образе** (наследие пакетной
  структуры исходного дерева CUDD — только `cudd.h`/`dddmp.h` установлены
  в `/usr/local/include`, не весь пакет). Проверено эмпирически: ни
  `dddmp.h`, ни `cudd.h` не используют ни одного идентификатора ИЗ
  `util.h` напрямую — сам `#include` вестигиальный. Решение — пустая
  заглушка `benchmarks/cudd_stubs/util.h`, добавлена в include path ПЕРЕД
  системным в `CMakeLists.txt`.
- **Порядок включения `<cstdio>`/`<cstdlib>` ДО `cudd.h`/`dddmp.h`**:
  `dddmp.h` использует `FILE`/предполагает объявленным `free`, но сам не
  подключает `<stdio.h>`/`<stdlib.h>` — если C++ные `<cstdio>`/`<cstdlib>`
  подключены ПОСЛЕ него в том же TU, компиляция падает на "FILE has not
  been declared".
- **Конфликт с BRiAl при попытке объединить с `verify/chain_utils.hpp`**:
  BRiAl тянет СОБСТВЕННЫЙ bundled `polybori/cudd/cudd.h` (с иным именем
  типа менеджера через `PBORI_PREFIX`) — оба заголовка в одной единице
  трансляции дают "conflicting declaration" на десятках функций CUDD.
  Решение — держать любой код, использующий `benchmarks/dddmp_loader.hpp`,
  СТРОГО отдельно от `core/anf_repr.hpp`/BRiAl; там, где нужно сравнить
  `Anf` с загруженным DDDMP `Bdd` (§10.3), данные обмениваются через
  промежуточный файл между двумя независимыми бинарями (`verify/
  diag_anf_sample_dump.cpp` → пишет точки + `Anf::evaluate()`, `verify/
  diag_dddmp_sweep.cpp` → читает и сверяет с загруженным `Bdd`) — тот же
  приём, что уже использован для witness SAT-солвера в §9.4.
- **`Dddmp_cuddBddArrayLoad` с `rootMatchMode=DDDMP_ROOT_MATCHNAMES` и
  `rootmatchnames=nullptr` — сегфолт** (эмпирически подтверждено):
  `MATCHNAMES` безусловно ожидает валидный массив имён для сопоставления.
  `DDDMP_ROOT_MATCHLIST` — "взять все корни файла как есть" — то, что
  нужно, и не падает при `nullptr`.

### 10.2. Находка при первом прогоне на разнообразных реальных файлах (расследовано, оказалось НЕ багом)

Первый прогон широкого набора `bdd-bench` завис на одном из файлов дольше
минуты. Расследование (добавлен временный счётчик рекурсивных вызовов +
`Cudd_DagSize` каждого загруженного корня): НЕ баг и не бесконечная
рекурсия — `Cookbook__arbiter.blif.bdd` действительно имеет корень с
`DagSize=131056` в СОБСТВЕННОМ представлении CUDD (32 переменные, но с
неудачной для BDD структурой) — перевод такого объёма узлов в Sylvan
(через несколько сотен тысяч `Ite()`-вызовов) занимает заметное, но
конечное время; попытка изолированно перепроверить файл, который
ПЕРВОНАЧАЛЬНО показался зависшим (`Cookbook__bilbo_lfsr.blif.bdd`, всего
201 узел) — на деле загрузился мгновенно: `std::filesystem::
directory_iterator` не гарантирует алфавитный порядок, поэтому визуально
"следующий" в выводе файл не обязательно был файлом, на котором
происходила остановка. Урок (пригодится и за пределами этой сессии):
не делать выводов о том, "какой файл завис", по одному только порядку
вывода `directory_iterator` — проверять конкретный файл изолированно.

Побочный, но реальный и оставленный в коде результат этого расследования:
мост теперь дополнительно сохраняет РЕАЛЬНЫЙ порядок переменных исходного
CUDD BDD (`Cudd_ReadPerm`) и передаёт его в `Bdd` через явный `var_to_level`
(`Bdd(root, n_vars, var_to_level)`, тот же паттерн, что и
`aig_to_bdd_with_heuristic`/`anf_to_bdd_with_heuristic`, §8) — компактность
исходного CUDD BDD достигается за счёт СОБСТВЕННОГО, обычно неидентичного
порядка переменных CUDD; наивная реконструкция в Sylvan с `bddVar
(логический_id)` эквивалентна ИДЕНТИЧНОМУ порядку и рисковала бы тем же
классом взрыва, что уже задокументирован для `aig_to_bdd`/`anf_to_bdd` без
эвристики (§8) — этот риск теоретический (не было эмпирически
воспроизведено расхождение именно из-за этого на проверенных файлах, все
`var_to_level` в проверенных случаях оказались identity-подобны), но
исправление корректно и дёшево, оставлено как более правильная реализация
по построению, а не только "работает на проверенных файлах".

### 10.3. Верификация — НЕЗАВИСИМАЯ, третьей стороной, без доверия к собственному коду

Лучшая доступная проверка для этого моста — не самопроверка, а сверка
против того же самого, но ПОСЧИТАННОГО ДРУГИМ ИНСТРУМЕНТОМ результата:

- **`bdd-bench` содержит категорию `EPFL__*`, совпадающую с уже
  использованными в этой сессии `benchmarks/data/epfl/random_control/
  {ctrl,int2float,cavlc,router}.aig`** (`EPFL__ctrl.blif.bdd`,
  `EPFL__int2float.blif.bdd`, `EPFL__cavlc.blif.bdd`,
  `EPFL__router.blif.bdd`) — независимая, третьей стороной посчитанная
  контрольная точка именно для `aig_to_bdd`. Прогон: построить `Bdd` через
  `aig_to_bdd(aig)` (наш код) И независимо загрузить `Bdd` того же PO 0
  через мост, сравнить (`ctrl`/`int2float`/`cavlc` — перебор всех точек,
  n=7/11/10; `router` — выборка 5000 точек, n=60). **Все 4 — PASS,
  включая router.aig** (тот самый случай, для которого в этой сессии уже
  была отдельно подтверждена находка про FORCE-эвристику, §8) — NodeCount
  отличается (8 vs 9, 73 vs 147, 80 vs 81, 59 vs 40 — разные порядки
  переменных у FORCE и у того, что использовал CUDD, оба валидны), но
  СЕМАНТИКА функции идентична, подтверждено независимо.
- **Встроенный эталон для реального ANF n=100/M=10000** (§9.1-9.2):
  `10-100x90-100.01.anf.bdd.bdd` — тот самый файл, для которого
  `anf_to_bdd` собственный проекта СОБСТВЕННЫЙ не может практически
  посчитать ответ (структурный взрыв, §9.1). Через обмен точками между
  двумя бинарями (см. §10.1) — 30 случайных точек, `Anf::evaluate()`
  против `Bdd::evaluate()` загруженного стороннего эталона — **все 30
  совпали.** Валидирует, что представление `Anf` в проекте корректно НА
  РЕАЛЬНЫХ ДАННЫХ там, где у нас единственный способ проверить это —
  сторонний, независимо посчитанный эталон (собственный `anf_to_bdd`
  недоступен как способ проверки именно для этого случая). Заодно
  подтверждает независимо, что NodeCount=141859 у этой функции — реальное
  свойство самой функции (подтверждено third-party расчётом, не артефакт
  нашей реализации `anf_to_bdd`/FORCE-эвристики).
- **Общий прогон загрузки** по всем 895 файлам `bdd-bench` (только
  успешная загрузка, без сравнения) — первые 8 файлов (в реальном
  алфавитном порядке `directory_iterator`, включая `Cookbook__arbiter.
  blif.bdd` с честными 131056 узлами в CUDD) загружаются корректно и за
  разумное время; девятый (`Cookbook__bilbo_lfsr.blif.bdd`, всего 201 узел
  — тривиальный по размеру) внутри ЭТОГО многочасового прогона зависает,
  хотя ИЗОЛИРОВАННО (тот же файл, тот же бинарь, но единственный аргумент)
  грузится мгновенно. Добавление `sylvan_gc()` между файлами (гипотеза:
  накопление узлов `arbiter` в общей Sylvan-таблице процесса) НЕ решило
  проблему — то есть накопление таблицы не единственная причина. Учитывая,
  что (а) сам мост уже независимо верифицирован корректным на реальных
  данных обоих типов (§10.3), и (б) реальный потребитель моста грузит один
  файл за раз, а не 895 подряд в одном процессе — эта находка
  задокументирована как ОТКРЫТЫЙ ВОПРОС про многофайловый прогон именно
  ЭТОГО диагностического инструмента (`verify/diag_dddmp_sweep.cpp` в
  режиме `load`), не как дефект `benchmarks/dddmp_loader.hpp` — дальнейшее
  расследование (профилирование Sylvan/CUDD `Cudd_Init`/`Cudd_Quit`
  накопления при многократном создании/уничтожении CUDD-менеджера в одном
  процессе) оставлено за рамками этой сессии.

### 10.4. Итог

Мост DDDMP→Sylvan реализован (`benchmarks/dddmp_loader.hpp`,
`load_dddmp_bdds`/`load_dddmp_bdd`) и независимо верифицирован против
third-party эталонов на реальных данных обоих типов, для которых был
изначально задуман (готовые BDD реальных схем И встроенные эталоны в
ANF-корпусе) — обе проверки пройдены. Открывает практический путь для
будущей работы: 895 файлов `bdd-bench` как датасет для `bdd_to_*`
(в дополнение к уже существующим `aig_to_bdd`-производным датасетам,
§5/§8), и 165 встроенных `.bdd.bdd`/`.fdd.fdd`/`.zdd.zdd` эталонов как
независимый способ проверки `anf_to_bdd`/`Anf` на n=100 без необходимости
полагаться на саму библиотеку для получения "правильного ответа".
