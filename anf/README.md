# anf/ — состояние трансляций, лучшие практики и находки

Пять функций трансляции с ANF (полином Жегалкина, BRiAl) на входе или
выходе: `tt_to_anf`, `anf_to_aig`, `anf_to_bdd`, `anf_to_thr`, `anf_to_tt`.
Этот файл — не описание API (см. doc-комментарии в соответствующих `.hpp`)
и не журнал изменений (см. `git log`), а **рассуждения и накопленные
находки**, которые стоят за нетривиальными решениями в этой папке — то, что
не выводится напрямую из кода, а было найдено экспериментально или
продумано отдельно. §1-2 — сводка для тех, кто хочет быстро понять текущее
состояние и как безопасно пользоваться этими функциями; §3 и далее —
подробный разбор, откуда каждый вывод взялся.

## 1. Обзор: состояние пяти трансляций

| Функция | Backend | Параллелизм | Корректность | Жёсткая граница n | Практическая граница | Известный отказ |
|---|---|---|---|---|---|---|
| `tt_to_anf` | BRiAl (`BoolePolynomial`) / `AnfFallback` | OpenMP (только Мёбиус-трансформ) | ✅ ctest `run_translation_tests` | n≤24 (`kMaxTruthTableVars`) | = жёсткой — но на **плотных** входах сборка полинома доминирует по времени, см. §5.4 | нет краха, только медленно на плотных n≈20-24 |
| `anf_to_tt` | BRiAl / `AnfFallback` | OpenMP (только Мёбиус-трансформ) | ✅ ctest | n≤24 | то же, что `tt_to_anf` (симметричная функция) | то же |
| `anf_to_aig` | `mockturtle::aig_network` | нет (TBB рассмотрен и отклонён, см. §4) | ✅ ctest | нет жёсткой | не найдено ни одного отказа вплоть до n=64 на всём корпусе §5 | не найден |
| `anf_to_bdd` | Sylvan/Lace (`sylvan::Bdd`) | нет (архитектурно запрещён, см. §3) | ✅ ctest (включая `Bdd::evaluate()`) | нет жёсткой | **зависит от структуры входа, не только от n** — см. §5.3 | экспоненциальный взрыв → крах процесса (`abort()`) на входах, чей граф взаимодействия переменных близок к паросочетанию с "далёкими" по индексу парами (пример — бент/IP-функции, крах уже при n≈44) |
| `anf_to_thr` | OR-Tools ILP (SCIP→SAT→CBC) | нет (сознательно, стоимость в самом solve) | ✅ ctest | n≤20 (проверка в коде) | **n≈12-16 на реальных (не пороговых по построению) функциях** — при n=20 одна точка уже занимала 34с | не крах, а неприемлемая по времени задержка на любой нетороговой функции |

Коротко: `anf_to_aig` — самая надёжная функция в папке, ни разу не подвела
ни на одном тесте этой сессии. `anf_to_bdd` и `anf_to_tt`/`tt_to_anf`
корректны всегда, но у каждой есть свой класс входов, на которых она
становится непригодной по производительности (у `anf_to_bdd` — вплоть до
краха процесса). `anf_to_thr` — единственная, чья практическая граница
серьёзно уже заявленной в коде контрактной.

## 2. Лучшие практики

**Перед вызовом `anf_to_bdd` на функции с неизвестной/непроверенной
структурой** (n≥30-40): если известно или подозревается, что переменные
входа естественно распадаются на пары/малые группы, которые встречаются
только друг с другом (паросочетание или близко к нему в графе
взаимодействия переменных — типичный случай для IP/бент-конструкций,
некоторых потоковых шифров) — переиндексируйте переменные так, чтобы члены
одной группы были СОСЕДНИМИ по индексу, ДО вызова. Ни одна из 4 встроенных
эвристик (§3.4) не решает это сама, см. §5.3 за количественным
подтверждением (`NodeCount` = `2^(n/2+1)-1` против `n+1` — вся разница
только в переиндексации). Если структура неизвестна и n большой — сначала
прогоните на меньшем n или в отдельном процессе с таймаутом: `Lace fatal
error` — это `abort()`, ловить его внутри вызывающего процесса нельзя.

**Перед вызовом `anf_to_bdd` на структуре, похожей на "звезда"** (одна
управляющая/хаб-переменная участвует почти во всех мономах) — умолчание
`Force` систематически слабее на таком классе (до 1.95× хуже лучшей
эвристики, см. §3.5.2). Вызовите `anf_to_bdd_with_heuristic(anf,
VariableOrderHeuristic::Degree)` явно.

**Перед вызовом `anf_to_thr`** на функции, которая не гарантированно
пороговая по построению — ожидайте секунды на n=16 и десятки секунд на
n=20 (см. §5.2), а не миллисекунды, которых можно было бы ожидать по
аналогии с остальными функциями папки. Если нужна массовая проверка
"является ли функция пороговой" на потоке реальных (не специально
сконструированных) кандидатов — закладывайте это время в бюджет заранее,
не только на пилотном прогоне.

**Перед вызовом `tt_to_anf`/`anf_to_tt` на плотной функции** (заметная доля
от 2ⁿ коэффициентов ненулевая) при n≥18-20 — основное время уйдёт в
последовательную сборку `BoolePolynomial` из коэффициентов, а не в сам
(параллельный) Мёбиус-трансформ, см. §5.4. На разреженных функциях
(типичные крипто-примитивы вроде χ/бент — единицы-десятки мономов даже при
n=64) этот эффект не проявляется вообще.

**Общий принцип для всех пяти функций**: ни одна не имеет универсальной
гарантии производительности на "любой" функции того же n — все
эвристики/солверы здесь принципиально приближённые (см. §3.6, cutwidth —
NP-трудная задача) или чувствительны к плотности/структуре. Граница n,
безопасная для одного класса входов, необязательно безопасна для другого
— решает не n сам по себе, а структура графа взаимодействия переменных
(для `anf_to_bdd`/`anf_to_thr`) или плотность коэффициентов (для
`tt_to_anf`/`anf_to_tt`). `anf_to_aig` — единственное исключение, за всё
время работы над этой папкой не нашлось входа, на котором она вела бы
себя плохо.

## 3. anf_to_bdd — почему порядок переменных вообще важен

### 3.1. Постановка задачи

`anf_to_bdd` строит BDD рекурсивным разложением Шеннона по ANF-полиному:
`p = p0 XOR x*p1`, узел — `Ite(bddVar(var), BDD(p1_xor_p0), BDD(p0))`. На
каждом шаге нужно решить, **какую переменную `var` разложить следующей** —
это и есть "порядок переменных". Для BDD порядок переменных — не
второстепенная деталь: для одной и той же функции BDD с "удачным" порядком
может быть линейного размера, а с "неудачным" — экспоненциального (это не
теоретическая абстракция: разница в наших измерениях доходила до 10×+ на
n=64 в §3.5, и до **экспоненциальной вплоть до краха процесса** на n=44 в
§5.3). Задача найти оптимальный порядок в общем случае NP-трудна (это
отдельно объясняется в §3.6) — поэтому весь дальнейший текст про
**эвристики**, не про точное решение.

### 3.2. Находка №1: порядок разложения в рекурсии сам по себе НЕ влияет на размер BDD

Первая интуиция (у которой физически напрашивалось "мы обходим полином в
удачном порядке — значит и BDD получится удачный") оказалась неверной, и
это стоило отдельной эмпирической проверки. Причина:
`sylvan::Bdd::bddVar(idx)` вызывает `sylvan_ithvar(idx)` — **физический
уровень** переменной в общей (shared, на весь процесс) BDD-структуре
Sylvan фиксирован её **индексом**, а не тем, в каком порядке наш код вызвал
`bddVar`/`Ite`. Без явной динамической переупорядочки (sifting) этот
уровень неизменен — и по теореме Брайанта о канонической форме ROBDD, для
фиксированного порядка уровней редуцированный BDD **единственен**.

Экспериментально (реальная сборка, ~65 тестовых случаев, включая
специально сконструированные асимметричные мультиплексоры): `NodeCount()`
у `choose_variable_min_index` (разложение по возрастанию индекса) и
`choose_variable_by_rank`/`compute_rank_by_length_freq` (разложение по
длине монома + частоте) оказался **буквально идентичен** на каждом случае.
Разница была только во **времени** построения (эффективность мемоизации —
удачный порядок разложения чаще приводит к совпадающим подполиномам между
ветками рекурсии, чаще попадая в кэш), причём разнонаправленная:
`min_index` быстрее на симметричных/случайных функциях, `length_freq_rank`
— на асимметричных структурах (мультиплексор-подобных). Именно поэтому
самая первая версия `anf_to_bdd` (до появления физической перестановки
уровня, см. §3.3) умолчанием использовала эту эвристику — единственным
критерием сравнения тогда было время.

### 3.3. Находка №2: физически переставить уровень можно, но `Bdd::Permute()` не помогает

Раз порядок разложения не переставляет физический уровень, единственный
способ реально получить другой (более компактный) порядок переменных в
итоговом BDD — строить узлы с `bddVar(order[var])` вместо `bddVar(var)`,
т.е. использовать индексы Sylvan напрямую как позиции в желаемом порядке.
Это подтверждённый, воспроизводимый эффект: на one-hot мультиплексоре 4×4
(n=36) четыре разных физических порядка дают четыре разных `NodeCount()`
— 571/1215/466/153 (натуральный/по частоте/по степени/FORCE
соответственно, см. §3.5) — то есть размер BDD *действительно* можно менять,
если переставлять физические индексы Sylvan, а не порядок обхода рекурсии
(в отличие от §3.2, где все эвристики давали один и тот же размер).

Но это ломает контракт "уровень Sylvan `i` == бит `assignment[i]`", на
котором молча стоят `Bdd::evaluate()` и потребители `Bdd::raw()`
(`bdd_to_aig.cpp`, `bdd_to_tt.cpp`). Было два кандидата на решение:

- **Вариант 1 — построить в переставленном порядке, потом `Bdd::Permute()`
  обратно к натуральному.** Проверено эмпирически (временный
  `bench_permute_test.cpp`, удалён после проверки): **не работает.**
  `Permute()` — это подстановка переменных (функциональная композиция), а
  не переименование существующей структуры. После `Permute()` обратно к
  натуральному порядку `NodeCount()` **точно совпадает** с тем, что дало бы
  построение сразу в натуральном порядке (25→25 при n=10, 571→571 при
  n=36) — выигрыш полностью пропадает. Это ожидаемо: раз Sylvan всегда
  хранит `level == index`, ROBDD для фиксированного (натурального) порядка
  единственен по теореме Брайанта — неважно, как к нему прийти.
- **Вариант 2 — расширить контракт `Bdd`.** Выбран. В `core/common.hpp`
  добавлено необязательное поле `var_to_level` (пусто = identity = текущее
  поведение всех остальных производителей `Bdd`) и метод
  `Bdd::var_at_level(level)`. `Bdd::evaluate()` использует
  `var_to_level_[i]` вместо `i`. Оба реальных потребителя `Bdd::raw()`/
  `TopVar()` поправлены: `bdd_to_aig.cpp` и `bdd_to_tt.cpp` вызывают
  `f.var_at_level(level)` вместо того, чтобы молча считать уровень
  индексом переменной. `bdd_to_anf.cpp`/`bdd_to_thr.cpp` — пока стабы
  (`NotImplemented`), но в их `.hpp` оставлены предупреждения на будущее.

Корректность всего механизма проверяется не отдельным юнит-тестом, а тем,
что штатный `verify::run_translation_tests` (`ctest` секция `anf_to_bdd`)
уже активно дёргает `Bdd::evaluate()` на выходе `anf_to_bdd()`
(ground_truth + Chow-параметры + степень АНФ + кофакторы) — так что любая
ошибка в `var_to_level`/`var_at_level` проявилась бы как FAIL на этом
тесте, а не молча.

### 3.4. Четыре эвристики

Все — в анонимном namespace `anf_to_bdd.cpp`, доступны через
`enum class VariableOrderHeuristic` (`anf_to_bdd.hpp`) и
`anf_to_bdd_with_heuristic(anf, heuristic)`. `anf_to_bdd(anf)` — тонкая
обёртка с зашитым умолчанием (`Force`, см. §3.5).

| Эвристика | Идея | Стоимость |
|---|---|---|
| `MinIndex` | Возрастающий индекс переменной, без учёта структуры | O(1) (identity) |
| `LengthFreqRank` | Ранг по (макс. длина монома переменной, затем частота), считается один раз | O(size полинома) |
| `Degree` | Ранг по числу РАЗЛИЧНЫХ соседей в графе взаимодействия переменных | O(Σ Lᵢ²) на граф + O(n log n) на сортировку |
| `Force` | Алгоритм FORCE (Aloul, Markov, Sakallah, 2003): 20 итераций "притяжения" к взвешенному центру тяжести соседей по графу взаимодействия | O(итерации × рёбра графа) |

Граф взаимодействия переменных (узлы — переменные ANF, ребро `u`-`v` —
встретились в одном мономе, вес — число совместных вхождений) строится
один раз, `O(Σ Lᵢ²)` по мономам длины `Lᵢ` — общая основа для `Degree` и
`Force`. Хранится как список смежности (hash map), не как плотная матрица
`n×n`.

`choose_variable_min_index` (старая отдельная функция) сохранена в коде
как `[[maybe_unused]]` — историческая точка сравнения, её роль теперь
играет `identity_rank` + generic `choose_variable_by_rank`.

**Известный, аналитически доказанный блайндспот всех четырёх** — на
графе-паросочетании (каждая переменная имеет ровно одного соседа) все
четыре эвристики вырождаются в порядок, максимально РАЗНОСЯЩИЙ пару, а не
сближающий её — см. §5.3 за полным разбором и §2 за практическим обходом.

### 3.5. Боевое сравнение — результаты

Методология (везде одинаковая): 1 прогрев + медиана 11 повторов, реальная
сборка в Docker-контейнере (`genetica-boolean-lib`), временные
bench-программы (`bench_bdd_heuristics.cpp`, `bench_bdd_heuristics_v2.cpp`)
удалены после того, как числа записаны здесь и в
`core/CONVENTIONS.md`-подобной памяти проекта (вне репозитория, см.
итоговый комментарий в `anf_to_bdd.cpp`).

#### 3.5.1. Первый раунд — one-hot мультиплексор

Иерархический one-hot мультиплексор (bank-биты на высоких индексах,
sub/data-биты — на низких, специально без отрицаний — BRiAl разворачивает
`(1+var)` по дистрибутивности и портит структуру монома, см. §4):

| размер | min_index | length_freq_rank | degree | force |
|---|---|---|---|---|
| 3×3 (n=21), NodeCount | 127 | 151 | 106 | **55** |
| 4×4 (n=36), NodeCount | 571 | 1215 | 466 | **153** |
| 4×4 (n=36), время, мс | 6.0 | 25.4 | 8.8 | **2.9** |

`force` победил с большим отрывом — на этом основании умолчание было
переключено с `length_freq_rank` на `force`. Но это проверка только на
**одном** семействе структур.

#### 3.5.2. Второй раунд — пять разнородных семейств, n до 64

По запросу пользователя: chain (путь, `x_i·x_{i+1}`), star/hub (одна
переменная во всех мономах — намеренно на *высоком* индексе, худший случай
для `min_index`), block (несвязные клики по 4 переменные), random sparse
(случайные мономы степени 2-3, без специальной структуры), плюс ещё
размеры one-hot mux. Крупнейший прогнанный размер на семейство:

| семейство (n) | min_index | length_freq_rank | degree | force |
|---|---|---|---|---|
| chain (64) | 127 | 1103 | 247 | **127** |
| star/hub, центр x_{n-1} (64) | 127 | **65** | **65** | 127 |
| block, клики по 4 (64) | **65** | 145 | **65** | **65** |
| random sparse (24) | 1711 | 1627 | 1793 | **149** |
| one-hot mux 4×7 (60) | 4771 | 9827 | 3826 | **1273** |

**Ни одна эвристика не доминирует на всех пяти семействах.** Минимаксное
сожаление (во сколько раз эвристика хуже лучшей в своём худшем случае):

- `min_index` — до 11.5× (random)
- `length_freq_rank` — до 10.9× (random), 8.7× (chain)
- `degree` — до 12.0× (random)
- `force` — максимум **1.95×** (star/hub), везде остальном — либо лучшая,
  либо наравне с лучшей

`force` остаётся умолчанием: её худший случай кратно мягче худшего случая
любой другой эвристики. Если для конкретного прикладного класса ANF
заранее известно, что он структурно похож на star/hub (доминирующая
управляющая переменная с высоким индексом) — вызывайте
`anf_to_bdd_with_heuristic(anf, VariableOrderHeuristic::Degree)` явно (см.
также §2).

**Авария по пути**: первая версия random sparse (2n мономов, степень 2-4)
взорвалась на n=32 — сотни тысяч узлов, десятки секунд на один вызов у
каждой эвристики, контейнер пришлось останавливать вручную. Не баг —
ожидаемое свойство неструктурных функций (см. §3.6). Плотность урезана
(n мономов вместо 2n, степень 2-3, n ≤ 24 для этого семейства).

### 3.6. Почему это можно (частично) предсказать математически

Размер BDD, построенного из "локальных" термов (мономы ANF, клозы CNF и
т.п.), контролируется **cutwidth/bandwidth графа взаимодействия
переменных** при выбранном порядке — классический факт теории decision
diagrams (используется, например, при компиляции CSP/SAT в relaxed BDD).
Хороший порядок = порядок с маленьким cutwidth = компактный BDD. Найти
порядок с минимальным cutwidth — NP-трудная задача, поэтому все четыре
эвристики выше — это разные приближённые эвристики именно для неё.

Зная это, часть результатов §3.5.2 предсказуема без запуска бенчмарка:

- **chain/block — почему `length_freq_rank` проваливается**: в обеих
  структурах все мономы **одинаковой длины** по построению (2 для chain, 4
  для block). Первичный ключ сортировки этой эвристики ("длина монома")
  вырождается в константу, остаётся только частота — а она тоже почти
  униформна (в block — у каждой переменной частота ровно 1). Эвристика
  теряет весь сигнал именно там, где структура простая.
- **block — почему `min_index` побеждает**: блоки построены как **смежные
  диапазоны индексов** (0-3, 4-7, ...) — натуральный порядок уже держит
  каждый блок вместе. Это свойство теста, не универсальное свойство
  `min_index`.
- **star/hub — почему проваливается именно `force`**: FORCE — квадратичное
  размещение (спринг-модель): минимизирует Σ(pos(u)-pos(v))². Для
  графа-звезды минимизатор этой суммы притягивает переменную с самой
  высокой степенью (хаб) к **центроиду соседей**, то есть к **середине**
  порядка. А для decision diagram хаб нужно решать **первым или
  последним** (крайняя позиция), чтобы cutwidth оставался маленьким —
  системное расхождение между квадратичной релаксацией и настоящей целью
  (минимальная линейная расстановка/cutwidth), задокументированное в
  литературе по VLSI-размещению именно для hub-топологий. `degree`/
  `length_freq_rank` здесь побеждают не случайно — они прямо ранжируют хаб
  первым по частоте/степени, без геометрической релаксации.
- **random — почему результат шумный**: раз задача NP-трудна, ни у одной
  локальной эвристики нет гарантии на конкретном экземпляре. Расхождение
  `force` в одну и другую сторону в зависимости от n/seed — ожидаемое
  поведение приближённого алгоритма без гарантии глобального оптимума, а
  не аномалия бенчмарка.
- **паросочетание (бент/IP-функции, §5.3) — почему проваливаются ВСЕ
  четыре**: cutwidth-логика выше объясняет относительное поведение эвристик
  друг относительно друга на структурах, где у них есть хоть какой-то
  сигнал для различения переменных. На чистом паросочетании (каждая
  переменная — ровно один сосед) сигнал отсутствует у всех сразу
  (единообразная степень/длина/частота) — вырождение в identity-подобный
  порядок не специфично для одной эвристики, это общий случай отсутствия
  структуры, различимой заложенными в код признаками.

Что предсказать было бы нельзя без прогона — это точные коэффициенты
(почему именно 8.7× на chain, а не 5× или 12×) и поведение на "генерически
случайных" структурах — для этого пришлось бы фактически решать NP-трудную
задачу точно.

## 4. Другие архитектурные решения (короче, см. код)

- **`anf_to_aig.cpp` — TBB рассмотрен и отклонён.** `mockturtle::aig_network`
  не потокобезопасна для конкурентного `create_and()`. Вариант "строить
  каждый моном в изолированной локальной сети, потом сливать" реализован и
  измерен на реальных данных (n=16..64, вплоть до 500 мономов) —
  устойчивого выигрыша не нашлось (от +150% до -40% в зависимости от
  конфигурации, без закономерности) из-за фиксированной цены создания
  `mockturtle::aig_network` на каждый моном. Файл остался последовательным,
  подробное обоснование — в комментарии в начале `anf_to_aig.cpp`. Это же —
  причина, почему `anf_to_aig` оказалась самой надёжной функцией папки
  (§1): вся сложность вынесена в проверенную библиотеку mockturtle, своей
  рекурсии/эвристик/солверов, которые могли бы завести в угол на
  специфической структуре входа, здесь просто нет.
- **`anf_to_bdd.cpp` — параллелизм архитектурно запрещён.** Bdd — только
  через Sylvan/Lace (core/CONVENTIONS.md), никакого TBB/OpenMP.
  `BoolePolynomial` физически не влезает в фиксированный слот
  `lace_task`, так что даже "легитимный" путь (Lace SPAWN/SYNC для самого
  `convert()`) потребовал бы переписать рекурсию на POD-хэндлы — не
  сделано (см. §7 за ссылкой на подробную хронологию этого тупика).
- **`anf_to_thr.cpp`** — солвер-фолбэк цепочка SCIP→SAT→CBC вместо жёстко
  заданного CBC (та же логика, что в `aig/aig_to_thr.cpp`). TBB сознательно
  не добавлен — стоимость функции почти целиком в самом ILP-solve, а не в
  последовательном построении ограничений.
- **`tt_to_anf.cpp`/`anf_to_tt.cpp` — OpenMP только для Мёбиус-трансформа.**
  Изначальный выбор (transform естественно ложится на плоский
  butterfly-паттерн, как в FFT) был структурно разумен, но эмпирически не
  дал выигрыша (§5.4) — сборка `BoolePolynomial` из коэффициентов,
  оставшаяся последовательной, доминирует по времени на плотных входах и
  никогда не была отдельным объектом оптимизации.
- **BRiAl-отрицание ломает структуру монома**: `(1+var)` разворачивается
  BRiAl'ем по дистрибутивности при умножении — попытка построить
  мультиплексор с отрицанием дала 81 моном длины 1-5 вместо задуманных
  16×5. One-hot кодирование (§3.5.1) — способ обойтись вообще без отрицаний
  там, где нужна намеренно асимметричная тестовая структура.

## 5. Корпус реальных функций (`anf/bench_real_corpus.cpp`) — методология и находки

В отличие от §3.5 (синтетические семейства chain/star/block/random/mux),
`bench_real_corpus.cpp` — ПОСТОЯННЫЙ инструмент (не удаляется после
прогона), гоняющий все 5 функций `anf/` на функциях, у которых источник —
математическое определение реального объекта, а не искусственно
сконструированная структура:

- **AES S-box** (n=8, 8 координатных функций) — таблица не переписана по
  памяти (см. предупреждение в `benchmarks/scripts/generate_sbox.cpp` о
  риске опечатки в таком переписывании), а вычислена в самом
  `bench_real_corpus.cpp` из математического определения Rijndael
  (мультипликативный обратный в GF(2⁸) + аффинное преобразование) —
  проверено дважды независимо (алгебраическое вычисление на PowerShell вне
  контейнера, побайтовое совпадение с Wikipedia).
- **χ (Keccak/SHA-3, Ascon)**, обобщённый с канонических 5 бит на
  произвольный n: `out_i = a_i XOR a_{i+2} XOR a_{i+1}·a_{i+2}` (mod n) —
  чистая формула, без таблиц. Растёт до n=64.
- **Бент-функция Майорана-МакФарланда (IP)**: `f(x,y) = XOR x_i·y_i` —
  каноническая бент-функция из крипто-литературы, чётный n до 64.

### 5.1. anf_to_thr на реальных (не пороговых по построению) функциях — на 3-6 порядков медленнее всего остального в этом же корпусе

Прежний тестовый набор для `anf_to_thr` (`growing_threshold_test_functions`
в `verify/reference_builders.cpp`) — AND/OR/MAJ, все пороговые ПО
ПОСТРОЕНИЮ, солвер сразу находит решение. `chi`/`bent` почти никогда не
пороговые функции (степень ≥2) — солвер обязан **доказать
инфизибильность**, качественно более тяжёлый путь:

| функция | n | anf_to_thr |
|---|---|---|
| chi | 16 | 1.7 с |
| chi | 20 | 34.3 с (не вошло в штатный `run_all()` — весь прогон корпуса шёл в тайм-аут) |
| bent | 16 | 6.0 с |

Для сравнения — все остальные 4 функции на тех же входах укладываются в
единицы миллисекунд даже при n=64. Комментарий в `anf_to_thr.cpp` ("LP
становится огромным после 20-22 переменных") был качественно верен, но
никто не измерял, насколько плохо уже при n=16-20 — реальные (не
специально пороговые) входы гораздо чаще бьют именно по медленному
(инфизибильность) пути, чем синтетический набор, гарантированно
пороговый по построению.

### 5.2. Главная находка: anf_to_bdd — экспоненциальный взрыв на bent/IP-функции при "разнесённой" паре переменных

`build_bent_mm(n)`: `f(x,y) = XOR_i x_i·y_i`, где `y_i = x_{n/2+i}` — пара
переменных одного монома разнесена на `n/2` позиций в естественном
порядке индексов. Измерено (`bent-diag <n> orig`, NodeCount — размер
построенного BDD):

| n | NodeCount | время |
|---|---|---|
| 32 | 131071 (=2¹⁷−1) | 3.1 с |
| 36 | 524287 (=2¹⁹−1) | 11.5 с |
| 40 | 2097151 (=2²¹−1) | 50.4 с |
| 44 | **крах** (Lace fatal error: task stack overflow, `abort()`) | — |
| 48 | **крах** | — |

Точная закономерность: `NodeCount = 2^(n/2+1) - 1`.

**Диагностика причины**: построена та же самая функция с ЕДИНСТВЕННЫМ
отличием — пара переменных монома соседняя (`build_bent_mm_interleaved`:
`y_i = x_{2i+1}`, `x_i = x_{2i}`, вместо `y_i = x_{n/2+i}`):

| n | NodeCount (interleaved) | время |
|---|---|---|
| 32 | 33 | 8.5 мс |
| 48 | 49 | 12.2 мс |
| 64 | 65 | 14.1 мс |

Точная закономерность: `NodeCount = n + 1` — ровно линейно, как и
предсказывает теория (bent/IP-функция имеет линейный BDD при
чередующемся порядке `x_0,y_0,x_1,y_1,...` и экспоненциальный — при любом
порядке, разносящем пару далеко, классический факт литературы по decision
diagrams, тот же принцип, что уже обсуждался в §3.6 про cutwidth).

**Почему ни одна из 4 существующих эвристик (§3.4) не спасает**: граф
взаимодействия переменных для `build_bent_mm` — идеальное паросочетание
(у каждой переменной РОВНО один сосед, степень 1 у всех). На таком графе:

- `MinIndex` — identity по определению, пара максимально разнесена всегда.
- `Degree`/`LengthFreqRank` — у ВСЕХ переменных одинаковый ранг (степень 1,
  длина монома 2, частота 1 у каждой) → tie-break по индексу → тоже
  identity.
- `Force` — обновление позиции переменной `v` — это позиция её
  единственного соседа (взвешивание по единственному ребру ничего не
  меняет). На паре `(a,b)` это **буквально обмен** `pos[a]`/`pos[b]`
  местами на каждой итерации — период 2. При чётном числе итераций
  (сейчас 20) сходится РОВНО к identity; при нечётном — к "зеркальному"
  обмену половин (`x_i` ↔ позиция `y_i` и наоборот) — тоже максимальное
  разнесение пары, просто в другую сторону. Ни один исход не сближает
  пару — блайндспот FORCE на чистом паросочетании воспроизводится
  аналитически, не только эмпирически.

**Практический вывод**: все четыре эвристики этого файла оптимизируют
"общую разбросанность" (степень/частоту/притяжение), но ни одна не
распознаёт частный случай "граф — union непересекающихся рёбер" — а это
ровно структура канонической бент-функции, одного из самых
распространённых объектов в криптографическом анализе булевых функций.
Не исправлено в этой сессии (риск того же типа, что и решения §3.5 —
чинить эвристику вслепую без возможности прогнать под нагрузкой не
стоит) — зафиксировано как известное ограничение (см. также §1, §2):

- `anf_to_bdd`/`anf_to_bdd_with_heuristic` может выродиться в
  экспоненциальный (в пределе — падающий процесс) размер BDD на входах,
  чей граф взаимодействия переменных близок к паросочетанию с "далёкими"
  парами по индексу.
- Практический обход для вызывающего кода уже сегодня: если для
  конкретного класса входов заранее известно, что мономы естественно
  группируются в пары/блоки (как для bent/IP-конструкций), переиндексировать
  переменные так, чтобы члены одной группы были СОСЕДНИМИ по индексу, ДО
  вызова `anf_to_bdd` — как показывает `build_bent_mm_interleaved`, это не
  требует изменений в самом `anf_to_bdd.cpp`.
- Возможное будущее направление (не реализовано, не измерено): пятая
  эвристика, явно ищущая паросочетание/кластеры малого диаметра в графе
  взаимодействия (например, через жадное сжатие ребра максимального веса)
  ДО применения FORCE/Degree — вне скоупа этой сессии.

### 5.3. Мёбиус-трансформ — реальный профит от OpenMP-параллелизма (по запросу пользователя)

**Первая попытка (через `tt_to_anf_openmp_scaling`/`anf_to_tt_openmp_scaling`
в `test_anf.cpp`, `benchmarks/openmp_scaling.hpp`) дала шум**: speedup
0.69–1.11x без закономерности от n, сам харнесс пометил результат
"НЕПОСЛЕДОВАТЕЛЕН между размерами". Причина — методологический конфаунд:
эти тесты замеряют `tt_to_anf()`/`anf_to_tt()` ЦЕЛИКОМ, а не только сам
трансформ — внутри, ПОСЛЕ трансформа, идёт последовательная сборка
`BoolePolynomial` из коэффициентов (`poly += mono` на каждый ненулевой
коэффициент), и `make_bench_function` в `test_anf.cpp` (popcount(idx)>n/2)
даёт плотный ANF — до половины из 2^n коэффициентов ненулевые.

Попутно, при разработке этого исправления, обнаружен и устранён
**отдельный настоящий баг инфраструктуры**: `benchmarks/tbb_scaling.hpp` и
`benchmarks/openmp_scaling.hpp` независимо объявляли одни и те же
`detail::median_of`/`kMeasuredRuns` в namespace `bmm::benchmarks` — ODR-
конфликт (redefinition), не всплывавший раньше, т.к. ни один `.cpp` не
подключал оба backend'а сразу. Вынесено в `benchmarks/scaling.hpp` (общий
для обоих).

**Изолированный замер** (`anf/bench_real_corpus.cpp mobius-diag <n>`,
намеренная копия `mobius_transform_sequential/parallel` мимо
`tt_to_anf()`/`anf_to_tt()`/BRiAl вообще — чистая скорость самого
трансформа на массиве байт, порог параллелизма `kParallelThreshold=2^18`
пересекается при n=18):

**Сам трансформ** — параллелизм НЕ даёт выигрыша даже в чистом виде:

| n | single | parallel | speedup |
|---|---|---|---|
| 18 (первый над порогом) | 5.2 мс | 6.5 мс | 0.80 |
| 20 | 22.2 мс | 23.3 мс | 0.95 |
| 22 | 106.7 мс | 104.4 мс | 1.02 |

Правдоподобная причина: это streaming XOR-проход по массиву байт
(butterfly-паттерн) — классически **memory-bandwidth-bound**, а не
compute-bound. При таком профиле добавление потоков почти никогда не
помогает — упираетесь в пропускную способность памяти, а не в число ядер
(не проверено отдельно замером пропускной способности памяти контейнера —
это лучшее текущее объяснение, не строго доказанный факт).

**Сборка полинома** (`poly += mono`) — закрывает вопрос, оставленный
открытым в первой версии этого README (гипотеза из ревью производительности
про потенциальную квадратичность цикла построения полинома из коэффициентов
ТТ, не измерявшаяся отдельно от преобразования Мёбиуса):

| n | poly_construction_ms | рост за +2 к n |
|---|---|---|
| 12 | 6.7 | — |
| 16 | 124.0 | — |
| 18 | 500.5 | ×4.04 |
| 20 | 2284.0 | ×4.56 |
| 22 | 10564.4 | ×4.63 |

Рост близок к линейному по числу мономов (×4.0-4.6 при четырёхкратном
росте их числа), НЕ квадратичный — опасение из первого ревью насчёт O(n²)
не подтвердилось.

**Но именно это — настоящее узкое место `tt_to_anf` на плотных входах**:
на n=22 сборка полинома — 10.56 с, сам трансформ (даже параллельный) —
0.1 с, т.е. трансформ — **около 1% общего времени**. Вся инженерная
работа над параллелизмом (OpenMP, порог, комментарии в коде) ушла именно
в эту часть — а не в сборку, которая реально доминирует и никогда не
была распараллелена. Даже идеальное 8-кратное ускорение трансформа
изменило бы общее время `tt_to_anf()` на плотной функции на доли
процента — это и объясняет, почему первая попытка замера через
`tt_to_anf_openmp_scaling` показала чистый шум: сигнал (эффект от
параллелизма трансформа) был на два порядка меньше того, что вообще
можно различить на фоне остального (последовательного) времени вызова.

**Важная оговорка по применимости**: это касается ТОЛЬКО плотных
функций — все три реальных семейства в самом корпусе (§5) разрежены (χ:
3 монома при любом n; bent: n/2 мономов, т.е. 32 при n=64) и сборка
полинома для них практически бесплатна. Единственный плотный случай в
корпусе — AES S-box (n=8, 110-145 мономов) — слишком мал, чтобы этот
эффект был заметен. На практике узкое место проявляется именно там, где
`tt_to_anf` вызывается на функции без структуры (плотный, «случайный»
ANF) при n≈18-24 — например, ground-truth построение эталонов в
`verify/`, а не обязательно на реальных крипто-функциях из этого корпуса.

**Решение по коду в этой сессии — не тронуто.** Ни распараллеливание
сборки полинома (реальное узкое место), ни удаление бесполезной, но и не
вредной в среднем OpenMP-ветки трансформа — не сделаны; зафиксировано
здесь как открытый пункт для будущей сессии.

### 5.4. Проверка стабильности (повторный прогон `run_all()`)

По запросу пользователя ("прогони полный run_all() ещё раз и убедись, что
всё стабильно") корпус пересобран и прогнан заново с нуля. Результат —
стабильный: `REALEXIT=0`, дошёл до `=== done ===`, ни одного краха.

параллельной сессией/работой над этим репозиторием (контейнер гонял
`bench_real_corpus_thr` — файл `thr/bench_real_corpus.cpp`, не созданный в
рамках этой сессии; отдельно шёл `cmake --build --target test_bdd`).
Одновременный `cmake --build`/запуск бинарника из двух контейнеров в один
`build/` — риск гонки за файлы сборки и/или искажения замеров конкуренцией
за CPU, поэтому этот повторный прогон был отложен до завершения той работы
(проверено `docker ps`/`docker top` перед стартом). Если это README читают
в похожей ситуации — стоит проверить `docker ps` на предмет других
контейнеров `genetica-boolean-lib`, использующих тот же bind-mount, прежде
чем гонять что-то требующее чистых по времени измерений.

### 5.5. Как повторить

```
MSYS_NO_PATHCONV=1 docker run --rm -v "D:/Proga/web/bmm-translib:/workspace" -w /workspace genetica-boolean-lib:latest bash -c "cmake --build build --target bench_real_corpus -j\$(nproc) && ./build/bench_real_corpus"
```

Точечная диагностика взрыва `anf_to_bdd` на конкретном n:
```
./build/bench_real_corpus bent-diag <n> orig          # исходная (разнесённая) пара
./build/bench_real_corpus bent-diag <n> interleaved    # соседняя пара — для сравнения
```
Каждый запуск — отдельный процесс: `Lace fatal error` — это `abort()`,
не ловится внутри процесса, поэтому подбор границы взрыва делается
снаружи (растущий n, отдельный запуск на каждое значение), не в самом
бинаре.

Изолированный замер Мёбиус-трансформа/сборки полинома на конкретном n
(§5.3), тоже отдельным процессом на каждое n (не строго обязательно —
здесь ничего не падает — но так проще держать вывод читаемым):
```
./build/bench_real_corpus mobius-diag <n>
```

Штатные (не изолированные, для контекста) OpenMP-секции в общем ctest:
```
./build/test_anf 'tt_to_anf_openmp_scaling'
./build/test_anf 'anf_to_tt_openmp_scaling'
```
