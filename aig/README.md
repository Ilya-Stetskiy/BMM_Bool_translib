# aig/ — параллелизм, безопасность, тесты на больших n

Анализ всех пяти функций трансляции в этой папке (`tt_to_aig`, `aig_to_bdd`,
`aig_to_anf`, `aig_to_thr`, `aig_to_tt`) по трём осям: соответствие
`core/CONVENTIONS.md` п.6 (какой параллелизм положен по правилам), реальная
потокобезопасность реализации, и поведение на входах вплоть до n=64. Как и
`anf/README.md` — это рассуждения, а не описание API (см. `.hpp`) и не
журнал изменений (см. `git log`).

## 0. Почему не все пять функций тестировались на n=64

`tt_to_aig` принимает `TruthTable` — по определению `n_vars() <=
kMaxTruthTableVars = 24` (`core/common.hpp`), больших входов физически не
существует. `aig_to_thr`/`aig_to_tt` принимают `Aig` произвольного размера,
но **сами** отклоняют `n_vars() > kMaxTruthTableVars` (`TooManyVariables`)
на первой строке, потому что их выход требует материализовать 2^n точек
(таблицу истинности или прогнать по ней ILP). Значит **только `aig_to_bdd`
и `aig_to_anf`** способны в принципе принять `Aig` с n=64 — тестирование "до
64 переменных" относится к ним; `tt_to_aig`/`aig_to_thr`/`aig_to_tt`
протестированы на максимуме, который они вообще допускают (n=24 и меньше),
отдельно.

## 1. Параллелизм и безопасность по функциям

### 1.1. `tt_to_aig` — НАЙДЕН И ИСПРАВЛЕН реальный дедлок

Первая (статическая) ревизия этого раздела ошибочно назвала мемоизацию
"структурно корректной" — double-checked locking (`std::atomic<bool>
ready` + `std::mutex mtx` на запись `entry->val`) сам по себе безопасен
как ИЗОЛИРОВАННЫЙ паттерн, но здесь `std::lock_guard<std::mutex>
lock(entry->mtx)` был взят **на всё время рекурсивного построения узла**,
включая вложенный `tg.wait()` — то есть поток, ожидающий чужой
`entry->mtx` (другой узел, который параллельно строит другой поток), сам
становится недоступен планировщику TBB как воркер. Статический анализ этого
не поймал; поймала находка совсем другого рода — исправление методологии
бенчмарков (см. §4 ниже, по запросу пользователя "исправь тесты, чтобы были
адекватные данные"): переход на медиану из 5 повторов вместо одного замера
увеличил число вызовов `tt_to_aig` в этом тесте, и на структурно
симметричной bench-функции ("majority"-подобная, много совпадающих
кофакторов у разных ветвей → много конкурентных попаданий на ОДИН и тот же
memo-ключ) реальный прогон `ctest` **завис на 25 минут** и упёрся в таймаут
CTest — тест провалился.

**Диагностика**: отдельный контейнер (без `--rm`, чтобы не терять процесс),
воспроизведено зависание, снят стек всех потоков (`cuda-gdb -p <pid> -batch
-ex "thread apply all bt"` — обычного `gdb` в образе нет, `cuda-gdb`
прекрасно работает и для host-процессов). Картина недвусмысленная: главный
поток (`ps -T`) — 3 секунды CPU за 60 секунд реального времени, ВСЕ
TBB-воркеры — 0 секунд CPU (полностью простаивают, не считают). Стек
показал несколько потоков в `pthread_mutex_lock()` на `tt_to_aig.cpp:113`
(тот самый `lock_guard`) на РАЗНЫХ адресах мьютекса (разные memo-записи),
пока потоки, реально считающие эти записи, сами вложены в
`tbb::detail::d1::task_group_base::wait()`, ожидая задачу, которую
исполнять уже некому — классический TBB-антипаттерн "блокирующий
ОС-примитив внутри кооперативного планировщика" исчерпывает арену:
воркеры либо заблокированы на `mutex.lock()` (не участвуют в
work-stealing), либо сами `tg.wait()`-ят за исполнителя, которого нет.

**Исправлено**: `entry->mtx` теперь защищает только МГНОВЕННУЮ публикацию
готового результата (без рекурсии внутри критической секции) — если поток
обнаруживает, что нужный узел ещё не готов, он не блокируется в ожидании
чужой работы, а считает его сам, избыточно (mockturtle структурно хэширует
одинаковые AND-узлы, так что повторный `create_and` для тех же аргументов
почти всегда не создаёт нового узла — избыточность практически бесплатна;
в худшем случае — несколько лишних неиспользуемых узлов в сети, не баг
корректности). Блокирующего ожидания чужой работы больше нет вообще —
цепочка "жду мьютекс → держащий его сам ждёт свободный поток" структурно
невозможна. Проверено: 5 повторных прогонов `tt_to_aig_tbb_scaling` подряд
после фикса — ни одного зависания, каждый укладывается в 1-2 секунды;
корректность (`tt_to_aig` секция, ground_truth+metamorphic+sat_encoding) —
PASS; полный `ctest` — 36/36, 145с (было зависание на 25 минут до фикса,
~120с до всех изменений этой сессии).

**Отдельно про `net_mutex`** (собственная находка первой ревизии, остаётся
в силе): все три мутирующие операции над общей `mockturtle::aig_network`
(`get_constant_safe`/`create_and_safe`/`create_or_safe`) защищены ОДНИМ
общим мьютексом — реальное построение AIG полностью сериализовано,
параллелится только вычисление кофакторов
(`kitty::cofactor0/cofactor1`) между двумя рекурсивными вызовами. Это НЕ
источник дедлока (обычный короткий mutex без вложенной блокирующей
рекурции внутри), но ограничивает потенциальный выигрыш от параллелизма.
Замерено эмпирически ПОСЛЕ обоих фиксов (`benchmarks/tbb_scaling.hpp`,
медиана 5 повторов + 1 прогрев, n=8/12/16 — единственные размеры, реально
доступные этой функции: вход `TruthTable` капнут `kMaxTruthTableVars=24`):
пять независимых прогонов дают spread ~0.5x-1.5x БЕЗ устойчивого
направления — честный вывод бенчмарка теперь "результат непоследователен
между размерами... не делайте выводов по одному размеру", а не ложно
уверенное "медленнее в 4 раза", которое печатал единичный замер до фикса
методологии. Реальный эффект `net_mutex`-сериализации на этих (суб-мс)
размерах неотличим от шума измерения — потенциально сдерживающий эффект
есть теоретически, но эмпирически не подтверждён и не опровергнут на
доступном этой функции диапазоне n.
(вход — `TruthTable`, ограничен n<=24 по определению, а таблица там жёстко
кэп 2^24 точек — уже дорогая сама по себе для построения `TruthTable`, не
входит в скоуп этого анализа).

### 1.2. `aig_to_bdd` — корректно следует правилу 1

Нет собственного параллелизма (только Sylvan/Lace через `&`/`!` у
`sylvan::Bdd`), простой топологический обход `net.foreach_gate`,
мемоизация — плоский `std::vector<sylvan::Bdd>` по `node_to_index`. Никаких
проблем безопасности — однопоточный код по построению. Единственная
переменная, которая могла бы повлиять на размер результата — порядок PI
(здесь фиксированный, натуральный, `bddVar(pi_idx)` в порядке
`foreach_pi`) — в отличие от `anf_to_bdd` (см. `anf/README.md`), здесь нет
выбора эвристики, поэтому нечего сравнивать помимо самого размера итогового
BDD на разных структурах (см. §2).

### 1.3. `aig_to_anf` — САМАЯ ВАЖНАЯ находка сессии

Код содержит развилку по `BMM_HAVE_BRIAL`:

```cpp
#if BMM_HAVE_BRIAL
    PolType p0_val = apply_comp(get_anf_rec(child0), ...);   // ПОСЛЕДОВАТЕЛЬНО
    PolType p1     = apply_comp(get_anf_rec(child1), ...);
#else
    tbb::task_group tg;
    tg.run([&] { p0 = get_anf_rec(child0); });                // ПАРАЛЛЕЛЬНО
    PolType p1 = get_anf_rec(child1);
    tg.wait();
#endif
```

**TBB-форк по двум фанинам работает ТОЛЬКО в fallback-ветке
(`AnfFallback`, без BRiAl) — в реальной сборке (`BMM_HAVE_BRIAL=1`, как в
этом контейнере и как ожидается в production) код полностью
последовательный**, несмотря на то что `core/CONVENTIONS.md` п.6 правило 2
формально требует TBB для этой функции. Раньше (см. память сессии
`bmm_translib_anf_parallelism_work.md`) это было замечено, но не
задокументировано и не проверено — "похоже на осознанную предосторожность
... но нигде не задокументировано".

**Проверено эмпирически в этой сессии, окончательно:**

1. **ThreadSanitizer подтверждает: BRiAl (`libbrial`/PolyBoRi) НЕ
   потокобезопасен.** Стресс-тест (8 потоков × 2000 итераций конкурентного
   `ring.variable(idx) * ring.variable(idx2)` на одном `BoolePolyRing`, без
   мьютекса) под `-fsanitize=thread` даёт **7 подтверждённых data race**
   ровно там, где и следовало ожидать структурно: `polybori::CCuddCore::
   release()` — нессинхронизированный decrement счётчика ссылок у
   обёрнутого CUDD-менеджера (`BoolePolyRing` физически оборачивает
   `CCuddCore`, т.е. классический CUDD-style ZDD-менеджер — та же
   архитектура, из-за которой в этом же проекте выбран Sylvan вместо CUDD
   для BDD, см. `core/CONVENTIONS.md` — CUDD никогда не проектировался как
   потокобезопасный). Осторожность в `aig_to_anf.cpp` **оправдана**, это не
   лишняя перестраховка.
2. **Заявленный TBB-параллелизм в текущей (BRiAl) сборке — иллюзорный.**
   Первый прогон бенчмарка "1 поток vs все ядра" без прогрева дал
   обманчивые ускорения 20x-90x — оказалось артефактом: `get_ring(n)` в
   `aig_to_anf.cpp` кэширует ОДИН `BoolePolyRing` на "последний
   использованный n" (`static std::unique_ptr<BoolePolyRing> active_ring`);
   первый по счёту вызов с НОВЫМ n платит цену пересоздания ring (аллокация
   нового CUDD-менеджера), второй — уже тёплый кэш. Раз я мерил "1 поток"
   первым, а "все ядра" вторым при одном и том же n — разница была чисто
   про warmup, не про потоки. После добавления прогрева (одного лишнего
   вызова перед обоими замерами) реальная "разница между 1 потоком и всеми
   ядрами" на структурном корпусе (chain/star/block/mux/random, n=16..64)
   легла в диапазон **0.99x-1.9x** без какой-либо закономерности от
   размера — то есть шум измерения вокруг 1.0x, ровно то, что ожидается от
   кода, где параллельных задач физически не создаётся.

**Итог**: `aig_to_anf` не соответствует требованию TBB из
`core/CONVENTIONS.md` п.6 в actively используемой конфигурации сборки, и
это, по всей видимости, правильное решение по существу (не баг) — но
формулировка правила в CONVENTIONS.md не делает для этого случая
исключения, и решение нигде не объяснено с доказательствами до этой
сессии. Рекомендация: явно задокументировать в `aig_to_anf.hpp`/
`CONVENTIONS.md`, что BRiAl (в отличие от `AnfFallback`) не
поддерживает конкурентное построение полиномов на одном ring, со ссылкой
на этот файл — а не оставлять `#if BMM_HAVE_BRIAL` без обоснования.
Обязательный бенчмарк `aig_to_anf_tbb_scaling` в `test_aig.cpp` формально
проходит (тест лишь проверяет, что функция реализована, не требует
минимального speedup) — но печатаемое им число ("ускорение xN") в текущем
виде **вводит в заблуждение**: без понимания про `get_ring` warmup и про
BRiAl-ветку легко принять шумовые 1.0-1.3x (или, того хуже, случайно
незамеренный warmup-артефакт) за настоящий параллелизм.

**Дополнение (по итогам фикса дедлока в `tt_to_aig.cpp`, §1.1)**: у
`AnfFallback`-ветки (`#else` в блоке выше) обнаружен ТОТ ЖЕ дедлок-паттерн
— `std::lock_guard<std::mutex> lock(entry->mtx)` держался поперёк
`tg.run()`/`tg.wait()`. В текущей сборке (`BMM_HAVE_BRIAL=1`) эта ветка
никогда не выполняется (BRiAl-путь последователен), поэтому баг не
проявлялся эмпирически и не был бы пойман никаким прогоном тестов в ЭТОМ
образе — только чтением кода. Исправлен тем же способом, что и
`tt_to_aig.cpp` (mutex защищает только мгновенную публикацию, без
блокирующего ожидания чужой работы). **Не проверено живым прогоном** — в
этом Docker-образе BRiAl — системный заголовок, доступный по умолчанию
всем единицам трансляции; форсировать `BMM_HAVE_BRIAL=0` для конкретно
этого файла без пересборки образа без BRiAl не получилось в разумное
время. Корректность проверена только для АКТИВНОЙ (BRiAl) ветки — там
логика публикации результата общая для обеих веток (единственное отличие
— вокруг чего именно стоит `#if`), так что риск регрессии в BRiAl-пути
минимален, но `AnfFallback`-путь остаётся непроверенным вживую.

### 1.4. `aig_to_thr` — TBB, корректный и полезный паттерн

`tbb::parallel_for` по `[0, 2^n)` симулирует AIG на всех точках. Перед
параллельным участком узлы AIG вытащены в плоский `std::vector<FlatGate>`
(структуры POD, без указателей на `mockturtle::aig_network`) — это ровно
тот паттерн, который в памяти проекта отмечен как правильный ("вытащи
мономы/узлы в обычный vector до параллельного участка"). Каждый поток
работает со своим локальным `node_vals` буфером и пишет только в свой слот
`tt[idx]` — никакой разделяемой мутабельной памяти между потоками, гонок
структурно быть не может. Замер (1 прогрев + 1 поток vs все ядра):

| n | 1 поток, мс | все ядра, мс | speedup |
|---|---|---|---|
| 8  | 15.5  | 16.4  | 0.94 |
| 12 | 516.0 | 451.9 | 1.14 |
| 14 | 1797.4| 1172.8| **1.53** |

Растущее ускорение с ростом n — ожидаемо: полное время функции = симуляция
(параллельная) + ILP-solve (последовательный, не параллелится, см.
`aig_to_thr.hpp`); по мере роста n доля симуляции в общем времени растёт
быстрее (2^n) относительно ILP (число ограничений тоже растёт как 2^n, но
solve — не наивная точка-за-точкой работа), поэтому Amdahl-эффект даёт
всё больший выигрыш. Единственная практическая забота — при n, близких к
`kMaxTruthTableVars=24`, ILP получает **до 16 миллионов** ограничений (по
одному на точку) — не проверялось эмпирически в этой сессии, является
вероятным узким местом при больших n (см. `aig_to_thr.hpp`, комментарий уже
предупреждает об этом).

### 1.5. `aig_to_tt` — OpenMP, потенциальный вопрос снят статическим анализом

`#pragma omp parallel for` по блокам из 64 точек, каждый поток создаёт СВОЙ
`custom_block_simulator` и вызывает `mockturtle::simulate<kitty::
static_truth_table<6>>(net, sim)` на общей `const` сети `net`. Проверено
по исходнику `mockturtle/algorithms/simulation.hpp`: `simulate()` строит
**локальный** `node_map` (`simulate_nodes` внутри) на каждый вызов — не
хранит и не мутирует никакое состояние внутри самой `aig_network`, поэтому
конкурентные вызовы `simulate()` на одной и той же const-сети из разных
OpenMP-потоков безопасны (нет разделяемого mutable state, аналогично тому,
как параллельные READ-обращения к любой неизменяемой структуре данных
безопасны). Не проверено под TSan (в отличие от BRiAl-случая, где
структурная причина была неочевидна и потребовала подтверждения) — здесь
структурная причина безопасности ясна по коду библиотеки без запуска.

## 2. Разнообразный корпус, n до 64 (aig_to_bdd/aig_to_anf)

Тот же набор структурных семейств, что и в `anf/README.md` §5.2 (chain,
star/hub, block, one-hot mux, random sparse), построенных напрямую через
`mockturtle::aig_network` (`create_and`/`create_xor`, без `tt_to_aig` —
та ограничена n<=24). Корректность проверялась на 200 случайных точках на
семейство/размер против независимого эталонного вычислителя той же формулы
(exhaustive невозможен при n=64) — **все случаи, все размеры: OK**, ни
одного расхождения.

| семейство (n=64) | aig_to_bdd NodeCount | aig_to_bdd время, мс | aig_to_anf speedup (1 vs N потоков) |
|---|---|---|---|
| chain | 127 | 2.7 | 1.25 |
| star/hub (центр x_63) | 127 | 3.5 | 1.39 |
| block (клики по 4) | 65 | 0.4 | 1.06 |
| one-hot mux 4×7 (n=60) | 4771 | 22.1 | 1.59 |
| random sparse (n=32, максимум для этой серии) | 13791 | 19.2 | 1.88 |

`aig_to_bdd`-числа для chain/star/block **буквально совпадают** с
`min_index`-строкой из таблицы `anf/README.md` §5.2 — ожидаемо: `aig_to_bdd`
использует тот же натуральный PI-порядок, что и эвристика `MinIndex` в
`anf_to_bdd`, и обе стороны компилируют одну и ту же функцию через один и
тот же принцип "BDD уровень == индекс переменной" (`core/CONVENTIONS.md`
п.1) — разные входные представления (AIG vs ANF) одной и той же логической
функции при одинаковом порядке переменных дают один и тот же ROBDD (снова
теорема Брайанта о канонической форме). Как и в `anf/`, семейство `random`
на большем n показывает экспоненциальный рост NodeCount (221 → 1711 →
13791 при n=16/24/32) — то же математическое свойство неструктурных
функций, не специфика AIG-пути (см. `anf/README.md` §6 за объяснением
через cutwidth/bandwidth). `aig_to_anf`-speedup колонка — тот самый шум
вокруг 1.0-1.9x из §1.3, ожидаемо без выраженной закономерности от n или
семейства (нет реального параллелизма, который мог бы масштабироваться).

## 3. Практические выводы

- **Безопасно как есть**: `aig_to_bdd` (нет параллелизма, нечему
  ломаться), `aig_to_thr` (корректный flat-array паттерн), `aig_to_tt`
  (простая read-only параллельная симуляция).
- **`tt_to_aig`**: ИСПРАВЛЕНО в этой сессии (не только проанализировано) —
  был реальный дедлок (`entry->mtx` держался поперёк рекурсии/`tg.wait()`,
  см. §1.1), подтверждён стек-трейсом зависшего процесса, воспроизведён
  через реальный провал `ctest` (25 минут таймаут) после ужесточения
  бенчмарк-методологии (§4). Исправлено на "не блокировать, а считать
  избыточно" — 5/5 повторных прогонов без зависаний, корректность и полный
  `ctest` (36/36, 145с) подтверждены. Остаточный вопрос (не баг, а
  архитектурное ограничение) — глобальный `net_mutex`, сериализующий все
  мутации сети; на измеренных n<=16 эффект неотличим от шума (§1.1). Если
  когда-нибудь придётся ускорять `tt_to_aig` по-настоящему — точка роста
  именно здесь (снять `net_mutex`, например через thread-local буферы
  `aig_network` с последующим слиянием, тот же паттерн, что рассматривался
  и был отклонён для `anf_to_aig.cpp`, см. `anf/README.md` §7).
- **`aig_to_anf`**: ДВЕ отдельные находки, обе учтены. (1) BRiAl-ветка
  (активная в этой сборке) — TBB отключён правильно, это не баг, а
  требует только правки документации (явный комментарий в
  `aig_to_anf.cpp`/`.hpp`, почему TBB-форк недоступен, со ссылкой на этот
  README + TSan-находку §1.3 — сам код документации ещё не дописан, это
  открытый TODO). (2) `AnfFallback`-ветка (не активна в этой сборке) —
  ИСПРАВЛЕНА в этой сессии: содержала тот же дедлок-паттерн (mutex поперёк
  `tg.wait()`), что и `tt_to_aig.cpp` (§1.1), найден при целенаправленной
  проверке ПОСЛЕ фикса `tt_to_aig.cpp` (не был бы пойман прогоном тестов
  в этом образе — BRiAl здесь всегда доступен, эта ветка никогда не
  компилируется в тестовый бинарь). Исправлен код, но живым прогоном
  именно этой ветки проверить не удалось (см. §1.3, "не проверено живым
  прогоном"). Дополнительно стоит рассмотреть, не сделать ли в
  `core/CONVENTIONS.md` п.6 явное исключение для функций, работающих с
  `BoolePolynomial` на одном общем `BoolePolyRing` — сейчас формулировка
  правила подразумевает, что TBB всегда возможен для `aig_to_*`/`anf_to_*`,
  что для BRiAl-веток не так.

## 4. Методология бенчмарков исправлена (медиана вместо единичного замера)

По отдельному запросу пользователя ("исправь тесты, чтобы были адекватные
данные") — `measure_scaling()`/`measure_scaling_omp()`
(`benchmarks/tbb_scaling.hpp`/`benchmarks/openmp_scaling.hpp`) раньше
делали ОДИН замер на каждую сторону (1 поток / все ядра) после одного
прогрева. На суб-миллисекундных функциях (типичные размеры тестового
набора) единичный замер — в основном шум OS-планировщика, не сигнал.
Исправлено на 1 прогрев + **медиана 5 независимых повторов** на каждую
сторону — общий фикс для ВСЕХ `*_tbb_scaling`/`*_openmp_scaling` секций
проекта разом (`benchmarks/*.hpp` — общая инфраструктура для `test_aig.cpp`/
`test_anf.cpp`/`test_thr.cpp`), не только для `aig/`.

Заодно исправлен `print_bench_summary()` (`benchmarks/scaling.hpp`):
раньше итоговая строка ("ускорение до xN") бралась как **MAX** speedup
среди всех размеров — при шумных данных это систематически завышало
впечатление о параллелизме, выбирая самый "везучий" размер (конкретный
пойманный случай: `aig_to_anf` — {0.92, 1.27, 1.02} превращались в
заголовок "ускорение до x1.27", хотя реального параллелизма там нет вообще,
см. §1.3). Теперь — МЕДИАНА speedup по всем размерам + явная пометка, если
знак эффекта не согласован между размерами ("результат непоследователен...
не делайте выводов по одному размеру").

**Побочный эффект фикса — вскрыл реальный дедлок**: увеличение числа
вызовов на размер (было 1+1+1=3, стало 1+5+5=11) подняло вероятность
попадания в дедлок `tt_to_aig.cpp` настолько, что полный `ctest` реально
завис на 25 минут и провалился по таймауту — см. §1.1 за диагностикой и
фиксом. Число повторов сознательно взято 5, не 11 (как в разовых
диагностических бенчмарках этой сессии) — некоторые из бенчмаркуемых
функций (`aig_to_thr`/`anf_to_thr` — ILP-solve, `tt_to_anf` — Мёбиус-
трансформ) на `kMaxReferenceAigVars=14` уже стоят ~1-2с ЗА ОДИН вызов, 11
повторов ощутимо утяжелили бы каждый `ctest`-прогон.

## 4а. BRiAl vs AnfFallback — правда ли BRiAl ограничивает параллелизм, и что даёт отказ от него

Прямой вопрос пользователя после §1.3/§1.1: "правда ли BRiAl блокирует
многопоточность, и можем ли мы получить БОЛЬШЕ параллельности без неё?".
TSan-находка (§1.3) уже отвечает "да, правда" — но не показывает, ЧТО
конкретно можно выиграть/потерять, если реально переключиться на
`AnfFallback` (которая, в отличие от BRiAl, не имеет разделяемого
CUDD-подобного менеджера и структурно НЕ подвержена этому классу гонок).

Добавлен `BMM_FORCE_ANF_FALLBACK` (`core/anf_repr.hpp`, тот же паттерн, что
`BMM_FORCE_VARIANT_FALLBACK` для `Result<T>` в `core/common.hpp`) —
форсирует `AnfFallback`-ветку независимо от того, физически доступен ли
BRiAl в include-путях. Собран временный (`aig/bench_anf_fallback.cpp`,
удалён после записи чисел) отдельный исполняемый файл со СВОЕЙ копией
`aig/aig_to_anf.cpp`, скомпилированной с этим макросом, на ТОМ ЖЕ корпусе,
что и §2 (chain/star/block/mux/random). Заодно это был первый живой прогон
именно `AnfFallback`-ветки после фикса дедлока в §1.3 (там я честно
написал, что проверить её вживую не смог) — прогнано без единого
зависания, corrrectness OK на всех случаях, вопрос из §1.3 закрыт: фикс
работает и в этой ветке тоже.

**Результат** (абсолютное время `aig_to_anf`, BRiAl — медиана 1 потока из
предыдущего раунда §2, AnfFallback — медиана 1 потока из нового прогона):

| случай (n) | BRiAl, мс | AnfFallback, мс | во сколько раз AnfFallback МЕДЛЕННЕЕ |
|---|---|---|---|
| chain (16→64) | 0.035 → 0.357 | 0.560 → 58.964 | **16x → 165x** |
| star/hub (16→64) | 0.033 → 0.393 | 1.024 → 34.425 | **31x → 88x** |
| block (16→64) | 0.012 → 0.055 | 0.036 → 1.105 | **3x → 20x** |
| one-hot mux (n=21→60) | 0.024 → 0.190 | 0.209 → 4.032 | **9x → 21x** |
| random sparse (16→32) | 0.055 → 0.216 | 0.891 → 7.822 | **16x → 36x** |

**И это BEZ учёта параллелизма** — сравнение "1 поток BRiAl" vs "1 поток
AnfFallback". С параллелизмом AnfFallback СТАНОВИТСЯ ХУЖЕ, не лучше: во
всех 19 измеренных случаях, кроме двух (`chain_n64`=1.06x,
`block4_n32`=1.06x), параллельная версия AnfFallback МЕДЛЕННЕЕ
однопоточной (0.5x-0.9x) — те же самые реальные `tg.run()`/`tg.wait()`
TBB-задачи, которых в BRiAl-ветке вообще нет, здесь честно выполняются
параллельно, но накладные расходы TBB не окупаются на этих размерах входа
даже при генуинном параллелизме (тот же эффект, что у `net_mutex` в
`tt_to_aig.cpp`, §1.1 — параллелизм сам по себе не панацея, если
единица работы дешевле накладных расходов на её распределение).

**Почему разрыв в 16x-165x, и почему растёт с n**: BRiAl хранит множество
мономов как ZDD (Zero-suppressed Decision Diagram) со структурным
разделением одинаковых поддеревьев между операциями — умножение/сложение
полиномов работает через операции над ZDD-структурой, а не перебором пар
мономов. `AnfFallback` — это буквально `std::set<Monomial>`: сложение —
merge двух отсортированных множеств (нормально), но **умножение —
декартово произведение мономов** (`mul_poly` в `aig_to_anf.cpp`, вложенный
цикл по всем парам `(m1, m2)`) без какого-либо структурного разделения
между промежуточными полиномами разных узлов AIG. Разрыв растёт с n именно
потому, что структурное разделение ZDD даёт БОЛЬШИЙ выигрыш там, где
больше overlapping-структуры между промежуточными полиномами — а на
маленьких n (n=16) оверлапа мало, разрыв всего 3x-31x.

**Ответ на вопрос пользователя**: технически параллелизм БЕЗ BRiAl получить
можно (`AnfFallback` не имеет разделяемого мутабельного менеджера, поэтому
структурно не подвержена гонке, и её TBB-ветка теперь, после фикса
дедлока, реально работает) — но это плохой обмен: вы получаете
"настоящий", но не окупающийся параллелизм (0.5x-0.9x) ценой алгоритма,
который на n=64 уже в 88-165 раз медленнее в один поток. Правильный путь к
БОЛЬШЕЙ параллельности без потери скорости BRiAl — не отказ от BRiAl, а
одно из:

1. **Отдельный `BoolePolyRing` на поток** (если BRiAl это вообще
   позволяет — не проверено в этой сессии): каждый поток получает СВОЙ
   CUDD-менеджер, без разделяемого мутабельного состояния, значит без
   гонки по построению; но результаты из разных ring'ов, вероятно, нельзя
   комбинировать напрямую (нужна явная конвертация/сериализация на
   границах) — архитектурно нетривиально, не проверено эмпирически.
2. **ZDD-аналог Sylvan для представления множества мономов** — тот же
   принцип, что уже используется в этом проекте для BDD (`core/
   CONVENTIONS.md`: Sylvan выбран вместо CUDD именно из-за потокобезопасной
   архитектуры на Lace) — но для ANF готового решения в текущем стеке
   зависимостей нет, пришлось бы либо искать сторонний потокобезопасный
   ZDD-пакет, либо писать свой. Существенный объём работы, не в скоупе
   этой сессии.
3. **Оставить как есть** — BRiAl-путь фактически последователен, но
   быстрый; накладные расходы честного TBB-параллелизма на
   `AnfFallback`-масштабах входа (n<=64) всё равно не окупаются даже без
   учёта разрыва в 16x-165x, так что "больше параллельности" здесь не
   значит "быстрее" ни при каком из проверенных вариантов.

`BMM_FORCE_ANF_FALLBACK` оставлен в `core/anf_repr.hpp` (не удалён вместе
с временным бенчмарком) — низкий риск (неактивная `#if`-ветка, пока макрос
явно не передан через `-D`), но полезен для будущих аналогичных проверок
без пересборки Docker-образа без BRiAl.

## 5. Как повторить

Оба диагностических инструмента (`aig/stress_brial_threads.cpp` под
`-fsanitize=thread`, `aig/bench_aig_corpus.cpp` с корпусом до n=64) были
временными и удалены из финального диффа после того, как числа записаны
здесь — тот же паттерн, что и у `anf/bench_*.cpp` (см. `anf/README.md`).
Полная история находок этой сессии, включая тупиковые ветки (первый прогон
бенчмарка без прогрева, давший ложные 20x-90x) — в памяти Claude Code
этого проекта, недоступна вне сессий с тем же механизмом памяти.
